# UI 设计资产规范（Harness 接口文档）

> **本文档是 `new-world-ops`（Harness 层）的接口规范，定义 ui-designer Agent 产出的设计资产标准格式。**
> 
> **项目特定的设计规则（如具体色值、组件名称、状态管理方案）应放在被管理项目自身的 `docs/design-system/` 目录下，而不是修改本文档。**

## 目的

为所有被 `new-world-ops` 管理的项目提供统一的设计资产输出格式，确保：
- Agent 产出的设计资产可被开发者和 Reviewer 一致理解
- 不同项目的设计资产在目录结构和文件类型上保持一致
- 设计到开发的交接有明确的标准

## 目录结构（强制）

每个 Issue 的设计资产必须放在统一的目录下：

```
designs/issue-{ISSUE_NUMBER}/
├── wireframe.svg      # 低保真线框图
├── mockup.html        # 高保真可交互原型
└── design-spec.md     # 设计规范与对接文档
```

> 例如：`designs/issue-42/wireframe.svg`

## 1. wireframe.svg

### 目的
快速表达信息架构、页面流程、区域划分，不纠结于视觉细节。

### 技术要求
- 使用 **SVG 1.1** 格式，单文件可浏览器直接打开
- 背景层：`#f8fafc`，元素层：`#e2e8f0` ~ `#94a3b8` 灰度
- 所有文字使用 `<text>` 标签，字号 12px ~ 16px
- 交互箭头使用 `<path>` + `<polygon>` 箭头标记

### 推荐尺寸（按项目类型选择）

| 项目类型 | 推荐画布尺寸 |
|---|---|
| **Flutter (Mobile)** | 375 × 812 |
| **Phaser3 (横屏)** | 1280 × 720 |
| **Phaser3 (竖屏)** | 750 × 1334 |
| **Web (桌面)** | 1440 × 900 |
| **Web (移动端)** | 375 × 812 |

*注：具体尺寸应以项目自身 `docs/design-system/` 中的规范为准。*

## 2. mockup.html

### 目的
高保真还原视觉和关键交互，让产品和开发对最终效果有直观预期。

### 技术要求
- **单文件 HTML**，所有 CSS 内联在 `<style>` 中，所有 JS 内联在 `<script>` 中
- 可直接双击在浏览器打开，无需服务器
- 占位图使用 `data:image/svg+xml;base64,...` 内联，禁止引用外部 CDN 图片
- 文字使用中文或项目约定的 i18n key

### 内容要求
- 必须展示 **至少 2 种状态**（如正常态 / 空数据态 / 错误态）
- 可交互元素（按钮、表单、弹窗）必须有明显的交互状态演示
- 动画需标注时长和缓动函数

*注：具体组件样式、主题变量、响应式断点应以项目自身 `docs/design-system/` 为准。*

## 3. design-spec.md

### 目的
把设计决策文档化，作为开发实现的"技术对接单"。

### 强制章节

```markdown
# Issue #{N} 设计规范

## 1. 设计目标
（一句话概括本次需求的核心设计目标）

## 2. 页面/场景清单
| 编号 | 名称 | 对应实现文件 | 备注 |
|------|------|--------------|------|
| 1 | 首页 | HomePage | 含空数据态 |

## 3. 视觉系统
### 3.1 色板
### 3.2 字体
### 3.3 间距系统

## 4. 组件清单
| 组件名 | 文件路径建议 | 输入/Props | 状态 |
|--------|--------------|------------|------|

## 5. 图片/图标资源
| 名称 | 尺寸 | 格式 | 用途 |
|------|------|------|------|

## 6. 交互说明

## 7. 待确认事项
```

*注：Flutter 项目应补充 Theme/BLoC 映射；Phaser3 应补充 Scene/分辨率适配；Web 应补充 CSS 变量和断点。具体格式以项目自身文档为准。*

## 项目级设计规范（Project 层）

Harness 层的 `ui-designer` Agent 在启动时会主动搜索并阅读项目内的设计系统文档。建议每个被管理项目包含以下文档之一：

```
<project-root>/
├── docs/
│   ├── design-system/
│   │   ├── README.md          ← 推荐：设计系统总入口
│   │   ├── colors.md
│   │   ├── typography.md
│   │   └── components.md
│   └── ui-design-spec.md      ← 项目级覆盖规范（可选）
```

如果项目同时存在 Harness 规范（本文档）和项目级规范，**以项目级规范为准**。

## Reviewer 审查设计资产的 Checklist

当 PR 涉及 UI 变更时，Reviewer 检查：

- [ ] `designs/issue-{N}/` 目录是否存在且包含三件套
- [ ] `wireframe.svg` 是否清晰表达了信息架构和流程
- [ ] `mockup.html` 是否能在浏览器独立打开
- [ ] `design-spec.md` 中的视觉系统是否与代码实现一致
- [ ] 代码实现是否与项目 `docs/design-system/` 的规范一致
- [ ] 新增组件/资源是否已在设计资产中登记

## 严重问题（Major）

### 1. `MatchingSwipeAnimation` 错误继承 `MatchingState`，违反 BLoC 状态语义正确性
- **位置**: `apps/mobile/lib/presentation/blocs/matching/matching_state.dart:86`
- **问题**: `MatchingSwipeAnimation` 是一个纯动画参数数据类（`dragOffsetX`、`dragRotation`、`showLikeBadge`、`showPassBadge`），却被声明为 `MatchingState` 的子类。根据 `docs/design-docs/flutter.md` 第 114–135 行的「BLoC 状态语义正确性」规范：
  > State 类命名必须准确反映其业务语义，禁止将一个 State 用于多个不相关的场景。
  
动画参数不是 Matching 功能的业务状态，而是伴随 `MatchingLoadSuccess` / `MatchingSwipeInProgress` 下发给 UI 的附加数据。将其放入状态层级会导致：
  1. 意外 `emit(MatchingSwipeAnimation())` 时 `BlocConsumer` 的 `builder` 无法处理，UI 会回退到默认空态。
  2. 状态语义混淆——`props` 中混入非业务状态字段。
- **影响**: 维护性下降；后续开发者容易误将动画参数当作独立状态 emit，导致 UI 异常。
- **修复**:
  ```dart
  // matching_state.dart
  class MatchingSwipeAnimation extends Equatable {  // ❌ 不是 MatchingState
    ...
  }
  ```
同时更新 `MatchingLoadSuccess` / `MatchingSwipeInProgress` 中对 `animation` 字段的类型注解（已是 `MatchingSwipeAnimation`，无需改动）。
- **依据**: `docs/design-docs/flutter.md` 第 114–135 行「BLoC 状态语义正确性」

## 风险与建议

1. **MatchingSocketCubit 的「Connected」语义风险**：`connect()` 在 `await readAccessToken()` 之后立即 `emit(MatchingSocketConnected())`，但此时 socket 实际连接可能尚未建立（`MatchingSocketClient.connect()` 是异步启动）。这是原有代码逻辑平移，不是本 PR 引入的新问题，但建议在后续迭代中增加 `MatchingSocketConnecting` 中间态，避免 UI 在连接真正建立前展示"已连接"。

2. **遗留 Profile 分离工作**：PR 已明确将 `ProfileSetupPage` / `ProfileEditPage` 的分离记入 `tech-debt-tracker.md`，建议下个迭代优先处理，防止新功能继续在旧混写模式上堆叠。

3. **模板检查清单流程**：建议 Agent 在提交重构类 PR 时，直接在 PR 描述顶部引用对应模板（本例为 `TEMPLATE-tech-debt.md`）并逐项勾选，减少 review 时的合规确认成本。

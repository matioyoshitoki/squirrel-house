## 审查结论

审查结论: **NEEDS_MAJOR_FIX**

PR #58 实现了 IM（即时通讯）全链路模块，覆盖 DB → API → Mobile → Design-System → 文档，整体架构方向正确，组件封装和 Maestro E2E 流也有配套产出。但存在 **6 个 Blocking 问题**，其中核心消息发送功能在 Mobile 端完全不可用（Socket 未连接），且后端 Service/Controller 层违反了 NestJS 编码规范（直接返回 Entity）。这些问题必须在合并前修复。

## 阻塞问题（Blocking）

### 1. Mobile 端 Socket 从未连接，消息发送功能完全不可用
- **位置**: `apps/mobile/lib/presentation/pages/chat_page.dart:36-40`
- **问题**: `ChatPage.initState` 中创建了 `ImBloc` 并加载历史消息，但 **从未发送 `ImConnectRequested` 事件**。全局搜索（`Grep` 验证）确认 `ImConnectRequested` 仅在 `im_event.dart` 和 `im_bloc.dart` 中存在，没有任何 UI 代码调用它。
- **影响**: `_socketClient` 始终为 `null`，`_onSendMessage` 中的 `_socketClient?.sendMessage(...)` 永远为空操作。用户点击发送后消息状态停留在 `sending`，永远不会到达服务端。
- **修复**:
  1. 在 `ChatPage.initState` 中，从 `AuthBloc` 或 `SecureStorage` 获取当前用户的 JWT token，并调用 `bloc.add(ImConnectRequested(token))`。
  2. 在 `ChatPage.dispose` 中调用 `bloc.add(const ImDisconnectRequested())` 以清理连接。
- **依据**: `prd/v1-mvp.md` 第 160 行明确要求“使用 Socket.io 实现实时消息收发”；`docs/design-docs/nestjs.md` Socket.io Gateway 规范要求客户端必须连接后才能收发事件。

### 2. `ImBloc` 连接状态会覆盖消息状态，导致 UI 显示加载中并丢失消息列表
- **位置**: `apps/mobile/lib/presentation/blocs/im/im_bloc.dart:68-75`
- **问题**: `_onConnect` 发射 `ImConnected()`，`_onDisconnect` 发射 `ImDisconnected()`。这两个状态类不包含消息列表。`ChatPage` 的 `BlocConsumer` builder 仅在 `state is ImMessagesLoaded || state is ImMessageSending || state is ImMessageSent || state is ImMessageRead` 时渲染消息列表，否则回退到 `CircularProgressIndicator`。
- **影响**: 一旦 Socket 连接/断开，聊天页面会从“显示消息”瞬间变成“全屏加载圈”。
- **修复**:
  - 方案 A：删除 `ImConnected` / `ImDisconnected` 这两个独立状态，改为在现有消息状态类中增加 `isConnected` 字段；
  - 方案 B：在 `ChatPage` builder 中增加对 `ImConnected` / `ImDisconnected` 的处理，保持上一状态的消息列表渲染。
- **依据**: `docs/design-docs/flutter.md` 状态管理规范要求 BLoC 状态应支持 UI 的连续渲染，不应因连接状态抖动导致页面骨架重置。

### 3. `ImService.getMessages` 直接返回 TypeORM Entity，违反分层规范
- **位置**: `apps/api/src/modules/im/im.service.ts:157-200`
- **问题**: `getMessages` 的返回类型显式声明为 `Promise<{ items: MessageEntity[]; next_cursor: string | null; has_more: boolean }>`，直接返回数据库 Entity。
- **影响**: 违反 NestJS 编码规范，存在泄露内部字段（如未来可能新增的敏感 DB 字段）的风险。
- **修复**:
  1. 新建 `MessageResponseDto`，字段与 `MessageEntity` 对应但显式排除内部字段；
  2. 在 `getMessages` 中将 `MessageEntity` 数组映射为 `MessageResponseDto[]` 后返回。
- **依据**: `docs/design-docs/nestjs.md` 第 56 行 “Service 层：禁止直接返回 Entity”；第 85 行 “所有出参使用显式 DTO，禁止直接返回 TypeORM Entity”。

### 4. `ImController` API 响应直接包含 TypeORM Entity
- **位置**: `apps/api/src/modules/im/im.controller.ts:56-66`
- **问题**: `getMessages` 调用 `imService.getMessages(...)` 后，将返回的 `data` 直接包装进 `{ code: 0, message: "success", data }`。由于 `ImService` 返回的是 `MessageEntity[]`，Controller 实际上在 HTTP 响应中序列化了 Entity。
- **影响**: 同 Blocking-3，违反 Controller 层“禁止直接返回 Entity”的规范。
- **修复**: 在 `ImService` 修复为返回 DTO 后，Controller 自然合规；若需临时修复，可在 Controller 中增加一层 `data.items.map(...)` 转换为 DTO。
- **依据**: `docs/design-docs/nestjs.md` 第 56 行 Controller 职责表：“禁止直接操作 Repository”；以及 DTO 规范：“禁止使用 Entity 作为接口类型”。

### 5. `MessageEntity` 唯一索引与幂等去重逻辑不一致
- **位置**: `apps/api/src/database/entities/message.entity.ts:13-15`
- **问题**: 唯一索引仅建立在 `client_message_id` 单列上：
  ```typescript
  @Index("idx_messages_client_message_id", ["client_message_id"], { unique: true })
  ```
但 `ImService.sendMessage` 的幂等去重查询使用的是 `(conversation_id, client_message_id)` 组合条件（`im.service.ts:115-118`）。
- **影响**: 如果同一用户在不同会话中因客户端 bug 产生了相同的 `client_message_id`，DB 层面会报唯一约束冲突（`ER_DUP_ENTRY`），但去重查询因限定了 `conversation_id` 而找不到已有记录，导致代码进入 `throw err` 分支，返回 500 错误而非幂等成功。
- **修复**:
  1. 将 Entity 索引改为复合唯一索引：`@Index("idx_messages_conv_client_msg_id", ["conversation_id", "client_message_id"], { unique: true })`；
  2. 同步修改 Migration 文件 `20250425000000-AlterMessagesSenderIdNullable.ts`（或新建 Migration）删除旧索引、创建新索引；
  3. 同步更新 `im.service.ts` 中 `catch (err)` 块的 `err.message.includes(...)` 匹配字符串，确保能捕获新的索引名错误信息。
- **依据**: `docs/modules/im.md` 边界情况第 1 条明确说明“客户端生成 `client_message_id`，服务端幂等去重”；幂等去重查询与唯一索引必须语义一致。

### 6. `SendMessageDto.client_message_id` 校验规则与服务端 fallback 逻辑矛盾
- **位置**: `apps/api/src/modules/im/dto/send-message.dto.ts:14-15` 与 `apps/api/src/modules/im/im.service.ts:76`
- **问题**: DTO 中 `client_message_id` 使用了 `@IsString()`（表示必填），但 `ImService.sendMessage` 中写了 `const clientMessageId = dto.client_message_id ?? uuidv7();`，暗示该字段可选。
- **影响**: 如果客户端请求体中省略 `client_message_id`，`class-validator` 会在 Controller/Guard 阶段拦截并返回 400，不会走到 Service 的 fallback 逻辑。两个层面的语义不一致。
- **修复**:
  - 方案 A：在 DTO 上增加 `@IsOptional()`，使 Service 的 fallback 逻辑真正生效；
  - 方案 B：删除 Service 中的 `?? uuidv7()` fallback，要求客户端必须提供（更符合当前 DTO 定义）。
推荐方案 A，因为客户端在断网重发场景下需要依赖服务端生成 ID。
- **依据**: `docs/design-docs/nestjs.md` 第 83-87 行 DTO 与校验规范：“DTO 的字段命名必须与 `packages/shared-types/` 中的接口定义完全一致”；接口契约应前后一致。

## 严重问题（Major）

### 1. `ChatPage` 未实现图片消息，与 PRD 需求不符
- **位置**: `apps/mobile/lib/presentation/pages/chat_page.dart:127-169`；`packages/design-system/lib/src/molecules/chat_bubble.dart:117-118`
- **问题**: `ChatPage` 的 `itemBuilder` 中所有消息都当作文本渲染；`SwChatBubble._buildContent` 明确注释 “For simplicity, treat all as text. Image handling can be extended.”。但 PRD 明确要求 MVP 支持图片消息。
- **影响**: 功能不完整，用户无法发送/接收图片。
- **修复**:
  1. 在 `ChatPage` 中根据 `msg.type` 区分文本与图片渲染；
  2. 在 `SwChatBubble` 中扩展 `_buildContent` 以支持图片预览；
  3. 在 `SwChatInputBar` 中启用 `onPickImage` 回调（组件已预留，但 `ChatPage` 未传入）。
- **依据**: `prd/v1-mvp.md` 第 154-157 行：“消息类型：MVP 阶段支持：文本、图片”。

### 2. IM 模块缺少离线用户推送通知实现
- **位置**: `apps/api/src/modules/im/im.service.ts:52-126`
- **问题**: `sendMessage` 中，消息持久化后仅通过 Socket.io 推送给在线用户，没有检测接收方是否在线，也没有调用 `PushService` 发送离线推送。
- **影响**: 离线用户收不到新消息提醒，违反 PRD 非功能需求。
- **修复**:
  1. 在 `sendMessage` 中，向接收方房间 emit 后，检查该用户是否在线（可通过 Socket.io `server.fetchSockets()` 或 Redis 维护在线状态）；
  2. 若不在线，调用 `PushService.sendPush(otherId, title, body, payload)` 触发推送。
- **依据**: `prd/v1-mvp.md` 第 163 行：“离线用户通过极光推送/Firebase 触发新消息提醒”。

### 3. `ChatPage` 离开会话时未清理 `ImBloc` 的当前会话上下文
- **位置**: `apps/mobile/lib/presentation/pages/chat_page.dart:43-47`
- **问题**: `dispose()` 只释放了 `_scrollController`，没有调用 `bloc.setCurrentConversation(null, null)`。
- **影响**: 用户离开 `ChatPage` 后，若 Socket 收到新消息，`ImBloc._onMessageReceived` 仍可能匹配到旧的 `_currentConversationId`，导致自动发送 read receipt 给错误的会话。
- **修复**: 在 `dispose()` 中增加 `bloc.setCurrentConversation(null, null)`。

### 4. `im.yaml` OpenAPI 契约与实际 API 返回结构不一致
- **位置**: `docs/api-contracts/im.yaml`
- **问题**:
  - `ConversationDto` 的 `required` 包含 `id`，但 `ImController.getConversations` 实际返回的字段名是 `conversation_id`；
  - `MessageDto.type` 的 `enum: [text, image]` 缺少 `system`（系统消息类型）；
  - `MessageDto.sender_id` 标记为 required，但数据库和 Entity 中该字段为 `nullable`（系统消息无 sender）。
- **影响**: 契约文档与代码不一致，会导致 OpenAPI 代码生成、前后端联调出错。
- **修复**:
  1. `ConversationDto` 将 `id` 改为 `conversation_id`；
  2. `MessageDto.type` 的 enum 增加 `system`；
  3. `MessageDto.sender_id` 增加 `nullable: true` 或从 required 中移除。
- **依据**: `docs/design-docs/nestjs.md` 第 87 行 “DTO 的字段命名必须与 `packages/shared-types/` 中的接口定义完全一致”；`docs/api-contracts/INDEX.md` 要求契约文件与代码同步。

### 5. `MatchListPage` 类名与职责不符
- **位置**: `apps/mobile/lib/presentation/pages/match_list_page.dart:10`
- **问题**: 页面类名为 `MatchListPage`，但实际渲染的是“消息/会话列表”（底部导航 Tab 文案为“消息”，AppBar 标题为“消息”）。而真正的“匹配列表”应在 Matching 模块。
- **影响**: 代码可读性差，维护者极易混淆。
- **修复**: 将类名和文件名统一重命名为 `ConversationListPage` / `conversation_list_page.dart`，并同步更新 `app.dart` 中的 import 和引用、`docs/modules/im.md` 中的文件索引。
- **依据**: `docs/design-docs/flutter.md` 命名规范：“Widget 类名必须以描述功能结尾”；`docs/modules/im.md` 也将该页面索引为 “Conversation List Page”。

### 6. IM BDD 测试的 Step Definitions 为空壳
- **位置**: `tests/bdd/step-definitions/im_*.steps.js`
- **问题**: 所有 IM 相关的 step definition 文件（如 `im_real_time_message_delivery.steps.js`）仅执行 `autoBindSteps(feature, [commonSteps])`，没有定义任何 IM 专属步骤（如 `When Alice sends "..."`）。
- **影响**: BDD 测试虽然存在 `.feature` 文件，但无法验证 IM 业务行为，失去了验收价值。
- **修复**: 补充 IM step definitions，至少覆盖：
  - Socket 连接模拟；
  - 消息发送 REST/Socket 调用；
  - 消息历史查询断言；
  - 已读回执断言。
- **依据**: `docs/design-docs/testing.md` 第 59-66 行 “数据闭环原则” 和 BDD Mock 规范。

### 7. `ImSocketClient` 缺少自动重连配置
- **位置**: `apps/mobile/lib/core/network/im_socket_client.dart:30-34`
- **问题**: `io.io()` 初始化时未配置 `reconnection: true`、重连间隔、最大重试次数等参数。
- **影响**: 网络闪断后 Socket 不会自动恢复，用户必须杀掉 App 重进才能重新连接。
- **修复**: 在 `io.io()` options 中增加 `reconnection: true`、`reconnectionAttempts: 5`、`reconnectionDelay: 1000` 等配置。

## 风险与建议

1. **核心功能不可用风险（高）**: 由于 `ImConnectRequested` 从未被调用，当前 PR 合并后 Mobile 端无法发送任何 Socket 消息。建议在修复后，至少通过以下方式验证：
   - 在 emulator 上运行 `make test-mobile-e2e`（`send_message.yaml` 流）；
   - 或本地启动 `make e2e-env-up`，用 Maestro 跑通 `im/send_message.yaml`。

2. **数据一致性风险（中）**: `MessageEntity` 的唯一索引与去重查询不一致，可能导致偶发的 500 错误。建议优先修改为复合唯一索引。

3. **技术债务风险（中）**: `MatchListPage` 命名与职责不符，越早重命名成本越低。若拖延，后续在导航、路由、埋点、E2E 中会产生更多隐式依赖。

4. **后续迭代建议**:
   - 补充 `apps/mobile/test` 层的 Widget Test，覆盖 `ChatPage` 在 `ImMessagesLoaded` / `ImError` / `ImMessageSent` 状态下的渲染；
   - 在 `ImBloc` 中增加连接状态心跳或断线检测，并暴露给 UI（如在 `ChatPage` AppBar 显示“连接中/已断开”提示）；
   - 考虑将 `ImBloc` 提升为全局状态（放在 `app.dart` 的 `MultiBlocProvider` 中），而非每个 `ChatPage` 独立创建，以便会话列表和聊天页共享 Socket 连接。

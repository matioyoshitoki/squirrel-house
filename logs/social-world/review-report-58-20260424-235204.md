## 严重问题（Major）

### 1. IM 后端新增代码零单元/集成测试覆盖

- **问题**: PR 新增了 IM 核心后端模块，但没有任何 `.spec.ts` 测试文件。`matching.service.spec.ts` 在同期被更新（说明项目有测试传统），但 IM 模块完全缺失。
- **影响**: 无法验证消息幂等去重、已读回执逻辑、会话权限校验、分页 cursor 边界、异常处理路径等核心行为。后续重构极易引入回归。
- **修复**:  
在 `apps/api/src/modules/im/tests/` 下新增：
  - `im.service.spec.ts`：覆盖 `sendMessage`（含重复 client_message_id 幂等）、`markAsRead`、`getMessages` 分页、`getConversations` 聚合、`verifyConversationMembership`
  - `im.gateway.spec.ts`：覆盖 Socket 认证拒绝、`message:send` 事件处理、`message:read` 事件处理、错误广播
  - `im.controller.spec.ts`：覆盖 `GET /conversations` 和 `GET /messages` 的 Guard、参数校验、响应结构
- **依据**: `docs/design-docs/testing.md` 第 17-22 行："后端：NestJS 使用 Jest，覆盖率目标 >80%"

### 2. 缺失 IM Socket 事件契约 YAML
- **位置**: `docs/api-contracts/` 目录下无 `im-events.yaml`
- **问题**: `im.gateway.ts` 发射了 5 个服务端事件：`message:ack`、`message:receive`、`message:read_receipt`、`conversation:updated`、`message:error`。但 `docs/api-contracts/` 下只有 `im.yaml`（REST 契约），没有 `im-events.yaml`（WebSocket 事件契约）。
- **影响**: 前后端事件契约不同步时无文档可核对；新 Agent 接入时无法从仓库获知完整事件列表和 Payload Schema。
- **修复**:  
新建 `docs/api-contracts/im-events.yaml`，定义所有事件：
  - `message:ack` — Payload: `{ client_message_id, sequence_id, id }`
  - `message:receive` — Payload: `{ id, conversation_id, sender_id, type, content, sequence_id, client_message_id, created_at }`
  - `message:read_receipt` — Payload: `{ conversation_id, last_read_message_id }`
  - `conversation:updated` — Payload: `{ conversation_id, updated_at }`
  - `message:error` — Payload: `{ message }`
- **依据**: `docs/design-docs/api.md` 第 89-118 行（不变量 A）："后端每新增一个 `server.emit()` 事件，必须同时满足：`docs/api-contracts/<module>-events.yaml` 中定义了该事件的 Schema"

### 3. Maestro E2E `send_message.yaml` 存在流程断点
- **位置**: `.maestro/flows/im/send_message.yaml` 第 22-34 行
- **问题**: Flow 先 `optional: true` 点击 `conversation_item`，然后 `optional: true` 断言 `chat_back_button`，随后**无条件**执行 `tapOn: { id: "chat_input_field" }` 和 `inputText: "你好"`。如果测试环境没有预置会话（如首次运行或新 emulator），用户仍停留在「消息」列表页，`chat_input_field` 不存在，测试直接失败。
- **影响**: E2E 无法在干净环境跑通，丧失回归测试价值。
- **修复**:  
提供两条路径之一：
  - **路径 A（推荐）**: 在 flow 开头通过 API 预创建 match + conversation（类似 `e2e-backend` 依赖注入测试数据），确保会话存在；
  - **路径 B**: 在点击 `chat_input_field` 前增加条件分支，若仍停留在消息列表页（assertVisible `conversation_item` 失败），则跳过发送测试并标记为 skipped。
- **依据**: `docs/design-docs/testing.md` 第 60-66 行："涉及数据库状态变更的功能，BDD/Maestro E2E 必须覆盖完整闭环：产生数据 → 查询数据 → 断言状态"

### 4. `ImBloc` 已读回执仅标记单条消息为 read，与 BDD 描述不符
- **位置**: `apps/mobile/lib/presentation/blocs/im/im_bloc.dart` 第 151-174 行
- **问题**: `_onReadReceipt` 中只有 `msg.id == event.lastReadMessageId` 的那一条消息状态被改为 `'read'`。但 BDD feature `im_read_receipts.feature` 要求："all unread messages from Alice should be marked as 'read'"（`tests/bdd/step-definitions/common.js` 第 184-185 行）。此外，UI 上虽然只显示最后一条消息的状态，但数据模型层面所有更早的已发送消息也应视为已读。
- **影响**: 若用户发送了多条消息，对方打开会话后，只有最后一条状态变为"已读"，前面的仍保持"sent"，与 IM 常规范式不符。
- **修复**:  
在 `_onReadReceipt` 中，将所有 `senderId == _currentUserId` 且 `sequenceId <= readMessageSequenceId`（或 `id == lastReadMessageId` 及之前所有消息）的状态均设为 `'read'`。
- **依据**: `tests/bdd/im_read_receipts.feature` 隐含需求 + `prd/v1-mvp.md` 第 147-152 行

## 轻微问题（Minor）

### 5. BDD Step Definitions 为空实现
- **位置**: `tests/bdd/step-definitions/im_*.steps.js`（6 个文件）
- **问题**: 所有 IM 的 BDD step 文件仅导入 `commonSteps`，而 `commonSteps` 中的 IM 相关 step 全为 `() => {}` 空函数。BDD 测试会"通过"但不验证任何业务行为。
- **影响**: BDD 验收测试流于形式，无法发现 API 回归。
- **修复**: 逐步为关键场景（发送消息、已读回执、会话列表）编写实际调用 API 的 step 实现。至少为 `im_start_chatting_after_match` 和 `im_real_time_message_delivery` 编写可运行的 steps。
- **依据**: `docs/design-docs/testing.md` 第 46-56 行："mock 实现必须与真实组件的业务契约一致"、"BDD 测试验证的是业务行为"

### 6. `ImBloc` 生命周期与 Socket 连接策略不经济
- **位置**: `apps/mobile/lib/presentation/pages/chat_page.dart` 第 72-81 行 / `app.dart` 第 72-82 行
- **问题**: `ChatPage` 每次进入都通过 `getIt<ImBloc>()`（factory）创建新 `ImBloc`，退出时 `BlocProvider` dispose 导致 `close()` 调用，进而断开 Socket。用户从聊天页返回消息列表后 Socket 断开，再次进入又重新连接。消息列表页无法实时收到 `conversation:updated` 事件。
- **影响**: 多设备/多端场景下实时性受损；频繁重连增加服务端负载。
- **修复**: 考虑将 `ImBloc` 提升为应用级单例（或至少在 `MainPage` 级别提供），在 `MatchListPage` 中监听 `conversation:updated` 以刷新列表。若 MVP 明确不追求此优化，可在 `docs/exec-plans/tech-debt-tracker.md` 中记录。
- **依据**: `docs/modules/im.md` 边界情况说明："离线补推：用户重连后，拉取 `last_ack_sequence_id` 之后的所有消息" — 暗示应维护长连接。

### 7. `PushService` 未在 IM 模块中集成
- **位置**: `apps/api/src/modules/im/im.gateway.ts`
- **问题**: `v1-mvp.md` 第 163 行要求"离线用户通过极光推送/Firebase 触发新消息提醒"，且 BDD feature `im_real_time_message_delivery.feature` 第 13-18 行断言离线时应触发推送。但 `ImGateway` / `ImService` 从未调用 `PushService.sendPush()`。
- **影响**: 离线用户收不到新消息推送。
- **修复**: 在 `ImGateway.handleSendMessage` 中，向 `user:${otherId}` 发射 `message:receive` 后，若该 room 中无在线 socket（可通过 `this.server.in(`user:${otherId}`).fetchSockets()` 判断），调用 `PushService.sendPush()` 发送离线提醒。
- **依据**: `prd/v1-mvp.md` 第 163 行 + `tests/bdd/im_real_time_message_delivery.feature`

### 8. `ImModule` 重复创建 Redis 连接
- **位置**: `apps/api/src/modules/im/im.module.ts` 第 28-31 行
- **问题**: `ImModule` 自行 `new Redis(redisConfig)` 创建 Redis 客户端，而 `MatchingModule` 也做了同样的事。这会导致两个独立的 Redis 连接。
- **影响**: 资源浪费；连接数翻倍。
- **修复**: 建议统一在 `AppModule` 或独立的 `RedisModule` 中提供 `REDIS_CLIENT`，各业务模块 `imports` 共享。若项目现有惯例如此，可列入技术债务。
- **依据**: `docs/design-docs/nestjs.md` 第 68-70 行："模块之间通过 Interface / EventEmitter 解耦"

### 9. `SwChatInputBar` 中的 `TextField` 未暴露 `onChanged` 给外部
- **位置**: `packages/design-system/lib/src/molecules/chat_input_bar.dart` 第 109-131 行
- **问题**: 组件内部用 `_controller.addListener` 监听文本变化，但没有向外部暴露 `onChanged` 回调。若父组件需要实时校验（如敏感词过滤），无法介入。
- **影响**: 扩展性受限。
- **修复**: 增加可选参数 `final ValueChanged<String>? onChanged;`，在 `_onTextChanged` 中调用 `widget.onChanged?.call(_controller.text)`。
- **依据**: 设计系统可扩展性最佳实践（非强制规范，属建议）

### 10. `send_message.yaml` 中 `assertVisible: "你好"` 可能受多条消息干扰
- **位置**: `.maestro/flows/im/send_message.yaml` 第 41 行
- **问题**: 若测试环境已存在历史消息包含"你好"，Maestro 的 `assertVisible` 可能匹配到旧消息而非刚发送的新消息。
- **影响**: E2E 假阳性。
- **修复**: 使用唯一测试文本如 `"E2E_TEST_HELLO_{{timestamp}}"`，或给最新消息气泡增加 `Semantics(identifier: 'latest_self_message')`。
- **依据**: `docs/design-docs/testing.md` 第 121-124 行：定位策略应优先使用稳定的 `resource-id`

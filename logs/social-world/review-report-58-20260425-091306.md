## 阻塞问题（Blocking）

### 1. Flutter Socket 从未连接，实时消息功能完全不可用
- **位置**: `apps/mobile/lib/app.dart:72-82`（ChatPage 路由创建 Bloc 但未连接 Socket）
- **问题**: `ChatPage` 通过 `BlocProvider` 创建 `ImBloc` 后，只调用了 `setCurrentConversation()` 和 `add(ImLoadMessages(...))`，**从未发送 `ImConnectRequested` 事件**。`ImBloc._onConnect` 因此永远不会执行，`ImSocketClient.connect()` 永远不会被调用。
- **影响**: 用户进入聊天页后，Socket.io 连接未建立，无法实时发送/接收消息；已读回执、会话更新推送全部失效。整个 IM 实时链路处于瘫痪状态。
- **修复**:  
在 `ChatPage.initState()` 或 `app.dart` 的 ChatPage 路由中，在 Bloc 创建后立即发送连接事件：
  ```dart
  // apps/mobile/lib/presentation/pages/chat_page.dart
  @override
  void initState() {
    super.initState();
    final bloc = context.read<ImBloc>();
    bloc.setCurrentConversation(widget.conversationId, widget.currentUserId);
    // 新增：连接 Socket
    final authState = context.read<AuthBloc>().state;
    if (authState is AuthAuthenticated) {
      bloc.add(ImConnectRequested(authState.token));
    }
    bloc.add(ImLoadMessages(conversationId: widget.conversationId));
  }
  ```
同时需要在 `ChatPage.dispose()`（或 Bloc 关闭前）发送 `ImDisconnectRequested`：
  ```dart
  @override
  void dispose() {
    context.read<ImBloc>().add(const ImDisconnectRequested());
    _scrollController.dispose();
    super.dispose();
  }
  ```
- **依据**: `apps/mobile/lib/presentation/blocs/im/im_bloc.dart:35-69` 中 `_onConnect` 依赖 `ImConnectRequested` 事件触发；全局搜索确认该事件在代码库中没有任何 `add()` 调用点。

## 严重问题（Major）

### 2. API 契约文档 `im.yaml` 与实际实现不符
- **位置**: `docs/api-contracts/im.yaml:7-24`
- **问题**: `/api/v1/conversations` 的响应契约定义 `data` 为 `ConversationDto` 数组：
  ```yaml
  data:
    type: array
    items:
      $ref: '#/components/schemas/ConversationDto'
  ```
但 `ImController.getConversations` 实际返回的是分页对象 `{ items, next_cursor, has_more }`（与 `common.yaml#/Pagination` 一致）。
- **影响**: 前端/外部消费者按契约生成客户端时会得到错误类型；OpenAPI 契约失去真实性。
- **修复**:  
修改 `docs/api-contracts/im.yaml` 第 17-23 行：
  ```yaml
  data:
    $ref: './common.yaml#/components/schemas/Pagination'
    # 或更精确地定义为包含 items: ConversationDto[] 的分页对象
  ```
- **依据**: `apps/api/src/modules/im/im.controller.ts:14-35` 实际返回 `Pagination` 结构；`docs/api-contracts/im.yaml` 定义为数组。

### 3. Service 层直接返回 TypeORM Entity（违反架构规范）
- **位置**: `apps/api/src/modules/im/im.service.ts:157-200`
- **问题**: `getMessages()` 的返回类型中 `items` 为 `MessageEntity[]`，Controller 直接将 Entity 序列化后返回。NestJS 编码规范明确规定："Service 禁止直接返回 Entity"。直接返回 Entity 会导致：
  1. 内部字段（如可能的未来敏感字段）意外暴露；
  2. API 响应结构与契约不一致（Entity 字段名/类型与 DTO 可能不同）；
  3. 系统消息的 `sender_id` 为 `null`，但 `im.yaml` 的 `MessageDto` 将 `sender_id` 标记为 `required`。
- **影响**: API 契约与实际响应存在结构性差异；违反项目分层原则。
- **修复**:  
在 `apps/api/src/modules/im/im.service.ts` 中新增 `MessageResponseDto`，`getMessages` 返回 `MessageResponseDto[]`：
  ```typescript
  export class MessageResponseDto {
    id: string;
    conversation_id: string;
    sender_id: string | null;
    type: string;
    content: string;
    client_message_id: string | null;
    sequence_id: number;
    status: string;
    created_at: string;
  }
  ```
并在 Service 中显式映射后再返回。同步更新 `im.yaml` 中 `sender_id` 为 `nullable: true`。
- **依据**: `docs/design-docs/nestjs.md` 分层原则表："Service 禁止直接返回 Entity"；`docs/api-contracts/im.yaml:99-102` 将 `sender_id` 标记为 required。

### 4. 发送消息后会话列表排序不更新
- **位置**: `apps/api/src/modules/im/im.service.ts:52-126`
- **问题**: `sendMessage()` 将消息写入 `messages` 表并增加 Redis 未读数，但**从未更新 `conversation_participants.updated_at`**。`getConversations()` 按 `cp.updated_at DESC` 排序，因此新消息不会导致会话在列表中置顶。
- **影响**: 用户收到新消息后，会话列表中的顺序不会变化，最近活跃的对话不会排在最前面，严重违背用户预期。
- **修复**:  
在 `sendMessage()` 中保存消息后，更新会话参与者的 `updated_at`：
  ```typescript
  // apps/api/src/modules/im/im.service.ts ~line 97
  if (otherId) {
    await this.redis.incr(`unread_count:${otherId}:${dto.conversation_id}`);
    // 新增：更新双方 participant 的 updated_at，使会话列表排序正确
    await this.participantRepo.update(
      { conversation_id: dto.conversation_id },
      { updated_at: new Date() },
    );
  }
  ```
- **依据**: `apps/api/src/modules/im/im.service.ts:230` 使用 `cp.updated_at DESC` 排序；MySQL `ON UPDATE CURRENT_TIMESTAMP` 仅在 `conversation_participants` 行被 UPDATE 时触发。

### 5. 离线推送未在 IM 模块中集成
- **位置**: `apps/api/src/modules/im/im.gateway.ts:90-142`
- **问题**: `handleSendMessage()` 通过 Socket.io 向接收者推送 `message:receive`，但未检查接收者是否在线。若接收者离线，消息只会持久化到 DB，**没有任何离线推送触发**。PRD `v1-mvp.md` 明确要求："离线用户通过极光推送/Firebase 触发新消息提醒"。`PushService` 已存在（`apps/api/src/modules/push/push.service.ts`）且已注入 `MatchingModule`，但 `ImModule` 未注入它，也未调用。
- **影响**: 离线用户收不到新消息提醒，必须主动打开 App 才能看到消息，严重影响 IM 核心体验。
- **修复**:  
  1. 在 `ImModule` 的 `providers` 中注入 `PushService`；
  2. 在 `ImGateway.handleSendMessage()` 中，若 `this.server.to(\`user:${otherId}\`).emit(...)` 后发现对方不在线（可通过 `server.in(\`user:${otherId}\`).fetchSockets()` 判断），则调用 `pushService.sendPush()` 发送推送。
- **依据**: `prd/v1-mvp.md` 技术要求章节："离线用户通过极光推送/Firebase 触发新消息提醒"；`apps/api/src/modules/push/push.service.ts` 已提供骨架实现。

## 轻微问题（Minor）

### 6. `ImBloc._onConversationUpdated` 为空实现
- **位置**: `apps/mobile/lib/presentation/blocs/im/im_bloc.dart:219-225`
- **问题**: 收到 `conversation:updated` 事件后无任何处理。虽然不会导致功能崩溃，但会话列表无法实时刷新。
- **修复**: 可选在 `ConversationListCubit` 中监听该事件并触发刷新，或至少添加 TODO 注释说明待实现。

### 7. `ChatPage` 无历史消息分页加载入口
- **位置**: `apps/mobile/lib/presentation/pages/chat_page.dart`
- **问题**: `ImMessagesLoaded` 携带 `hasMore` 和 `nextCursor`，但 UI 中没有"下拉加载更多"或"滚动到顶部自动加载"的实现。用户只能看到最近 20 条消息。
- **修复**: 在 `ListView.builder` 中增加 `NotificationListener<ScrollNotification>` 或 `ScrollController` 监听，当滚动到顶部且 `hasMore` 为 true 时触发 `ImLoadMessages(cursor: nextCursor)`。

### 8. `SendMessageDto.client_message_id` 为必填但服务端写死 fallback
- **位置**: `apps/api/src/modules/im/dto/send-message.dto.ts:14-15` 与 `apps/api/src/modules/im/im.service.ts:76`
- **问题**: DTO 用 `@IsString()` 将 `client_message_id` 设为必填，但 Service 中写了 `dto.client_message_id ?? uuidv7()` 的防御逻辑。由于 DTO 校验会拦截空值，这段 fallback 永远不会执行，属于无效防御代码。
- **修复**: 保持必填即可，删除 Service 中的无意义 fallback，或改为在 Gateway 层校验缺失时生成。

### 9. `im-events.yaml` 未在 `im.yaml` 中引用
- **位置**: `docs/api-contracts/im.yaml`
- **问题**: `im-events.yaml` 单独维护了 WebSocket 事件契约，但 `im.yaml` 未通过 `$ref` 引用它。两个文件处于孤岛状态。
- **修复**: 在 `im.yaml` 底部增加注释或 `components/schemas` 引用，说明 WebSocket 事件契约见 `im-events.yaml`。

## 风险与建议

1. **实时消息是 IM 的生死线**: 阻塞问题 #1 必须在合并前修复并验证。建议在修复后，使用两个模拟器同时登录两个测试账号，端到端验证发送→接收→已读回执的全链路。

2. **会话列表排序是体验核心**: 问题 #4 会导致"最新消息不置顶"，用户极易感知。建议一并修复，否则上线后用户反馈会很差。

3. **契约一致性需要回归**: 问题 #2 和 #3 涉及 API 契约与代码的一致性。建议引入 `make check-docs` 或类似工具，在 CI 中校验 OpenAPI 契约与实际 Controller 返回结构的匹配度。

4. **离线推送可作为 follow-up**: 问题 #5 影响体验但属于 MVP 预留能力（`PushService` 本身就是 skeleton）。如果工期紧张，可在合并后单独开 Issue 跟进，但建议在本 PR 中至少将 `PushService` 注入 `ImModule` 并加上 TODO。

5. **加载历史消息**: 问题 #7 是体验缺口，但 20 条消息的 MVP 场景下风险可控，可作为后续迭代优化。

## 审查结论

**审查结论: NEEDS_MAJOR_FIX**

PR 实现了 IM 模块的核心功能（消息收发、已读回执、会话列表、Socket.io 实时推送），整体架构清晰、单元测试覆盖较完整。但存在 **2 个 Blocking 问题**（运行时崩溃 + SQL 查询错误）和 **3 个 Major 问题**（契约不一致、E2E 路径错误、类型不匹配），必须在合并前修复。

## 阻塞问题（Blocking）

### 1. MatchListPage 中 BlocListener<ImBloc> 缺少 Provider，导致运行时崩溃
- **位置**: `apps/mobile/lib/presentation/pages/match_list_page.dart:59`
- **问题**: `MatchListPage` 使用了 `BlocListener<ImBloc, ImState>` 来监听 `ImConversationListRefreshRequested` 事件并刷新会话列表，但 `MainPage`（`apps/mobile/lib/app.dart:139`）的 `MultiBlocProvider` 中只注册了 `MatchingBloc` 和 `MatchListCubit`，**没有注册 `ImBloc`**。`flutter_bloc` 的 `BlocListener` 会通过 `BlocProvider.of<T>(context)` 向上查找 Bloc，找不到时会抛出 `BlocProviderNotFoundException`，导致用户进入「消息」Tab 时直接崩溃。
- **影响**: 所有用户都无法进入会话列表页面，App 必现崩溃。
- **修复**:
在 `app.dart` 的 `MainPage.build` 中 `MultiBlocProvider` 添加 `ImBloc` 注册：
  ```dart
  MultiBlocProvider(
    providers: [
      BlocProvider(create: (_) => getIt<MatchingBloc>()),
      BlocProvider(create: (_) => getIt<MatchListCubit>()),
      BlocProvider(create: (_) => getIt<ImBloc>()), // 新增
    ],
    ...
  )
  ```
- **依据**: `flutter_bloc` 官方行为 + 代码 diff 中 `MatchListPage` 对 `ImBloc` 的依赖证据。

### 2. 原生 SQL 数组参数绑定错误，导致会话列表查询失败
- **位置**: `apps/api/src/modules/im/im.service.ts:300-311`
- **问题**: `getConversations` 方法使用原生 SQL 查询最新消息：
  ```typescript
  WHERE conversation_id IN (?)
  ```
参数传入的是 `[conversationIds]`（`string[]` 数组）。本项目使用 **mysql2 驱动**，其原生 `query()` 方法**不会自动将单个 `?` 展开为 `IN (?, ?, ?)`**。mysql2 会将整个数组序列化为一个字符串值（如 `"id1,id2,id3"`），导致 SQL 语义错误，查询结果为空或直接报错。
- **影响**: 当用户有 2 个及以上会话时，`getConversations` API 返回的 `last_message` 全部为空，会话列表无法显示最后一条消息摘要；在部分 SQL 模式下会直接抛出语法错误。
- **修复**:
手动生成与数组长度匹配的占位符：
  ```typescript
  const placeholders = conversationIds.map(() => '?').join(',');
  const msgResults = await this.dataSource.query(
    `
    SELECT m.* FROM messages m
    INNER JOIN (
      SELECT conversation_id, MAX(sequence_id) as max_seq
      FROM messages
      WHERE conversation_id IN (${placeholders})
      GROUP BY conversation_id
    ) latest ON m.conversation_id = latest.conversation_id AND m.sequence_id = latest.max_seq
  `,
    conversationIds,
  );
  ```
- **依据**: `docs/design-docs/db.md` 第 91-152 行「原生 SQL 查询规范」中明确说明使用 `?` 位置参数，但未提及数组自动展开；结合 mysql2 驱动实际行为。

## 严重问题（Major）

### 3. OpenAPI 契约中 MessageDto.type 枚举不完整
- **位置**: `docs/api-contracts/im.yaml:103-104`
- **问题**: `MessageDto` 的 `type` 字段枚举仅定义为 `[text, image]`，但后端 `ImService.createSystemMessage()` 会生成 `type = 'system'` 的消息，且 `GET /messages` 接口会将这些 system 消息返回给客户端。`im-events.yaml` 中 `message:receive` 事件的 type 枚举已经正确包含 `[text, image, system]`，但 REST 契约 `im.yaml` 与之不一致。
- **影响**: 契约与实际响应不一致，违反「契约优先」原则；`openapi-generator` 生成的 Dart/TS 模型会对 `system` 值反序列化失败或丢弃。
- **修复**:
在 `docs/api-contracts/im.yaml` 的 `MessageDto.type` 中增加 `system`：
  ```yaml
  type:
    type: string
    enum: [text, image, system]
  ```
同步运行 `make generate-types` 更新生成代码。
- **依据**: `docs/design-docs/api.md` 第 82-83 行「契约优先：接口变更必须先在 YAML 中定义，再生成代码」；`prd/v1-im.md` 第 52-54 行技术要求中明确 `type` 包含 `text`/`image`/`system`。

### 4. Maestro E2E runFlow 相对路径错误
- **位置**: `.maestro/flows/im/send_message.yaml:18`、`.maestro/flows/im/view_conversation_list.yaml:15`
- **问题**: 两个 IM E2E flow 都包含：
  ```yaml
  - runFlow: includes/login-as-test-user.yaml
  ```
Maestro 的 `runFlow` 路径**相对于当前 flow 文件所在目录**解析。从 `.maestro/flows/im/send_message.yaml` 出发，该路径指向 `.maestro/flows/im/includes/login-as-test-user.yaml`，但实际文件位于 `.maestro/includes/login-as-test-user.yaml`（少了一层 `flows/im/`）。
- **影响**: Maestro E2E 测试无法找到被引用的子 flow，测试直接失败。
- **修复**:
将路径修正为：
  ```yaml
  - runFlow: ../../includes/login-as-test-user.yaml
  ```
- **依据**: Maestro 官方文档中 `runFlow` 的路径解析规则 + `.maestro/` 目录结构证据。

### 5. shared-types 中 MessageReceivedEvent.sender_id 类型不匹配
- **位置**: `packages/shared-types/src/im-socket-events.ts:13`
- **问题**: `MessageReceivedEvent` 接口定义 `sender_id: string`，但数据库 `messages.sender_id` 已改为 `nullable`（`string | null`），且 `im.gateway.ts` 在发送 system 消息时实际传递 `message.sender_id`（值为 `null`）。TypeScript 类型与实际运行时值不匹配。
- **影响**: 类型系统失去保护作用，前端消费时若按 `string` 处理可能引发空指针异常。
- **修复**:
  ```typescript
  export interface MessageReceivedEvent {
    id: string;
    conversation_id: string;
    sender_id: string | null; // 修正
    ...
  }
  ```
- **依据**: `apps/api/src/database/entities/message.entity.ts:23-24` 中 `sender_id` 已声明为 `string | null` + `im.gateway.ts:113` 实际传递 `message.sender_id`（system 消息时为 null）。

## 轻微问题（Minor）

### 6. IM BLoC 中消息列表无上限，存在内存泄漏风险
- **位置**: `apps/mobile/lib/presentation/blocs/im/im_bloc.dart:15`
- **问题**: `_messages` 是 `ImBloc` 的类级列表，每次加载历史消息时 `insertAll(0, result.items)` 会无限增长。长时间聊天或频繁加载历史消息后，内存占用会持续上升。
- **影响**: 长会话场景下内存占用增加，极端情况下可能导致 OOM。
- **修复建议**: 限制内存缓存消息数量（如最多保留 500 条），或引入本地数据库（sqflite）缓存，超出部分从本地读取。
- **依据**: `prd/v1-im.md` 第 196 行提到「客户端本地缓存：Mobile 使用本地数据库缓存最近 N 条消息」。

### 7. Socket 客户端缺少显式重连策略配置
- **位置**: `apps/mobile/lib/core/network/im_socket_client.dart:30-34`
- **问题**: `socket_io_client` 虽然默认有自动重连，但 PRD `v1-im.md` 明确要求「Mobile 侧配置 5 次重连，退避策略」。当前代码未显式配置 `reconnectionAttempts` 和 `reconnectionDelay`。
- **影响**: 弱网环境下重连行为不可控，可能导致频繁重连或重连次数不足。
- **修复建议**:
  ```dart
  _socket = io.io(namespaceUrl, <String, dynamic>{
    'transports': ['websocket'],
    'autoConnect': true,
    'auth': <String, dynamic>{'token': token},
    'reconnectionAttempts': 5,
    'reconnectionDelay': 1000,
    'reconnectionDelayMax': 5000,
  });
  ```
- **依据**: `prd/v1-im.md` 第 246 行「Socket.io 连接支持自动重连（Mobile 侧配置 5 次重连，退避策略）」。

### 8. Migration down 方法在存在 NULL 数据时会失败
- **位置**: `apps/api/src/database/migrations/20250425000000-AlterMessagesSenderIdNullable.ts:17-27`
- **问题**: `down()` 方法先将 `sender_id` 改回 `NOT NULL`，再重建外键。如果回滚时 `messages` 表中已存在 `sender_id = NULL` 的 system 消息，`MODIFY sender_id CHAR(36) NOT NULL` 会因数据冲突而失败。
- **影响**: Migration 无法回滚，违反「Migration 必须可回滚」原则。
- **修复建议**: 在 `down()` 中先清理或更新 `sender_id IS NULL` 的记录（如设置默认值或删除 system 消息），再执行 `MODIFY`。
- **依据**: `docs/design-docs/db.md` 第 89-90 行「Migration 必须可回滚：每个 Migration 的 `down()` 方法必须正确实现」。

## 风险与建议

1. **运行时崩溃风险（高）**: `MatchListPage` 的 `BlocListener<ImBloc>` 是必现崩溃点，必须优先修复。
2. **数据查询风险（高）**: `getConversations` 的原生 SQL 数组参数问题会导致生产环境会话列表功能异常，需立即修正。
3. **契约漂移风险（中）**: `im.yaml` 与 `im-events.yaml` 对 `type` 枚举的定义不一致，随着 system 消息场景的增多，前后端类型冲突会加剧。建议在修复后重新运行 `make generate-types` 并提交生成代码。
4. **E2E 可维护性（中）**: Maestro flow 的子流程引用路径错误说明 E2E 脚本未经实际运行验证。建议在 CI 中增加 Maestro flow 语法检查步骤（如 `maestro test --dry-run`）。
5. **Migration 回滚风险（低）**: 当前 Migration 的 `down()` 在含有 system 消息时无法回滚。MVP 阶段 system 消息量小，风险可控，但应在技术债务中记录。

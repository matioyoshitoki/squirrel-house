## 风险与建议

1. **Redis 故障导致数据不一致**: `ImService.sendMessage` 中 Redis `incr` 位于数据库事务之外，且无独立的 try-catch。若 Redis 在事务提交后故障，`incr` 会抛出异常，发送方收到 `message:error`，但消息已持久化且接收方下次加载时仍能看到该消息（只是未读数未增加）。建议将 Redis 操作纳入错误降级逻辑，而非直接抛异常。

2. **消息同步过滤自身消息**: `ImService.getMessagesSince` 通过 `.andWhere("m.sender_id != :userId", { userId })` 过滤了用户自己发送的消息。若用户从另一设备发送消息，当前设备重连后无法通过 `message:sync` 补拉到这些自消息。MVP 阶段可接受，但建议在 `docs/modules/im.md` 中明确记录此限制。

3. **未读数无过期/上限策略**: Redis 中 `unread_count:{user_id}:{conversation_id}` 只增不减（除已读时重置为 0），长期运行可能导致 Redis 中存在大量冷 key。建议后续增加过期策略或定期清理。

# PR #58 代码审查报告

## 审查结论

审查结论: **NEEDS_MAJOR_FIX**

PR 实现了 IM（即时通讯）模块的核心功能，包含后端 Socket.io Gateway、消息服务、前端聊天页面/会话列表、Design System 组件及 E2E 测试。整体架构方向正确，设计系统组件符合规范，Maestro E2E 测试覆盖到位。

但存在 **2 个 Major 问题**：
1. Mobile 端 ImBloc 的已读回执逻辑存在边界 bug，会导致所有己方消息被错误标记为已读；
2. 后端离线推送未集成 PushService，PRD 要求的离线消息推送未实现；
3. BDD 测试只有 feature 文件，缺少可执行的 step 实现。

## 阻塞问题（Blocking）

无。

## 严重问题（Major）

### 1. ImBloc 已读回执逻辑错误标记所有己方消息为已读
- **位置**: `apps/mobile/lib/presentation/blocs/im/im_bloc.dart:153-163`
- **问题**: 当 `_onReadReceipt` 收到 `lastReadMessageId` 且该消息 ID 不在当前内存中的 `_messages` 列表时（`lastReadIndex == -1`），条件 `shouldMarkRead = lastReadIndex == -1 || i <= lastReadIndex` 会成立，导致**所有己方消息**被标记为 `read`。这在用户未加载全部历史消息时（例如只加载了最近 20 条，但对方已读了更早的消息），会错误地将当前可见的较新消息也标为已读。
- **影响**: 消息发送方的 UI 上，未读的新消息会错误显示“已读”状态，用户体验受损。
- **修复**:
  ```dart
  // 仅当 lastReadMessageId 存在于当前消息列表时才批量标记
  final shouldMarkRead = lastReadIndex != -1 && i <= lastReadIndex;
  ```
或者，如果希望在 lastReadMessageId 不在列表时也标记（例如列表中只有更早消息），需要更精确地比较 sequence_id 而不是列表索引。
- **依据**: `prd/v1-mvp.md` US-4 要求“Alice should see the message status change to '已读'”，而当前实现会导致错误的消息被标记；`tests/bdd/im_read_receipts.feature` 也要求仅已读消息状态变化。

### 2. 离线消息推送未实现（PushService 未在 IM Gateway 中调用）
- **位置**: `apps/api/src/modules/im/im.gateway.ts:90-142`
- **问题**: `handleSendMessage` 在消息保存后，仅通过 `this.server.to(\`user:${otherId}\`)` 向接收者推送 `message:receive` 事件。若接收者不在线（Socket 未连接），消息只会被持久化且 Redis unread_count 增加，但**不会触发任何推送通知**。`PushService` 已在 `apps/api/src/modules/push/push.service.ts` 中定义，但 `ImGateway` 未注入和使用它。
- **影响**: 离线用户无法收到新消息提醒，违反 PRD 中“离线用户通过极光推送/Firebase 触发新消息提醒”的验收标准。
- **修复**:
  1. 在 `ImGateway` 构造函数中注入 `PushService`。
  2. 在 `handleSendMessage` 中，emit 到 `user:${otherId}` 后检查该 room 内是否有在线 socket，若无则调用 `pushService.sendPush(...)`。
  ```typescript
  // 在 ImGateway 构造函数中添加
  private readonly pushService: PushService,

  // 在 handleSendMessage 中 emit 后添加
  const sockets = await this.server.in(`user:${otherId}`).fetchSockets();
  if (sockets.length === 0) {
    await this.pushService.sendPush(
      otherId,
      '新消息',
      '您收到一条新消息',
      { conversation_id: dto.conversation_id },
    );
  }
  ```
- **依据**: `prd/v1-mvp.md` 中 US-4 技术要求明确提到“离线用户通过极光推送/Firebase 触发新消息提醒”；`tests/bdd/im_real_time_message_delivery.feature` 第 13-18 行要求“a push notification should be triggered for Bob”。

### 3. BDD 测试缺少可执行的 Step 实现
- **位置**: `tests/bdd/step-definitions/im_*.steps.js`
- **问题**: 所有 IM 相关的 BDD step 定义文件（如 `im_start_chatting_after_match.steps.js`、`im_read_receipts.steps.js` 等）均使用 `autoBindSteps(feature, [commonSteps])`，即**仅依赖通用步骤**，没有针对 IM 业务行为（如 Socket.io 连接、消息发送、已读回执、未读数验证）编写具体的 Given/When/Then 实现。这意味着这些 `.feature` 文件虽然存在，但无法执行有效的验收测试。
- **影响**: BDD 测试无法验证 IM 核心业务流程的闭环（写 → 读 → 断言），违背 `docs/design-docs/testing.md` 中“数据闭环原则”。
- **修复**: 为每个 IM feature 编写具体的 step 实现，至少覆盖：
  - 建立两个测试用户的 Socket.io 连接（`im` namespace）
  - 发送消息并断言对方收到 `message:receive` 事件
  - 断言 Redis `unread_count` 变化
  - 发送 `message:read` 并断言对方收到 `message:read_receipt`
- **依据**: `docs/design-docs/testing.md` 第 56-67 行强调“数据闭环原则”和“mock 实现必须与真实组件的业务契约一致”；`prd/v1-mvp.md` 中每个用户故事均要求 BDD 覆盖。

## 轻微问题（Minor）

### 4. API 契约字段名与控制器返回值不一致
- **位置**: `docs/api-contracts/im.yaml:56-74`
- **问题**: `ConversationDto` 中定义的字段为 `id`，而 `ImController.getConversations` 实际返回的字段名为 `conversation_id`。Mobile 端解析代码 (`im_repository_impl.dart:99`) 使用的是 `conversation_id`，与控制器一致，但与 OpenAPI 契约不一致。
- **修复**: 将 `docs/api-contracts/im.yaml` 中 `ConversationDto` 的 `id` 改为 `conversation_id`。
- **依据**: `docs/design-docs/nestjs.md` 第 86 行要求“DTO 的字段命名必须与 `packages/shared-types/` 中的接口定义完全一致”。

### 5. ChatPage 空消息列表时固定显示系统提示语
- **位置**: `apps/mobile/lib/presentation/pages/chat_page.dart:118-124`
- **问题**: 当消息列表为空时，直接显示固定的系统消息“你们已匹配成功，开始聊天吧”。但如果会话有消息后被清空（如未来实现删除功能），或用户进入一个有历史消息的会话但加载失败，这个固定文案不准确。
- **修复**: 将系统消息的显示与“消息列表为空”状态解耦，仅在会话确实没有任何消息（包括系统消息）时显示该欢迎语，或从后端拉取真正的 system message。
- **依据**: `prd/v1-mvp.md` US-4 要求“Alice should see the system message”，但并未要求在没有系统消息时也显示该文案。

### 6. ImBloc 语义状态混用
- **位置**: `apps/mobile/lib/presentation/blocs/im/im_bloc.dart:117-118`
- **问题**: 当收到对方消息时（`_onMessageReceived`），emit 的是 `ImMessageSent` 状态。该状态语义上表示“我发送的消息已确认”，却被用于“收到新消息”场景，状态语义混乱。
- **修复**: 建议新增 `ImMessageReceived` 状态，或在 UI 层统一用 `ImMessagesLoaded` 表示消息列表更新。
- **依据**: `docs/design-docs/flutter.md` 要求状态管理保持可读性。

### 7. `getMessages` 分页 next_cursor 逻辑缺少注释
- **位置**: `apps/api/src/modules/im/im.service.ts:189-199`
- **问题**: `nextCursor` 取的是 `messages[0].sequence_id`，而 `messages` 经过 `reverse()` 后 `messages[0]` 是本次返回的最旧消息。该逻辑虽然结果正确，但非常反直觉，缺少注释说明，未来维护者容易误改。
- **修复**: 在 `nextCursor` 赋值处添加注释，说明 cursor 是“当前页最旧消息的 sequence_id，用于下一页加载更旧的消息”。

## 风险与建议

1. **推送降级策略**: 当前 `PushService` 是 MVP skeleton（仅打印日志）。建议在 `ImGateway` 中集成时，将推送调用包裹在 try/catch 中，确保推送失败不影响消息发送主流程。

2. **Socket 房间在线状态检查**: `ImGateway` 中判断用户是否在线可使用 `this.server.in(\`user:${userId}\`).fetchSockets()`，但需注意这会增加一次异步 IO。如果推送是高频场景，可考虑维护一个内存中的在线用户映射（如 Redis / 进程内 Map）来优化。

3. **消息加载分页边界**: `ChatPage` 目前只在 `initState` 时加载一次消息，未实现下拉加载更多历史消息。虽然当前 PR 范围可能不包含，但建议在下个迭代补充 `ListView` 的 `onNotification` 滚动监听以触发分页加载。

4. **BDD 测试闭环**: 强烈建议在下个迭代中补齐 IM 相关 BDD 的 step 实现，尤其是“发送 → 查询会话列表 → 断言未读数”的完整链路。当前只有 feature 文件无法提供回归保护。

5. **Design-Only PR 流程回顾**: 本次 PR 同时包含了 design-system 组件（`SwChatBubble`、`SwChatInputBar`、`SwConversationListItem`）和业务逻辑代码。根据 `AGENTS.md` 中 Flutter 规范，Design-Only PR 应先走且“禁止直接写入 `packages/`”。本次 PR 可能是功能开发 PR 直接合入了 design-system 组件（这在执行计划允许范围内，因为规范说“后续功能开发 PR 统一迁入”），但建议确认是否已有对应的 `designs/issue-5/MANIFEST.md` 记录此次迁移。

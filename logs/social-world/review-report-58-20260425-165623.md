## 审查结论

审查结论: **NEEDS_MAJOR_FIX**

PR #58 实现了 Issue #5（V1 即时通讯系统），覆盖范围完整：DB Entity + Migration、REST API、Socket.io Gateway、Flutter BLoC / Page、Design-System 组件、Maestro E2E、BDD feature 文件及后端单元测试。整体架构遵循项目规范，契约同步工作到位。

但存在一个 **Blocking** 运行时崩溃缺陷（`MatchListPage` 无法找到 `ImBloc`），以及多个 **Major** 级功能/状态管理问题（Socket 生命周期管理不当、消息状态 "delivered" 从未被设置、跨会话消息丢失）。这些问题必须在合并前修复。

## 阻塞问题（Blocking）

### 1. MatchListPage 运行时崩溃：ProviderNotFoundException for ImBloc
- **位置**: `apps/mobile/lib/presentation/pages/match_list_page.dart:59`
- **问题**: `MatchListPage` 的 `build` 方法直接使用了 `BlocListener<ImBloc, ImState>`，但 `ImBloc` 并未注入到 `MatchListPage` 所在的 Widget 树中。`MainPage` 只提供了 `MatchingBloc`、`MatchListCubit`、`ConversationListCubit`；`SocialWorldApp` 只提供了 `AuthBloc`。`ImBloc` 仅在 `/chat` 路由的 `BlocProvider` 中局部创建。因此进入「消息」Tab 时 Flutter 会抛出 `ProviderNotFoundException`，App 直接崩溃。
- **影响**: 用户无法进入会话列表页面，IM 核心流程完全不可用。
- **修复**:
方案 A（推荐，最小改动）：
  1. 将 `ImBloc` 的注册方式从 `registerFactory` 改为 `registerLazySingleton`（或在 `app.dart` 层级单独维护一个全局 Socket 连接）。
  2. 在 `_SocialWorldAppBody.build()` 中，用 `BlocProvider(create: (_) => getIt<ImBloc>())` 包裹 `MaterialApp.router`，使其全局可访问。
  3. 移除 `ChatPage` 路由中局部的 `BlocProvider(create: (_) => getIt<ImBloc>())`，改为直接使用 `context.read<ImBloc>()`。
  4. 在 `ChatPage.dispose()` 中**不要**调用 `ImDisconnectRequested`，改为在 `ImBloc` 内部通过引用计数或生命周期管理来决定何时真正断开 Socket（避免每次进出聊天页都断连重连）。
- **依据**: `flutter.md` 规范要求 "状态管理使用 `flutter_bloc`（Cubit 为主，BLoC 为辅）"，并强调依赖注入。当前注入范围与消费范围不匹配，属于架构级连通性缺陷。通过 `git diff` 中 `app.dart` 与 `match_list_page.dart` 的代码交叉验证可确认该问题。

## 严重问题（Major）

### 2. Socket 连接生命周期管理不当：频繁断连重连
- **位置**: `apps/mobile/lib/presentation/pages/chat_page.dart:57`
- **问题**: `ChatPage.dispose()` 发送 `ImDisconnectRequested`，导致每次离开聊天页都会断开 Socket。由于 `ImBloc` 是 factory 创建（每进一次聊天页新建一个），用户从会话列表进入聊天 A、返回列表、再进入聊天 B，会经历「连接→断开→连接」两次完整握手。这不仅浪费资源，还会丢失 `conversation:updated` 等实时事件（因为列表页没有活跃的 Socket 连接）。
- **影响**: 实时消息推送在会话列表页不可用；用户体验差，电量/流量消耗增加。
- **修复**:
  1. 按 Blocking 问题 #1 的修复方案，将 `ImBloc` 提升为全局单例，Socket 在登录后保持长连接，仅在注销或 App 后台时断开。
  2. `ChatPage.dispose()` 中移除 `ImDisconnectRequested`，改为仅重置当前会话 ID（`ImSetCurrentConversation` 传 `null`）。
- **依据**: `docs/modules/im.md` 流程图显示 Socket 连接是全局通信通道，不应随页面生命周期频繁重建。`prd/v1-mvp.md` 要求 "消息发送到达时间 < 1s"，频繁断连会显著增加延迟。

### 3. 跨会话消息在本地被静默丢弃
- **位置**: `apps/mobile/lib/presentation/blocs/im/im_bloc.dart:142`
- **问题**: `_onMessageReceived` 中，如果收到的消息不属于 `_currentConversationId`，直接 `return`，既不缓存也不转发。虽然 `conversation:updated` 会触发列表刷新，但当用户稍后点进该会话时，必须等待 API 拉取才能看到这条消息。在弱网环境下，用户可能短暂看到旧消息列表，体验不佳。更严重的是，如果 `conversation:updated` 事件丢失或列表刷新失败，用户完全不知道收到了新消息。
- **影响**: 弱网/边缘场景下实时消息感知延迟；违反 PRD "Bob should receive the message within 1 second" 的实时性预期。
- **修复**:
  1. 在 `ImBloc` 中维护一个按 `conversationId` 分级的消息缓冲池（`Map<String, List<MessageItem>>`），收到非当前会话消息时写入对应缓冲区。
  2. 当用户切换会话（`ImSetCurrentConversation`）时，将该缓冲区的消息合并到 `_messages` 并清空缓冲区。
  3. 或者至少将非当前会话消息计入 `ConversationListCubit` 的未读/最后一条消息状态，确保列表页能立即感知。
- **依据**: `prd/v1-mvp.md` US-4 验收标准 "Bob should receive the message within 1 second" 要求接收方实时感知，不应依赖 API 轮询。

### 4. 消息状态 "delivered" 从未被实际设置
- **位置**: `apps/mobile/lib/presentation/blocs/im/im_bloc.dart`, `apps/api/src/modules/im/im.gateway.ts:88-148`
- **问题**: `MessageDto.status` 枚举包含 `sending`, `sent`, `delivered`, `read` 四个状态。当前实现中：
  - `sending`：客户端发送前设置 ✅
  - `sent`：收到服务端 `message:ack` 后设置 ✅
  - `read`：收到 `message:read_receipt` 后设置 ✅
  - `delivered`：**从未被设置**。服务端在 `message:receive` 中只推送给接收方，没有回传任何事件给发送方来标记 "已送达"。
- **影响**: UI 中 `SwChatBubble` 的 "已送达" 状态永远不会显示；PRD 验收标准 "the message status should progress from 'sending' to 'sent' to 'delivered'" 未满足。
- **修复**:
  1. 在 `ImGateway.handleSendMessage` 中，向接收方 emit `message:receive` 后，增加一个 `message:delivered` 事件回传给发送方；或
  2. 在 `ImSocketClient` / `ImBloc` 中，将 `message:ack` 同时视为 "sent"，然后在收到对方已读之前保持 "sent" 状态，移除 "delivered" 的中间状态以简化模型（需同步修改 `im.yaml` 和 `im-events.yaml`）。
- **依据**: `prd/v1-mvp.md` 明确验收标准包含 sent → delivered → read 的状态流转。`docs/api-contracts/im-events.yaml` 中缺少 `message:delivered` 事件定义。

### 5. `sendMessage` DTO 的 `@MaxLength(500)` 对图片类型也生效
- **位置**: `apps/api/src/modules/im/dto/send-message.dto.ts:7-9`
- **问题**: `SendMessageDto.content` 被无条件限制 `@MaxLength(500)`。当 `type = "image"` 时，`content` 通常存储图片 URL，长度可能超过 500（尤其是带签名的 OSS URL）。这会导致图片消息被服务端拒绝。
- **影响**: 图片消息发送失败，违反 PRD "MVP 阶段支持：文本、图片"。
- **修复**:
  ```typescript
  @ValidateIf((o) => o.type === "text")
  @MaxLength(500)
  content: string;
  ```
或在校验管道中做条件校验。
- **依据**: `prd/v1-mvp.md` 明确 MVP 支持文本和图片两种消息类型。

### 6. 缺少 Flutter 层 BLoC / Cubit 单元测试
- **位置**: `apps/mobile/test/`
- **问题**: PR 在 `apps/mobile/test/` 下未新增任何 IM 相关的单元测试或 Widget 测试。`ImBloc` 包含复杂的状态机（连接、发送、接收、ACK、已读回执、重试、加载历史消息），`ConversationListCubit` 也包含分页和异常处理逻辑，但均无测试覆盖。
- **影响**: 后续重构或新增功能时极易引入回归缺陷。
- **修复**:
  1. 新增 `apps/mobile/test/presentation/blocs/im_bloc_test.dart`，覆盖以下场景：
     - 连接/断开 Socket
     - 发送消息（成功、失败、重试）
     - 接收消息（当前会话 / 其他会话）
     - ACK 更新状态
     - 已读回执更新状态
     - 加载历史消息（首次加载、分页加载）
     - 缓存上限裁剪
  2. 新增 `apps/mobile/test/presentation/blocs/conversation_list_cubit_test.dart`，覆盖加载、分页、异常态。
- **依据**: `docs/design-docs/testing.md` 要求 "纯业务逻辑、Cubit/Bloc：Unit test"。

## 轻微问题（Minor）

### 7. `getConversations` 分页 cursor 使用 `updated_at` 存在精度冲突风险
- **位置**: `apps/api/src/modules/im/im.service.ts:251-255`
- **问题**: 会话列表分页以 `conversation_participants.updated_at` 作为 cursor。如果同一用户在极短时间内（同一毫秒）有两个会话被更新（例如同时收到两条消息），下一页可能跳过或重复记录。
- **影响**: 极低概率下的分页边界异常。
- **修复**: 使用复合 cursor `(updated_at, conversation_id)` 做分页，或改用 `participant.id`（自增主键）作为稳定 cursor。
- **依据**: `docs/design-docs/api.md`（WIP）中关于游标分页的稳定性最佳实践。

### 8. `im.yaml` 中 `last_message` 标记为 required 但与实现不一致
- **位置**: `docs/api-contracts/im.yaml:58-62`
- **问题**: `ConversationDto` 的 `required` 列表包含 `last_message`，但 `ImService.getConversations` 的实现中 `last_message` 可能为 `null`（当会话没有任何消息时）。虽然当前种子数据总是插入 system message，但生产环境中如果 `match:created` 事件处理失败，该字段会为 `null`。
- **影响**: 客户端生成的 OpenAPI 模型可能因反序列化 null 值而抛出异常。
- **修复**: 将 `last_message` 从 `required` 列表中移除，并在 `packages/shared-types` 和 `apps/mobile/lib/models/generated/im/` 中重新生成类型。
- **依据**: 实际代码返回类型 `last_message: ... | null` 与 OpenAPI 契约的 `required` 声明冲突。

### 9. Maestro E2E flow 中关键断言标记为 `optional: true`
- **位置**: `.maestro/flows/im/send_message.yaml:25-33`, `.maestro/flows/im/view_conversation_list.yaml:25-38`
- **问题**: `send_message.yaml` 中 "点击进入聊天页" 和 "验证返回按钮" 等步骤标记为 `optional: true`。如果测试环境没有预置会话，这些步骤被跳过，但后续非 optional 步骤（如 `chat_input_field`）会因页面未跳转而失败。更合理的做法是让这些步骤非 optional，这样失败时能更快定位根因（缺少测试数据）。`view_conversation_list.yaml` 的空状态断言也全部 optional，导致该 flow 在无论空态还是列表态都会 "通过"，失去断言价值。
- **影响**: E2E 测试的故障定位能力下降；可能产生 false positive。
- **修复**: 
  - `send_message.yaml` 中，将会话项点击改为非 optional，并前置一个明确的断言确保测试数据已注入。
  - `view_conversation_list.yaml` 中，空状态与列表项的断言应至少有一组是非 optional 的（根据测试环境预期状态决定）。
- **依据**: `docs/design-docs/testing.md` 要求 "E2E 测试必须覆盖完整闭环"，且 "改动点必须被对应层级的测试覆盖到"。全部 optional 的断言无法覆盖任何闭环。

### 10. `PushService` 仅为骨架实现，但 `MatchingModule` 和 `ImModule` 均依赖 `PushModule`
- **位置**: `apps/api/src/modules/push/push.service.ts`, `apps/api/src/modules/matching/matching.module.ts`, `apps/api/src/modules/im/im.module.ts`
- **问题**: `PushService.sendPush` 仅打印日志，未接入真实推送通道（JPush / FCM）。虽然 MVP 允许 skeleton，但 `PushModule` 被两个核心模块导入，意味着后续替换实现时需要同时修改多处。这属于已识别的技术债务，应在 `tech-debt-tracker.md` 中记录。
- **影响**: 离线用户无法收到新消息提醒；未来替换推送服务商时改动面大。
- **修复**: 在 `docs/exec-plans/tech-debt-tracker.md` 中记录："PushService 为 MVP skeleton，需在 M4 联调阶段替换为真实推送 SDK"。
- **依据**: `prd/v1-mvp.md` 技术要求 "离线用户通过极光推送/Firebase 触发新消息提醒"。

## 风险与建议

1. **架构风险：IM Socket 生命周期绑定页面**
当前 `ImBloc` 是 factory 创建、绑定 `ChatPage` 生命周期，导致 Socket 无法为会话列表提供实时更新。建议将 Socket 连接提升至 App 级别，由全局 `ImBloc` 或专门的 `ImSocketManager` 单例管理。`ChatPage` 只负责发送/接收当前会话的事件，不管理连接状态。

2. **数据一致性风险：`updated_at` 分页**
会话列表分页依赖 `updated_at` 时间戳。在高并发消息场景下，同一毫秒内的多条消息可能导致分页不稳定。建议后续迭代引入复合游标或自增 ID 游标。

3. **消息丢失风险：跨会话消息未缓冲**
当前实现依赖 `conversation:updated` + API 轮询来感知非当前会话的新消息。如果 Socket 在会话列表页未连接（当前确实未连接），用户必须手动下拉刷新才能看到最新消息。建议全局 Socket 常驻，并在本地按会话缓存消息摘要。

4. **技术债务记录**
   - `PushService` 为 skeleton，需接入真实推送。
   - `ImBloc` 缺少 Flutter 单元测试，需在 M4 补充。
   - `delivered` 状态缺失，需补充 `message:delivered` 事件或简化状态机。

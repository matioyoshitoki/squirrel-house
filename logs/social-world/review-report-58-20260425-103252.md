## 阻塞问题（Blocking）

### 1. ChatPage 加载历史消息时自动跳回底部，导致分页功能完全不可用
- **位置**: `apps/mobile/lib/presentation/pages/chat_page.dart:133-136`
- **问题**: `BlocConsumer` 的 `listener` 在 `ImMessagesLoaded` 状态时调用 `_scrollToBottom()`。当用户上滑到顶部加载更早消息时，`ImLoadMessages` 触发分页请求，返回后 emit `ImMessagesLoaded`，listener 立即将列表滚动到最底部。用户无法阅读刚加载的历史消息。
- **影响**: 聊天页面的「加载更多」功能完全不可用，核心 IM 体验断裂。
- **修复**: 修改 listener，仅在**初始加载**或**新消息到达**时滚动到底部，分页加载时不滚动：
  ```dart
  // chat_page.dart ~133
  listener: (context, state) {
    if (state is ImMessageSent) {
      _scrollToBottom();
    }
    if (state is ImMessagesLoaded && state.nextCursor == null) {
      // 仅初始加载（无更多历史消息时）滚动到底部
      _scrollToBottom();
    }
  },
  ```
更优方案：在 `initState` 中监听首次加载完成，或引入 `isInitialLoad` flag。
- **依据**: `git diff` 中 `chat_page.dart` 的 `BlocConsumer` 实现；功能完整性验证。

## 严重问题（Major）

### 2. ImGateway WebSocket DTO 未启用 ValidationPipe，class-validator 装饰器完全失效
- **位置**: `apps/api/src/modules/im/im.gateway.ts:36-37`（类级别）及 `92-95`、`155-158`（方法级别）
- **问题**: `@MessageBody() dto: SendMessageDto` 虽然使用了 DTO 类型，但 Gateway 类和方法上均未添加 `@UsePipes(new ValidationPipe())`。NestJS WebSocket 默认不继承全局 `ValidationPipe`，因此 `SendMessageDto` 上的 `@IsIn(["text", "image"])`、`@MaxLength(500)` 等装饰器不会被评估。非法类型、超长内容、空字段可直接透传到 Service 层。
- **影响**: 数据完整性风险；非法消息可被持久化；违反 `nestjs.md` 中「所有入参使用 class-validator + class-transformer」的强制规范。
- **修复**: 在 Gateway 类添加管道：
  ```typescript
  import { UsePipes, ValidationPipe } from "@nestjs/common";

  @WebSocketGateway({ namespace: "im" })
  @UsePipes(new ValidationPipe({ whitelist: true, transform: true }))
  export class ImGateway implements ... {
  ```
- **依据**: `docs/design-docs/nestjs.md` 第 81-86 行「DTO 与校验」；`apps/api/src/modules/im/dto/send-message.dto.ts` 中的校验装饰器；NestJS 官方文档。

### 3. ImController 手动解析 Query 参数，未使用 DTO 校验
- **位置**: `apps/api/src/modules/im/im.controller.ts:14-67`
- **问题**: `limit` 参数通过 `parseInt(limit, 10)` 手动解析和 clamp，而非使用带 `class-validator` 的 Query DTO。`nestjs.md` 明确要求「所有入参使用 class-validator + class-transformer」。
- **影响**: 代码风格与项目架构规范不一致；参数校验逻辑散落在 Controller 中，难以复用和测试。
- **修复**: 新增 `GetConversationsQueryDto` 和 `GetMessagesQueryDto`：
  ```typescript
  // dto/get-conversations-query.dto.ts
  export class GetConversationsQueryDto {
    @IsOptional()
    @IsInt()
    @Min(1)
    @Max(50)
    @Type(() => Number)
    limit = 20;

    @IsOptional()
    @IsString()
    cursor?: string;
  }
  ```
并在 Controller 方法参数中使用 `@Query() query: GetConversationsQueryDto`。
- **依据**: `docs/design-docs/nestjs.md` 第 81-86 行「DTO 与校验」。

### 4. 会话列表实时刷新未实现（conversation:updated 事件为空处理）
- **位置**: `apps/mobile/lib/presentation/blocs/im/im_bloc.dart:219-226`
- **问题**: `_onConversationUpdated` 方法仅包含 TODO 注释，没有任何实际逻辑。当 Socket 收到 `conversation:updated` 事件（如新消息到达）时，`ConversationListCubit` 管理的会话列表不会自动更新，用户必须手动下拉刷新才能看到最新消息或未读数变化。
- **影响**: IM 核心体验的实时性缺失；用户可能错过新消息提示。
- **修复**: 在 `_onConversationUpdated` 中触发 `ConversationListCubit` 刷新，或至少刷新当前会话的未读数：
  ```dart
  void _onConversationUpdated(
    ImConversationUpdated event,
    Emitter<ImState> emit,
  ) {
    // 触发会话列表刷新（可通过 EventBus 或全局状态管理）
    // 临时方案：emit 一个专用状态供上层监听后调用 cubit.refresh()
    emit(ImConversationListRefreshRequested(event.conversationId));
  }
  ```
并在 `MatchListPage` 中通过 `BlocListener<ImBloc, ImState>` 监听该状态并调用 `fetchConversations()`。
- **依据**: `prd/v1-mvp.md` US-4「Real-time message delivery」；`docs/modules/im.md` 中「会话管理」职责描述。

### 5. 消息在 Socket 断连时静默丢失，且无重试机制
- **位置**: `apps/mobile/lib/presentation/blocs/im/im_bloc.dart:99-104`
- **问题**: `_onSendMessage` 中调用 `_socketClient?.sendMessage(...)`。如果 Socket 未连接（`_socketClient` 为 null 或 `connected == false`），`sendMessage` 不会执行任何操作，但 UI 中的临时消息已进入 `_messages` 列表且状态为 `"sending"`，永远不会变为 `"sent"` 或 `"failed"`。
- **影响**: 弱网或断线场景下，用户发送的消息永远卡在「发送中」状态，无失败提示，也无法重试。
- **修复**: 在发送前检查连接状态，未连接时将消息标记为 `"failed"` 或加入本地待发队列：
  ```dart
  if (_socketClient?.isConnected ?? false) {
    _socketClient!.sendMessage(...);
  } else {
    // 标记失败或入队重试
    _messages[index] = tempMessage.copyWith(status: 'failed');
    emit(ImMessageSent(List.unmodifiable(_messages)));
  }
  ```
- **依据**: 功能完整性 / 边界情况处理；`docs/modules/im.md`「边界情况」第 1 条提到重复发送处理，但未覆盖断线场景。

## 轻微问题（Minor）

### 6. MessageResponseDto 定义在 Service 文件内，未放入 dto/ 目录
- **位置**: `apps/api/src/modules/im/im.service.ts:15-25`
- **问题**: `MessageResponseDto` 直接内联在 Service 中，而项目 `nestjs.md` 规范要求 DTO 统一放在 `dto/` 目录。
- **修复**: 将 `MessageResponseDto` 移至 `apps/api/src/modules/im/dto/message-response.dto.ts` 并导出。
- **依据**: `docs/design-docs/nestjs.md` 第 81-86 行。

### 7. SwChatBubble 可能使用了 Legacy Token 别名
- **位置**: `packages/design-system/lib/src/molecules/chat_bubble.dart:54`、`169`、`182`
- **问题**: 使用了 `DesignTokens.secondaryColor`。根据 `flutter.md` Token 使用指南，legacy 别名（如 `surfaceColor`）禁止在新代码中使用。虽然文档未完整列出所有别名，但同项目的 `chat_input_bar.dart` 和 `conversation_list_item.dart` 对相似语义均使用 `colorScheme.outline`，而 `chat_bubble.dart` 却使用 `DesignTokens.secondaryColor`，存在不一致。
- **修复**: 确认 `tokens.dart` 中 `secondaryColor` 是否为 legacy 别名；若是，替换为 `Theme.of(context).colorScheme.outline` 或提案新增语义 Token。
- **依据**: `docs/design-docs/flutter.md` 第 163-177 行 Token 使用指南。

### 8. ImGateway afterInit 通过类型强转修改内部 CORS 配置
- **位置**: `apps/api/src/modules/im/im.gateway.ts:49-58`
- **问题**: `afterInit` 将 `server` 强转为 `unknown` 再修改 `engine.opts.cors`，属于非公开 API 的 hacky 操作，可能在 Socket.io 版本升级后失效。
- **修复**: 优先通过 NestJS 的 `WebSocketGateway` 构造函数选项或全局 CORS 配置处理跨域，而非运行时修改内部引擎对象。
- **依据**: 代码可维护性最佳实践。

## 风险与建议

1. **Redis 连接数增长**: `ImModule` 和 `MatchingModule` 各自独立提供 `"REDIS_CLIENT"`，会创建两个 Redis 连接。建议在 `AppModule` 或 `CommonModule` 中统一提供全局单例，减少连接开销。
2. **Message pagination cursor 的鲁棒性**: `getMessages` 使用 `sequence_id < cursor` 做分页，依赖数据库自增 `sequence_id` 的单调性。如果未来出现消息删除或 soft-delete，需确认不会影响分页逻辑。
3. **PushService 仅为骨架**: 当前 `PushService.sendPush` 仅打印日志，MVP 阶段可接受，但需在 `docs/exec-plans/tech-debt-tracker.md` 中记录该债务，避免遗漏生产环境集成。
4. **离线补推未实现**: `docs/modules/im.md「边界情况」` 第 4 条提到「离线补推：用户重连后，拉取 `last_ack_sequence_id` 之后的所有消息」，但代码中未见该逻辑。建议在下个迭代补充，或在技术债务中明确记录。

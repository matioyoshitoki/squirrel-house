## 阻塞问题（Blocking）

### 1. `getConversations` 原生 SQL 中 `IN (?)` 数组参数在 mysql2 下无法正确展开
- **位置**: `apps/api/src/modules/im/im.service.ts:239-249`
- **问题**: 代码将 `conversationIds`（字符串数组）作为单个 `?` 占位符传入：
  ```typescript
  const msgResults = await this.dataSource.query(
    `SELECT m.* FROM messages m
     INNER JOIN (
       SELECT conversation_id, MAX(sequence_id) as max_seq
       FROM messages
       WHERE conversation_id IN (?)
       GROUP BY conversation_id
     ) latest ON ...`,
    [conversationIds],
  );
  ```
`mysql2` 驱动不会自动将数组展开为 `IN (?, ?, ?)`，而是将其序列化为逗号分隔的**单字符串**（如 `'id1,id2,id3'`）。这导致 SQL 执行时 `IN` 子句只匹配到这个拼接后的字符串，**查询结果为空或仅匹配第一条记录**，会话列表中的 `last_message` 将全部为 `null`。
- **影响**: 会话列表无法显示任何会话的最后一条消息，IM 核心功能不可用。
- **修复**: 将占位符展开为与数组长度相同的 `?` 列表，或改用 TypeORM QueryBuilder：
  ```typescript
  // 方案 A：动态生成占位符
  const placeholders = conversationIds.map(() => '?').join(',');
  const msgResults = await this.dataSource.query(
    `... WHERE conversation_id IN (${placeholders}) ...`,
    conversationIds,
  );
  ```
  ```typescript
  // 方案 B（推荐）：使用 QueryBuilder
  const msgResults = await this.messageRepo
    .createQueryBuilder('m')
    .innerJoin(
      qb => qb
        .select('conversation_id, MAX(sequence_id) as max_seq')
        .from(MessageEntity, 'sub')
        .where('sub.conversation_id IN (:...ids)', { ids: conversationIds })
        .groupBy('conversation_id'),
      'latest',
      'm.conversation_id = latest.conversation_id AND m.sequence_id = latest.max_seq'
    )
    .getMany();
  ```
- **依据**: `docs/design-docs/db.md` 第 91-121 行明确规定 mysql2 原生 SQL 必须使用 `?` 位置参数，且 `IN (?)` 对数组的隐式展开不被 mysql2 支持；`precommit-checks.sh` 会扫描 `:paramName` 但不会捕获 `IN (?)` 数组陷阱。

## 严重问题（Major）

### 2. `ChatPage` 消息重试逻辑导致重复消息显示
- **位置**: `apps/mobile/lib/presentation/pages/chat_page.dart:189-200`
- **问题**: `onRetry` 回调在点击重试时生成**新的** `clientMessageId`，而原 `failed` 状态的消息仍然保留在 `_messages` 列表中。重试后用户界面会同时看到两条消息：旧的 `failed` 消息和新的 `sending` 消息。ack 回来时只会更新新消息，旧消息将永远停留在 `failed` 状态。
- **影响**: 用户体验受损，聊天界面可能出现重复消息或僵尸消息。
- **修复**: 重试时应复用原消息的 `clientMessageId`，并在发送前将原消息状态从 `failed` 重置为 `sending`：
  ```dart
  onRetry: msg.status == 'failed' && isSelf
      ? () {
          // 1. 先更新本地消息状态为 sending
          final index = _messages.indexWhere((m) => m.clientMessageId == msg.clientMessageId);
          if (index != -1) {
            _messages[index] = MessageItem(..., status: 'sending');
            setState(() {}); // 或发送 BLoC 事件
          }
          // 2. 使用相同的 clientMessageId 重发
          context.read<ImBloc>().add(
            ImSendMessage(
              conversationId: msg.conversationId!,
              content: msg.content,
              clientMessageId: msg.clientMessageId!,
            ),
          );
        }
      : null,
  ```
- **依据**: PRD `v1-mvp.md` US-4 要求"消息发送到达时间 < 1s"，重复消息显示违背核心体验。

### 3. `SwChatInputBar` 未限制最大输入长度
- **位置**: `packages/design-system/lib/src/molecules/chat_input_bar.dart:112-135`
- **问题**: `TextField` 未设置 `maxLength: 500`，仅通过 `_isOverLimit` 在 UI 层提示超限。用户仍可通过粘贴等方式输入超过 500 字符的内容，且 `onSend` 可能将超限内容传递给后端。
- **影响**: 可能绕过前端的 500 字符限制，导致后端再次抛出 `BusinessException`（虽然后端有校验），体验不佳。
- **修复**: 在 `TextField` 中添加 `maxLength: _maxLength`：
  ```dart
  TextField(
    controller: _controller,
    focusNode: _focusNode,
    maxLines: 4,
    minLines: 1,
    maxLength: _maxLength, // 新增
    textInputAction: TextInputAction.send,
    // ...
  )
  ```
- **依据**: PRD `v1-mvp.md` 第 156 行明确规定"单条文本限制 500 字符"。

### 4. `ImBloc` 消息列表无上限缓存，长聊天记录存在内存泄漏风险
- **位置**: `apps/mobile/lib/presentation/blocs/im/im_bloc.dart:21`
- **问题**: `_messages` 列表在 `ImLoadMessages` 事件中通过 `_messages.insertAll(0, result.items)` 无限增长，没有消息数量上限或淘汰策略。对于长时间聊天的用户，内存占用会持续增加。
- **影响**: 长时间使用聊天功能后应用可能因内存压力被系统回收。
- **修复**: 限制内存中缓存的消息数量（如最多 200 条），超出时从尾部淘汰：
  ```dart
  const int _maxCachedMessages = 200;
  
  // 在 _onLoadMessages 和 _onMessageReceived 中
  if (_messages.length > _maxCachedMessages) {
    _messages.removeRange(0, _messages.length - _maxCachedMessages);
  }
  ```
- **依据**: `docs/design-docs/flutter.md` 第 457 行"禁止在 BLoC 中直接操作 `BuildContext`"与此无关，但内存管理属于代码健壮性范畴。

### 5. `ImSocketClient` 回调中缺少数据类型安全检查
- **位置**: `apps/mobile/lib/core/network/im_socket_client.dart:43-67`
- **问题**: 所有 Socket 事件监听都直接执行 `data as Map<String, dynamic>`，未检查 `data is Map`。如果服务端因 bug 发送了 `null`、数组或字符串，将导致 `TypeError` 运行时崩溃。
- **影响**: Socket 通信异常时前端直接崩溃。
- **修复**: 添加类型安全检查：
  ```dart
  _socket?.on('message:receive', (data) {
    if (data is Map<String, dynamic>) {
      onMessageReceive(data);
    }
  });
  ```
- **依据**: `docs/design-docs/flutter.md` 第 459 行"禁止使用 `dynamic` 传递 API 数据"。

### 6. OpenAPI 契约 `MessageDto` 与后端实现字段 nullable 不一致
- **位置**: `docs/api-contracts/im.yaml:87-116`
- **问题**: 
  - `MessageDto.client_message_id` 未标记 `nullable: true`，但后端 `MessageResponseDto` 中定义为 `string | null`（系统消息为 `null`）。
  - `ConversationDto` 要求 `id` 字段，但后端 `ImController` 返回的是 `conversation_id`。
- **影响**: 生成的客户端类型与实际 API 响应不一致，可能导致 TypeScript/Dart 类型错误。
- **修复**: 
  ```yaml
  MessageDto:
    properties:
      client_message_id:
        type: string
        nullable: true  # 新增
  ```
并将 `ConversationDto.id` 改为 `conversation_id`，或调整后端返回字段名。
- **依据**: `docs/design-docs/api.md` 第 78-83 行"契约优先：接口变更必须先在 YAML 中定义，再生成代码"。

## 轻微问题（Minor）

### 7. `withOpacity` 已废弃，建议迁移到 `withValues(alpha: ...)`
- **位置**: 
  - `apps/mobile/lib/presentation/pages/match_list_page.dart:95`
  - `apps/mobile/lib/presentation/pages/match_list_page.dart:104`
- **问题**: Flutter 3.27+ 中 `Color.withOpacity` 已被标记为废弃。
- **修复**: 将 `DesignTokens.textPrimary.withOpacity(0.3)` 替换为 `DesignTokens.textPrimary.withValues(alpha: 0.3)`。

### 8. `ChatPage` 每次离开页面都断开 Socket 连接
- **位置**: `apps/mobile/lib/presentation/pages/chat_page.dart:60-61`
- **问题**: `dispose` 中发送 `ImDisconnectRequested`，导致从聊天页返回会话列表时 Socket 断开。如果用户频繁切换会话，会造成不必要的重连开销。
- **建议**: 将 Socket 连接生命周期提升到应用级别（如 `App` 层级），仅在用户登出时断开。

### 9. `ImBloc.setCurrentConversation` 直接方法调用而非事件驱动
- **位置**: `apps/mobile/lib/presentation/blocs/im/im_bloc.dart:274-277`
- **问题**: BLoC 的公共状态变更接口应该是事件（Event），而不是直接方法调用。
- **建议**: 添加 `ImSetCurrentConversation` 事件，在 `ChatPage.initState` 中通过 `add()` 触发。

### 10. `ImGateway` Socket 认证未复用 NestJS `JwtService`
- **位置**: `apps/api/src/modules/im/im.gateway.ts:57-72`
- **问题**: 直接调用 `jwt.verify(token, jwtConfig.secret)`，与项目中其他模块（如 `JwtAuthGuard`）的认证方式不一致。
- **建议**: 注入 `JwtService` 并复用其 `verify` 方法，保持认证逻辑统一。

### 11. `match_list_page.dart` 文件名与功能不匹配
- **位置**: `apps/mobile/lib/presentation/pages/match_list_page.dart`
- **问题**: 该页面现在显示的是**会话列表**而非匹配列表，但文件名仍为 `match_list_page.dart`。
- **建议**: 重命名为 `conversation_list_page.dart` 并在路由中同步更新，或在文档中明确标注此技术债务。

## 风险与建议

1. **数据一致性风险（高）**：`Blocking` 级别的 SQL `IN (?)` 问题必须修复后才能合并，否则会话列表的 `last_message` 字段将永远为空，直接影响 IM 核心体验。

2. **消息重试 UX 风险（中）**：建议优先修复 `onRetry` 的 `clientMessageId` 复用问题，避免用户看到重复/僵尸消息。

3. **Socket 连接稳定性（低）**：当前每次离开聊天页都断开 Socket，MVP 阶段可接受，但建议在下一个迭代中将 Socket 生命周期提升到 App 级别。

4. **OpenAPI 契约漂移（低）**：`MessageDto.client_message_id` 和 `ConversationDto.id` 字段名与实现不一致，建议更新 YAML 并重新生成 shared-types/mobile models，确保前后端契约严格一致。

5. **E2E 测试可运行性（中）**：Maestro flow 中 `send_message.yaml` 依赖 `includes/login-as-test-user.yaml` 和预置的匹配数据（通过 `seed-dev.ts` 注入），需确保 CI 环境中种子数据正确执行。`seed-dev.ts` 中已为 E2E 用户创建了双向匹配和会话，这一点做得很好。

## 阻塞问题（Blocking）

### 1. MainPage 缺少 ConversationListCubit 的 BlocProvider

- **位置**: `apps/mobile/lib/app.dart:138-143`
- **问题**: `MainPage` 的 `MultiBlocProvider` 只注册了 `MatchingBloc` 和 `MatchListCubit`，未注册 `ConversationListCubit`。`MatchListPage` 在 `initState()` 中直接调用 `context.read<ConversationListCubit>().fetchConversations()`，会因找不到 Provider 而抛出 `ProviderNotFoundException`，导致「消息」Tab 页面崩溃。
- **影响**: 用户无法进入会话列表页面，IM 核心流程完全不可用。
- **修复**:
  ```dart
  // apps/mobile/lib/app.dart:138-143
  return MultiBlocProvider(
    providers: [
      BlocProvider(create: (_) => getIt<MatchingBloc>()),
      BlocProvider(create: (_) => getIt<MatchListCubit>()),
      BlocProvider(create: (_) => getIt<ConversationListCubit>()), // 添加此行
    ],
    child: Scaffold(...),
  );
  ```
- **依据**: `docs/design-docs/flutter.md` 第 38-41 行规定状态管理使用 `flutter_bloc`，且 `MatchListPage` 通过 `context.read<ConversationListCubit>()` 获取 Cubit，必须在 Widget Tree 中通过 `BlocProvider` 提供。

## 严重问题（Major）

### 2. ChatPage 未实现消息发送失败后的重试机制

- **位置**: `apps/mobile/lib/presentation/pages/chat_page.dart:190-200`
- **问题**: `SwChatBubble` 组件已支持 `onRetry` 回调（当 `status == 'failed'` 时显示错误图标并响应点击），但 `ChatPage` 在构建 `SwChatBubble` 时未传入 `onRetry` 参数。当 Socket 未连接导致消息发送失败（`ImBloc` 将状态设为 `failed`）后，用户点击错误图标无任何反应。
- **影响**: 消息发送失败后用户必须重新输入内容才能再次发送，体验极差。
- **修复**:
  ```dart
  // apps/mobile/lib/presentation/pages/chat_page.dart
  SwChatBubble(
    type: isSelf ? SwChatBubbleType.self : SwChatBubbleType.other,
    content: msg.content,
    // ... 其他参数 ...
    onRetry: msg.status == 'failed' && isSelf
        ? () => context.read<ImBloc>().add(
              ImSendMessage(
                conversationId: msg.conversationId!,
                content: msg.content,
                clientMessageId: msg.clientMessageId ?? const Uuid().v4(),
              ),
            )
        : null,
  );
  ```
- **依据**: PRD `v1-mvp.md` US-4 要求消息发送到达时间 < 1s，且 `docs/design-docs/flutter.md` 第 449-451 行要求核心页面必须处理加载/错误/空态。

### 3. ImBloc 已读回执使用列表索引而非 sequence_id 比较

- **位置**: `apps/mobile/lib/presentation/blocs/im/im_bloc.dart:174-184`
- **问题**: `_onReadReceipt` 中通过列表索引 `i <= lastReadIndex` 判断消息是否应标记为已读：
  ```dart
  final lastReadIndex = _messages.indexWhere((m) => m.id == event.lastReadMessageId);
  for (var i = 0; i < _messages.length; i++) {
    if (msg.senderId == _currentUserId) {
      final shouldMarkRead = lastReadIndex == -1 || i <= lastReadIndex;
      // ...
    }
  }
  ```
该逻辑假设 `_messages` 严格按 `sequence_id` 升序排列且连续。若历史消息加载存在间隙（例如用户加载了较旧的历史消息，中间跳过了部分消息），则列表索引与消息时间顺序不一致，可能导致错误地将较新消息标记为已读，或未将较早消息标记为已读。
- **影响**: 已读状态显示不正确，用户体验受损。
- **修复**: 使用 `sequenceId` 比较替代列表索引：
  ```dart
  final lastReadSeq = lastReadIndex == -1
      ? null
      : _messages[lastReadIndex].sequenceId;
  for (var i = 0; i < _messages.length; i++) {
    final msg = _messages[i];
    if (msg.senderId == _currentUserId) {
      final shouldMarkRead = lastReadSeq == null ||
          (msg.sequenceId != null && msg.sequenceId! <= lastReadSeq);
      // ...
    }
  }
  ```
- **依据**: `docs/modules/im.md` 第 70-73 行明确说明消息顺序使用数据库自增 `sequence_id`，客户端应按序排列。

### 4. ImSocketClient 缺少连接状态监控与显式重连策略

- **位置**: `apps/mobile/lib/core/network/im_socket_client.dart`
- **问题**: `ImSocketClient` 仅通过 `autoConnect: true` 建立初始连接，未显式配置 `reconnection` 参数，也未监听 `connect_error` / `disconnect` 事件向 `ImBloc` 汇报。当应用从后台回到前台、或网络切换后，Socket 可能处于断开状态而不被上层感知，`isConnected` 返回过时状态。
- **影响**: 用户在网络恢复后可能无法立即发送消息，或错误地认为消息已发送成功。
- **修复**: 在 `ImSocketClient.connect()` 中添加显式重连配置和事件监听：
  ```dart
  _socket = io.io(namespaceUrl, <String, dynamic>{
    'transports': ['websocket'],
    'autoConnect': true,
    'reconnection': true,
    'reconnectionAttempts': 5,
    'reconnectionDelay': 1000,
    'auth': <String, dynamic>{'token': token},
  });
  _socket?.on('disconnect', (_) => onError({'message': 'Disconnected'}));
  _socket?.on('connect_error', (err) => onError({'message': 'Connect error: $err'}));
  ```
- **依据**: `docs/design-docs/flutter.md` 第 52-53 行要求网络层使用 Dio + 拦截器处理统一错误；Socket 层同样需要统一的连接状态管理。

## 轻微问题（Minor）

### 5. ImGateway 连接认证失败无日志记录

- **位置**: `apps/api/src/modules/im/im.gateway.ts:49-68`
- **问题**: `handleConnection` 中 JWT 验证失败或缺少 token 时直接 `client.disconnect(true)`，catch 块为空，未记录任何日志。
- **修复**: 在 catch 块中添加 `this.logger.warn(...)` 记录断开原因。
- **依据**: `docs/design-docs/nestjs.md` 第 106-111 行要求错误日志包含 `traceId` 和用户 ID。

### 6. PushService 在多个模块中重复注册

- **位置**: `apps/api/src/modules/im/im.module.ts:29`、`apps/api/src/modules/matching/matching.module.ts:35`
- **问题**: `PushService` 同时在 `ImModule` 和 `MatchingModule` 的 `providers` 数组中注册。虽然 `PushService` 无状态，但 NestJS 会为每个模块创建独立实例，违背单例预期。
- **修复**: 将 `PushService` 移至 `common/` 或独立的 `PushModule` 中，通过 `exports` 共享。
- **依据**: `docs/design-docs/nestjs.md` 第 68-70 行要求模块间通过 Interface / EventEmitter 解耦，禁止直接 import 其他模块的内部 Service。

### 7. seed-dev.ts 中 preferred_gender 枚举值与 PRD 不一致

- **位置**: `apps/api/src/scripts/seed-dev.ts:78`
- **问题**: Seed 脚本中使用了 `"all"`，而 `prd/v1-profile.md` 第 139 行明确规定 `preferred_gender` 枚举为 `male`、`female`、`both`。
- **影响**: 可能导致推荐算法中性别过滤逻辑异常（Matching SQL 中判断 `? = 'both'`，不会匹配 `"all"`）。
- **修复**: 将 seed 脚本中的 `"all"` 改为 `"both"`。
- **依据**: `prd/v1-profile.md` 第 139 行。

### 8. E2E send_message.yaml 的前置依赖说明不够健壮

- **位置**: `.maestro/flows/im/send_message.yaml`
- **问题**: Flow 使用 `optional: true` 跳过 `conversation_item` 点击，若测试环境未预置会话数据，后续 `tapOn: { id: "chat_input_field" }` 将在错误的页面执行导致失败。虽然注释说明了前置条件，但 Maestro 执行失败时的错误信息不直观。
- **修复**: 在 `conversation_item` 步骤后添加 `assertVisible: { id: "chat_input_field" }`（非 optional），确保进入聊天页后再继续。

## 风险与建议

1. **消息顺序一致性风险**：`ImBloc` 中 `_messages` 列表的维护依赖于服务端返回的升序数据和本地追加逻辑。建议在后端 `getMessages` API 中显式保证 `sequence_id` 升序返回，并在 Mobile 端添加防御性校验（若发现乱序则触发全量刷新）。

2. **Redis 未读数与 DB 状态不一致风险**：`markAsRead` 同时更新 `conversation_participants.last_read_message_id` 和 Redis `unread_count`。若 Redis 故障或重启，未读数会丢失。建议增加定时任务从 DB 重新计算未读数作为兜底。

3. **并发重排风险**：`getConversations` 使用 `updated_at` 作为游标，若并发场景下 `updated_at` 出现相同值，可能导致分页重复或跳过。建议在游标中复合 `conversation_id`（如 `"updated_at#conversation_id"`）增强唯一性。

4. **设计资产迁移合规性**：PR 中 `packages/design-system/` 已正确迁移 `designs/issue-5/` 中的组件（`sw_chat_bubble.dart`、`sw_chat_input_bar.dart`、`sw_conversation_list_item.dart`），并补充了 Widgetbook 用例和 Golden Test，符合 `AGENTS.md` 中 Design-Only PR 的迁移规范。

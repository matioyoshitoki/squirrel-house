## 审查结论

审查结论: **NEEDS_MAJOR_FIX**

PR #58 实现了 V1 IM（即时通讯）模块的核心功能：消息收发、已读回执、会话列表、系统消息推送。整体架构方向正确，全链路（DB → API → Mobile → Design System → E2E）均有覆盖，但存在 **4 个 Blocking 级别的运行时缺陷** 和 **7 个 Major 级别的规范/架构/测试缺陷**，必须在合并前修复。

## 阻塞问题（Blocking）

### 1. ImBloc 在 Event Handler 外部调用 `emit`，导致运行时 StateError
- **位置**: `apps/mobile/lib/presentation/blocs/im/im_bloc.dart:61-62`
- **问题**: `onError` 回调在 Socket 发生错误时直接调用 `emit(ImError(...))`。但 `_onConnect` 是同步 event handler，执行完即返回；当 Socket 后续发生错误时，`emit` 已不在有效的 handler 上下文中，flutter_bloc 会抛出 `StateError: emit was called after an event handler completed normally.`
- **影响**: Socket 连接失败或断线时 App 直接崩溃
- **修复**: 将 socket 错误封装为独立 event，通过 `add()` 投递到 Bloc 事件循环：
  ```dart
  onError: (data) {
    add(ImSocketErrorReceived(data['message'] as String? ?? 'Socket error'));
  },
  ```
并在 `ImBloc` 中注册 `on<ImSocketErrorReceived>((event, emit) => emit(ImError(event.message)));`
- **依据**: `docs/design-docs/flutter.md` 中 BLoC 状态管理规范 + flutter_bloc 官方文档约束

### 2. ImBloc 消息未按 Conversation 隔离，跨会话消息会串扰到当前聊天
- **位置**: `apps/mobile/lib/presentation/blocs/im/im_bloc.dart:103-117`, `210-213`
- **问题**: 
  1. `_parseSocketMessage` 未解析并保存 `conversation_id`（虽然 Gateway payload 包含该字段）；
  2. `_getConversationIdForMessage` 直接返回 `_currentConversationId` 而非从消息中读取；
  3. 因此 `_onMessageReceived` 中的条件判断永远为 `true`，任何会话的 socket 消息都会被追加到当前显示的消息列表。
- **影响**: 用户在 Conversation A 聊天时，如果收到 Conversation B 的新消息，B 的消息会错误显示在 A 的聊天界面中
- **修复**: 
  1. 在 `MessageItem` 中增加 `conversationId` 字段；
  2. 在 `_parseSocketMessage` 中解析 `data['conversation_id']`；
  3. `_onMessageReceived` 中严格比对 `event.message.conversationId == _currentConversationId`，不匹配则丢弃或仅更新会话列表
- **依据**: PRD `v1-mvp.md` US-4 中「Alice opens the conversation」的会话隔离要求

### 3. ChatPage 在 `ImMessageRead` 状态下返回空消息列表，导致 UI 清空
- **位置**: `apps/mobile/lib/presentation/pages/chat_page.dart:236-244`
- **问题**: `_getMessagesFromState` 对 `ImMessageRead` state 返回 `[]`（第 240-243 行注释甚至承认这是故意留空）。当对方已读消息时，Socket 触发 `_onReadReceipt`，Bloc emit `ImMessageRead`，ChatPage 重新 build 并调用 `_getMessagesFromState`，消息列表突然变为空，界面显示 "你们已匹配成功，开始聊天吧" 系统提示。
- **影响**: 用户看到对方已读后，自己的消息列表瞬间清空，功能完全不可用
- **修复**: 
  ```dart
  if (state is ImMessageRead) {
    return state.messages; // 或从 Bloc 的 _messages 返回
  }
  ```
同时需要在 `ImBloc._onReadReceipt` 中维护 `_messages` 列表的已读状态（或至少不破坏现有列表）。
- **依据**: `git diff` 中的代码证据 + PRD 中「Read receipt」功能完整性要求

### 4. MatchingService 事务中调用 ImService.createSystemMessage 未共用 Transaction Manager
- **位置**: `apps/api/src/modules/matching/matching.service.ts:462-465`, `apps/api/src/modules/im/im.service.ts:369-385`
- **问题**: `MatchingService.checkAndCreateMutualMatch` 在 `dataSource.transaction(async (manager) => { ... })` 内部调用 `this.imService.createSystemMessage()`，但 `createSystemMessage` 使用的是 `this.messageRepo.save(message)`（标准 Repository，不绑定事务 manager）。因此 system message 不在同一事务中：若事务回滚，`matches`/`conversations`/`participants` 被回滚，但 `messages` 表中仍残留一条指向不存在 conversation 的孤儿记录。
- **影响**: 数据不一致，极端情况下产生无关联的 system message
- **修复**: 方案 A（推荐）：让 `createSystemMessage` 接受可选的 `EntityManager`：
  ```typescript
  async createSystemMessage(
    conversationId: string,
    content: string,
    manager?: EntityManager,
  ): Promise<MessageEntity> {
    const repo = manager ? manager.getRepository(MessageEntity) : this.messageRepo;
    const message = repo.create({ ... });
    await repo.save(message);
    return message;
  }
  ```
并在 `MatchingService` 中传入 `manager`。
- **依据**: `docs/design-docs/nestjs.md` 第 129-142 行事务发射规则中的事务边界要求 + PRD `v1-matching.md` US-MATCH-3 中「事务：匹配创建必须使用数据库事务，确保 `matches` + `conversations` + `conversation_participants` 原子性」

## 严重问题（Major）

### 5. Design System 组件命名违反 `Sw` 前缀强制规范
- **位置**: 
  - `packages/design-system/lib/src/molecules/chat_bubble.dart` (类名 `ChatBubble`)
  - `packages/design-system/lib/src/molecules/chat_input_bar.dart` (类名 `ChatInputBar`)
  - `packages/design-system/lib/src/molecules/conversation_list_item.dart` (类名 `ConversationListItem`)
- **问题**: 所有新增 design-system 组件均未以 `Sw` 为前缀
- **影响**: 命名空间污染，与 flutter.md 的强制规范冲突，未来难以通过 linter 自动识别 design-system 组件
- **修复**: 重命名为 `SwChatBubble`、`SwChatInputBar`、`SwConversationListItem`，同步修改 `social_world_design_system.dart` 的 export 语句、所有引用处（`chat_page.dart`、`match_list_page.dart`、`widgetbook/main.dart`）以及 Maestro flow 中涉及的 id（如有）
- **依据**: `docs/design-docs/flutter.md` 第 48 行："设计系统组件必须以 `Sw` 为前缀"

### 6. IM Gateway 直接内联构造 emit Payload，未使用 shared-types 契约类型
- **位置**: `apps/api/src/modules/im/im.gateway.ts` 多处（如第 101-105 行 `message:ack`、第 113-122 行 `message:receive`、第 156 行 `message:read_receipt`）
- **问题**: Gateway 直接手写 `{}` 对象作为 payload，未从 `packages/shared-types` 导入类型安全的 DTO/Interface
- **影响**: 前后端契约不同步风险高，字段命名漂移（如 `sequence_id` vs `sequenceId`）难以在编译期发现
- **修复**: 在 `packages/shared-types` 中定义 Socket Event DTO（如 `MessageReceivedEvent`、`MessageAckEvent`、`ReadReceiptEvent`），Gateway 中导入并实例化这些类型
- **依据**: `docs/design-docs/nestjs.md` 第 144-151 行："Payload 类型：必须使用 `shared-types` 中定义的 DTO/Interface，禁止内联 `any`"

### 7. MatchingService 直接 import 并注入 ImService，违反模块间解耦规范
- **位置**: `apps/api/src/modules/matching/matching.service.ts:25`, `matching.module.ts:30`
- **问题**: Matching 模块直接依赖 Im 模块的内部 Service，且引入了 `forwardRef` 循环依赖
- **影响**: 模块边界模糊，Matching 模块的单元测试必须 Mock ImService，增加了耦合度
- **修复**: 通过 `EventEmitter2` 发布 `match:created` 事件，在 `ImModule` 中监听该事件并创建 system message。移除 `MatchingService` 对 `ImService` 的直接依赖和 `forwardRef`。
- **依据**: `docs/design-docs/nestjs.md` 第 70 行："模块之间通过 Interface / EventEmitter 解耦，禁止直接 import 其他模块的内部 Service"

### 8. ConversationListCubit 分页逻辑缺陷，导致无法加载更多或无限重复加载
- **位置**: `apps/mobile/lib/presentation/blocs/im/conversation_list_cubit.dart:32-36`
- **问题**: 
  1. `ImRepository.getConversations` 返回类型为 `List<ConversationItem>`，**丢弃了 API 返回的 `next_cursor` 和 `has_more`**；
  2. Cubit 中 `nextCursor` 被错误地设为传入的 `cursor`（旧值）而非服务端返回的新游标：
     ```dart
     nextCursor: result.isNotEmpty ? cursor : null,
     ```
  3. 当首次加载完成后，`nextCursor` 为 `null`，但 `hasMore` 可能为 `true`；UI 会触发加载更多，传入 `cursor: null`，导致**重复加载第一页**，形成无限循环（或至少无法加载后续页）。
- **影响**: 会话列表分页功能完全不可用；用户超过 20 个会话时只能看到前 20 个
- **修复**: 
  1. 修改 `ImRepository` 接口：
     ```dart
     Future<({List<ConversationItem> items, bool hasMore, String? nextCursor})> getConversations(...)
     ```
  2. `ImRepositoryImpl` 从响应中解析 `data['next_cursor']` 和 `data['has_more']`；
  3. Cubit 中使用服务端返回的 `nextCursor`：
     ```dart
     nextCursor: result.nextCursor,
     hasMore: result.hasMore,
     ```
- **依据**: `docs/api-contracts/im.yaml` 中 `ListConversations` 的响应结构包含 `next_cursor` / `has_more`；PRD 功能完整性要求

### 9. Maestro E2E 中 `conversation_item_` 精确匹配无法命中动态 id
- **位置**: `.maestro/flows/im/send_message.yaml:23-24`, `.maestro/flows/im/view_conversation_list.yaml:36-38`
- **问题**: Maestro 的 `id` 匹配是精确匹配 Android `resource-id`。但 `ConversationListItem` 的 `Semantics` identifier 是动态生成的 `conversation_item_$nickname`（如 `conversation_item_Alice`），Maestro flow 中写死 `conversation_item_` 永远匹配不到任何元素。
- **影响**: E2E 测试中的会话列表点击步骤永远失败（但因标记了 `optional: true` 被静默跳过）
- **修复**: 
  - 方案 A：将 `ConversationListItem` 的 `identifier` 改为固定值 `conversation_item`，Maestro 通过列表索引或文本匹配定位；
  - 方案 B：保留动态 id，但 Maestro 使用 `tapOn: "Alice"` 通过昵称文本定位（需确保昵称在 semantics 中可访问）。
- **依据**: `docs/design-docs/flutter.md` Accessibility 章节："`identifier: 'xxx'` 生成稳定的 `resource-id`" + Maestro 精确匹配行为

### 10. 新增 Design System 组件缺少 Golden Test 与 Widget Test
- **位置**: `packages/design-system/lib/src/molecules/chat_bubble.dart`, `chat_input_bar.dart`, `conversation_list_item.dart`
- **问题**: PR 新增 3 个 design-system 组件，但 `packages/design-system/test/` 下未增加任何对应 golden test 或 widget test
- **影响**: 无视觉回归保护，后续样式变更可能破坏 IM 核心 UI 而无人知晓
- **修复**: 
  1. 在 `packages/design-system/test/im_components_golden_test.dart` 中补充 `SwChatBubble`（self / other / system / sending / failed 状态）、`SwChatInputBar`（empty / typing / over-limit 状态）、`SwConversationListItem`（default / unread / long-message 状态）的 golden test；
  2. 运行 `flutter test --update-goldens` 生成基准图并提交。
- **依据**: `docs/design-docs/testing.md` 第 333-375 行 Golden File Testing 规范："新增/修改的 `packages/design-system` 组件必须编写 Golden Test"

### 11. Maestro flow 过度使用 `optional: true`，丧失 E2E 验收价值
- **位置**: `.maestro/flows/im/send_message.yaml`（第 24、29、33、35、40、41、44、45、48、49 行均为 `optional: true`）
- **问题**: 核心断言步骤（如点击输入框、输入文本、点击发送、断言消息可见）全部被标记为 optional，意味着即使 IM 核心功能完全损坏，E2E 也会全部跳过并通过。
- **影响**: E2E 测试失去回归验证意义
- **修复**: 仅对「会话可能为空」这类条件分支步骤保留 `optional`（如首次进入无会话时的空状态）。以下步骤必须移除 `optional`：
  - 切换消息 Tab (`tab_messages`)
  - 点击输入框 (`chat_input_field`)
  - 输入消息内容
  - 点击发送 (`chat_send_button`)
  - 断言消息可见 (`assertVisible: "你好"`)
- **依据**: `docs/design-docs/testing.md` 第 326-331 行："涉及 Mobile 用户流程的 PR，必须包含可运行的 Maestro E2E 测试"

## 风险与建议

1. **循环依赖风险**: `MatchingModule ↔ ImModule` 通过 `forwardRef` 建立循环依赖，虽然 NestJS 支持，但增加了启动时复杂度和潜在的初始化顺序问题。建议通过 EventEmitter 解耦（见 Major-7）。

2. **Socket 连接管理风险**: `ChatPage` 每次进入都通过 `getIt<ImBloc>()` 创建新的 Bloc 和 Socket 连接，退出时断开。如果用户在会话列表和聊天页之间频繁切换，会导致 Socket 频繁重连。建议在 App 生命周期内维持单一 Socket 连接（如将 `ImBloc` 提升为全局单例或在 `MainPage` 层级管理）。

3. **Redis 未读数一致性风险**: `markAsRead` 中 `await this.redis.set(..., "0")` 与 `participantRepo.update` 是两个独立操作，无事务。如果 Redis 操作成功但 DB update 失败，会导致未读数不一致。建议增加补偿机制或统一在 DB 事务成功后才清 Redis。

4. **消息顺序风险**: `getMessages` 中 `messages.reverse()` 修改了数组（虽然 TypeScript 中 `slice` 已复制），但如果 `hasMore=false` 时 `results` 直接赋值给 `messages` 再 `reverse`，会修改原数组。虽然 `results` 是局部变量，不影响外部，但建议改用 `[...messages].reverse()` 保证不可变性。

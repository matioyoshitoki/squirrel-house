## 严重问题（Major）

### 1. UIButton.setTexture() 替换背景后丢失交互能力
- **位置**: `apps/phaser3/src/ui/UIButton.ts:175-182`
- **问题**: `setTexture()` 调用 `this.background.destroy()` 后执行 `this.createBackground()`，但**没有重新调用 `setupInteraction()`**。新创建的 `background` GameObject 未设置 `setInteractive`，导致按钮在 `setTexture()` 后完全失去点击、Hover、Pressed 响应。
- **影响**: 任何依赖动态切换按钮纹理的业务场景（如换肤、状态切换）将导致按钮不可用。
- **修复**: 在 `setTexture()` 末尾追加 `this.setupInteraction()` 调用，并确保旧事件监听已被清理：
  ```ts
  setTexture(texture: string): void {
    this.config.texture = texture;
    this.background.destroy();
    this.createBackground();
    this.remove(this.label);
    this.add(this.label);
    this.setupInteraction(); // 新增
  }
  ```
- **依据**: PRD-004 5.3 要求 `setTexture()` 为动态切换纹理的公开 API，其交互功能必须保持可用。

### 2. UIButton Disabled 态 alpha = 0.6 违反 PRD-004 强制规范
- **位置**: `apps/phaser3/src/ui/UIButton.ts:152-153`, `:167-168`
- **问题**: PRD-004 3.6 组件状态机章节明确要求 **"DISABLED 态 … alpha ≤ 0.5"**，但 `UIButton` 在 `setupInteraction()` 和 `setDisabled(true)` 中均硬编码 `this.setAlpha(0.6)`。
- **影响**: 视觉验收环节会被判定为不通过；与 PRD-004 的合规性要求冲突。
- **修复**: 将两处 `0.6` 改为 `0.5`：
  ```ts
  this.setAlpha(0.5);
  ```
- **依据**: PRD-004 3.6 原文 — "DISABLED 态必须调用 disableInteractive()，alpha ≤ 0.5"。（注：`docs/design-docs/ui-style-guide/03-component.md` 3.1 节写为 0.6，属于文档内部矛盾，应以更高层级 PRD-004 为准。）

### 3. ProgressToastManager 自定义 startY 对 ToastManager 失效
- **位置**: `apps/phaser3/src/city-builder/ui/ProgressToast.ts:100-104`
- **问题**: `ProgressToastManager` 构造函数接收可选的 `startY` 参数并保存在 `this.startY`，但底层 `ToastManager` 通过 `ToastManager.getInstance(scene)` 获取，该单例**不接受**自定义 `startY`，始终使用默认值 `scene.cameras.main.height - 100`。当 `ProgressToastManager` 使用自定义 `startY` 时，`ToastManager.repositionToasts()` 在 Toast dismiss 后会按默认 `startY` 重新计算位置，导致垂直堆叠错位。
- **影响**: 若业务层传入非默认 `startY`，Toast 队列的自动补位动画会跳到错误坐标。
- **修复**: 方案 A（推荐）：让 `ProgressToastManager` 直接管理 Y 坐标，不依赖 `ToastManager` 的 `repositionToasts`；或在 `register` 时传入自定义的 `startY` 并覆盖单例内部值。方案 B：恢复 `ToastManager` 支持带 `startY` 的工厂方法，如 `ToastManager.getInstance(scene, startY)`。
- **依据**: `ProgressToastManager` 原实现通过 `new ToastManager(scene, startY)` 传递了该值，PR 修改后此链路断裂。

### 4. designs/issue-19 设计资产文档存在多处与实现不一致或含运行时错误
- **位置**: 
  - `designs/issue-19/src/ui/UIDialog.ts:62-65`
  - `designs/issue-19/src/ui/UIToast.ts:67, 162`
  - `designs/issue-19/src/ui/UIAlert.ts:60, 124`
  - `designs/issue-19/design-spec.md` 3.3 / 3.4
- **问题**: 
  1. **UIDialog 设计文件含 `this` 前置访问错误**：`panelConfig.onOverlayClick` 使用箭头函数 `() => { this.cancel(); }`，该配置对象在 `super(scene, panelConfig)` 之前构造，箭头函数提前捕获了尚未初始化的 `this`，在真实运行时会导致 ReferenceError（`Must call super constructor`）。实际源码已修复，但设计资产作为后续开发者的参考文档，残留致命错误。
  2. **UIToast / UIAlert 设计文件缺少 `baseY`**：构造器中执行 `this.y += 20`，`show()` 中 tween 到 `this.y - 20`。若 `show()` 被调用两次，第二次会以已 tween 后的 `this.y` 为基准再减 20，造成持续上浮漂移。实际源码已引入 `baseY` 修复，设计文件未同步。
  3. **design-spec.md warning 色值与实现不一致**：设计规范 3.3/3.4 表格中 warning 色条写为 `UITheme.goldBrown`，但实际代码与更新后的 `03-component.md` 均使用 `UITheme.warning`（`0xffaa00`）。
- **影响**: 设计资产（`designs/issue-19/`）是 Issue #19 的交付物之一，会被后续 Agent 或开发者引用。错误的代码示例和规范不一致会导致后续实现踩坑。
- **修复**: 
  - `designs/issue-19/src/ui/UIDialog.ts`: 参照实际源码，将 `onOverlayClick` 设为 `undefined`，在 `super()` 后手动绑定 overlay 事件。
  - `designs/issue-19/src/ui/UIToast.ts` 与 `UIAlert.ts`: 引入 `private baseY: number`，在 `show()` 中使用 `y: this.baseY`。
  - `designs/issue-19/design-spec.md`: 将 warning 类型色条统一改为 `UITheme.warning`。
- **依据**: AGENTS.md 维护规则第 5 条 — "发现代码与文档不一致时，更新对应文档或在 tech-debt-tracker 中追加条目"。

## 轻微问题（Minor）

### 1. docs/frontend/ui-components.md 未按 PRD-004 要求重写
- **位置**: `docs/frontend/ui-components.md`
- **问题**: PRD-004 9.3 文档验收清单明确要求 "`docs/frontend/ui-components.md` 已重写为本 PRD 内容"。当前该文件仅在头部加了迁移提示，主体内容仍为旧版设计参考，未被重写。
- **修复**: 将该文件内容重写为指向 PRD-004、`docs/modules/client/ui.md` 和 `docs/design-docs/ui-style-guide/` 的导航页，或删除旧内容并放置重定向说明。
- **依据**: PRD-004 9.3 文档验收清单。

### 2. UIDialog 中 `overlay.off('pointerdown')` 的鲁棒性
- **位置**: `apps/phaser3/src/ui/UIDialog.ts:70`
- **问题**: `this.overlay.off('pointerdown')` 会移除 overlay 上所有 `pointerdown` 监听器。虽然当前 UIPanel 只注册了一个 `onOverlayClick` 空回调，但如果未来 UIPanel 追加其他监听器，会被误删。
- **修复**: 使用具名函数引用或 `removeAllListeners('pointerdown')`，并在注释中说明意图。

### 3. 部分组件文件末尾存在多余空行或缺少换行符
- **位置**: `apps/phaser3/src/ui/UIPanel.ts`（末行空行缺失后多了一个空行）、`UIDialog.ts`、`UIAlert.ts`、`UIToast.ts`
- **问题**: `git diff` 显示原 `export default` 删除后，部分文件末尾留下不一致的空白。不影响功能，建议统一格式化。
- **修复**: 运行 `pnpm --filter phaser3 lint` 自动修复（当前测试环境缺少 node_modules，无法验证）。

### 4. UIButton 构造函数未暴露 `depth` 参数
- **位置**: `apps/phaser3/src/ui/UIButton.ts:11-22`
- **问题**: `UIButtonConfig` 接口和 PRD-004 5.3 均缺少 `depth` 字段。虽然按钮通常不需要显式 depth，但 PRD-004 接口设计中 `UIButtonConfig` 未列出 `depth`，与其他组件不一致。
- **修复**: 如需严格对齐 PRD-004，可补充 `depth?: number` 并使用 `getNextDepth(UIDepthLayer.UI, scene)` 作为默认值。若故意省略，建议在 `docs/modules/client/ui.md` 中注明。

## 风险与建议

1. **设计资产与源码漂移风险**：`designs/issue-19/` 中的源码文件（`src/ui/*.ts`）与实际 `apps/phaser3/src/ui/*.ts` 存在功能性差异（`baseY`、`this` 前置访问、`closeOnOverlay` 绑定方式）。建议后续流程中明确：
   - `designs/issue-xxx/src/` 为**设计草稿**，迁移到 `apps/phaser3/src/` 后应以 `apps/` 目录为准；
   - 或建立自动化 diff 检查，防止设计资产中的错误代码被后续 Agent 直接复制粘贴。

2. **ToastManager 单例参数丢失风险**：将 `ToastManager` 构造函数设为 `private` 并强制使用 `getInstance(scene)` 虽然实现了单例，但丢失了 `startY` 配置能力。建议评估是否需要在 `getInstance` 中支持可选的 `startY` 覆盖，或让业务层完全自行管理 Y 坐标。

3. **UIButton 交互状态机已改善**：PR 将 `onClick` 从 `pointerdown` 的 tween `onComplete` 中移到了 `pointerup`，并新增了 `pointerupoutside` 取消逻辑，显著改善了手感，符合 PRD-004 3.6 状态机要求。但需配合 Major #1 的修复，确保 `setTexture()` 后状态机仍然完整。

4. **旧 API 兼容策略**：UIToast 保留了 `title`、`startTime`、`markAsComplete()` 等旧接口，并仅在检测到旧参数时自动 `show()`，这种渐进式迁移策略合理。但建议在 `docs/modules/client/ui.md` 中标注旧 API 的废弃计划与移除版本。

## 阻塞问题（Blocking）

### 1. Layer 2 通用交互组件完全缺失
- **位置**: `apps/phaser3/src/ui/`
- **问题**: PRD-004 明确要求新增 6 个 Layer 2 组件：`UISlider`、`UIToggle`、`UIStepper`、`UIDropdown`、`UIPagination`、`UILoading`。PR diff 中这 6 个文件完全不存在。
- **影响**: PRD-004 的交付范围是 Layer 1（重做）+ Layer 2（新增）。Layer 2 缺失意味着本 PR 未实现核心交付物，七大业务模块的前端开发将缺乏基础交互组件支撑。
- **修复**: 按 PRD-004 第 5.18~5.23 节接口规范实现上述 6 个组件，每个组件配套单元测试。
- **依据**: PRD-004 第 4.2 节 Layer 2 组件清单及第 9.1 节功能验收清单。

### 2. 使用 emoji/字符作为图标，违反 Foundation 禁止清单 F-01
- **位置**: 
  - `apps/phaser3/src/ui/UIAlert.ts:65-69` — `getTypeIcon()` 返回 `'✓'`, `'!'`, `'×'`, `'i'`
  - `apps/phaser3/src/ui/UIIconButton.ts:70-100` — `icon` 参数直接作为 `scene.add.text()` 渲染
  - `apps/phaser3/src/ui/UIListItem.ts:72` — `icon` 参数直接作为文本渲染
  - `apps/phaser3/src/ui/UIResourceDisplay.ts:64` — `icon` 参数直接作为文本渲染
  - `apps/phaser3/src/city-builder/ui/BuildingSelector.ts:26-33` — 硬编码中文字符 `'灵'`, `'林'` 等作为建筑图标
- **问题**: `docs/design-docs/ui-style-guide/01-foundation.md` 1.5 节明确禁止：「F-01 使用 emoji 作为图标或按钮」。PRD-004 第 2.1 节 P2 用户故事验收标准也明确「无 emoji」。当前实现使用 Unicode 符号（✓、×、!）和中文字符作为图标，与规范的国风水墨美术资产脱节。
- **影响**: 视觉风格不统一，破坏国风氛围，无法通过视觉验收。
- **修复**: 
  - `UIAlert`：移除 `getTypeIcon()`，改用 `UIIcon` 组件加载纹理，或仅用颜色条区分类型。
  - `UIIconButton`、`UIListItem`、`UIResourceDisplay`：`icon` 参数应改为纹理键名（`string` 类型但用于 `scene.add.image()`），缺失时使用 `UIIcon` 的回退矩形。
  - `BuildingSelector`：使用 `UIIcon` 加载实际建筑图标纹理。
- **依据**: `docs/design-docs/ui-style-guide/01-foundation.md` 第 1.5 节 F-01；PRD-004 第 2.1 节 P2 验收标准。

### 3. 深度值（depth）设置与 PRD-004 规范冲突
- **位置**:
  - `apps/phaser3/src/ui/UIDialog.ts:53` — `this.setDepth(5000)`
  - `apps/phaser3/src/ui/UIToast.ts:75` — `this.setDepth(4000)`
- **问题**: PRD-004 第 3.5 节深度规范明确规定：Dialog = 3000，Overlay = 4000，Toast = 5000+。当前 `UIDialog` 设置了 5000（抢占了 Toast 层级），`UIToast` 设置了 4000（落在了 Overlay 层级）。
- **影响**: 弹窗可能遮挡 Toast，Toast 可能被遮罩层覆盖，导致玩家看不到关键提示。
- **修复**:
  - `UIDialog.ts:53`: `this.setDepth(UIDepthLayer.Dialog)` 或 `getNextDepth(UIDepthLayer.Dialog, scene)`
  - `UIToast.ts:75`: `this.setDepth(UIDepthLayer.Toast)` 或 `getNextDepth(UIDepthLayer.Toast, scene)`
- **依据**: PRD-004 第 3.5 节深度规范表；`designs/issue-18/design-spec.md` 第 6.2 节深度分配状态。

## 严重问题（Major）

### 4. 组件接口与 PRD-004 规范严重不符（关键功能缺失）

以下组件未按 PRD-004 规定的接口实现，导致下游业务模块无法按契约使用：

| 组件 | 缺失/错误接口 | PRD-004 要求位置 | 当前代码位置 |
|------|--------------|-----------------|-------------|
| `UIPanel` | 无 `show()`/`hide()` 标准动画 | 5.5 节 | `UIPanel.ts` 仅提供 `getContentBounds()`、`addContent()` |
| `UIDialog` | 未继承 `UIPanel` | 5.6 节 | `UIDialog.ts:22` 继承 `Phaser.GameObjects.Container` |
| `UIDialog` | 无 `closeOnOverlay` 选项 | 5.6 节 | 遮罩设置了 `setInteractive()` 但无关闭逻辑 |
| `UIAvatar` | 无 `quality`、`isCircular`、`showBorder` | 5.9 节 | `UIAvatar.ts:11-20` 接口未定义这些字段 |
| `UIProgressBar` | `tweenTo()` 无中断保护 | 5.10 节 | `UIProgressBar.ts:109` 未调用 `killTweensOf` |
| `UIToast` | `ToastType` 与 `UINotificationType` 不一致 | 5.11 节 | `UIToast.ts:9` 为 `'build' \| 'upgrade' \| ...`，而 PRD 要求 `UINotificationType` |
| `UIResourceDisplay` | 无 `showWan`、`wanPrecision` | 5.12 节 | `UIResourceDisplay.ts:11-19` 接口缺失 |
| `UISegmentedControl` | 选项使用 `id` 而非 `key` | 5.13 节 | `UISegmentedControl.ts:10` 使用 `id` |
| `UISegmentedControl` | 无单项 `isDisabled` 支持 | 5.13 节 | `UISegmentedOption` 无 `isDisabled` |
| `UISegmentedControl` | 无 `setOptionDisabled()` 方法 | 5.13 节 | 类中无此方法 |
| `UIBadge` | 无 `maxWidth` 超长截断 | 5.16 节 | `UIBadge.ts:13-17` 接口缺失 |
| `UITooltip` | 无 `showDelay`（默认 300ms） | 5.17 节 | `UITooltip.ts:11-16` 接口缺失，直接显示 |

- **影响**: 业务模块（Fish、Weapon、Market 等）按 PRD-004 契约调用组件时会遇到编译错误或运行时行为不符。
- **修复**: 按上表逐项补齐接口和实现，确保与 PRD-004 第 5 节一致。
- **依据**: PRD-004 第 5 节各组件详细设计。

### 5. 动画叠加冲突：未先 `killTweensOf` 再播放新动画
- **位置**:
  - `UIButton.ts:91-96` — Hover 放大 tween 未先 kill 现有 twee

  - `UIButton.ts:103-108` — Hover 恢复 tween 未先 kill
  - `UIButton.ts:111-120` — Pressed tween 未先 kill
  - `UIIconButton.ts:115-120` — 同上
  - `UIAvatar.ts:90-98` — Pressed tween 未先 kill
  - `UIIcon.ts:73-82` — Pressed tween 未先 kill
  - `UITooltip.ts:121-125` — Show alpha tween 未先 kill
- **问题**: PRD-004 第 3.4 节动画统一标准明确禁止：「禁止连续触发时动画叠加，必须先 `killTweensOf(target)`」。当前实现中，快速连续点击或悬停移出再悬停时，多个 tween 会同时作用于同一目标，导致视觉跳变。
- **影响**: 快速交互时动画跳帧、冲突，影响用户体验。
- **修复**: 在每个 `this.scene.tweens.add()` 之前调用 `killTweensOf(this.scene, this)`（或对应目标）。例如：
  ```ts
  killTweensOf(this.scene, this);
  this.scene.tweens.add({ targets: this, scale: 1.05, ... });
  ```
- **依据**: PRD-004 第 3.4 节禁止项第 3 条。

### 6. 测试覆盖严重不足
- **位置**: `apps/phaser3/src/ui/__tests__/`
- **问题**: 
  - 仅 11 个测试文件覆盖约 19 个组件，缺失：`UIAlert.test.ts`、`UIBadge.test.ts`、`UITooltip.test.ts`、`UIScrollView.test.ts`、`UIInput.test.ts`、`UIText.test.ts`、`UIProgressBar.test.ts`、`UIResourceDisplay.test.ts`。
  - 现有测试极度浅层：绝大多数断言仅为 `expect(x).toBeDefined()`，未验证实际行为。
  - `UIButton.test.ts:23-30` 的 `onClick` 测试通过 `emitEvent(bg, 'pointerdown')` 触发，但由于 `MockTweenManager` 同步执行 `onComplete`，测试看似通过但并未验证真实事件流。
  - PRD-004 第 7.3 节要求：单元测试覆盖率 ≥ 80%，每个组件覆盖初始化、四态交互、资源回退、动画、内存清理。
- **影响**: 无法自动化验证组件行为，回归风险极高。
- **修复**: 
  1. 为所有未覆盖组件补充测试文件。
  2. 使用 `UIMockScene` + `simulateClick` / `simulateHover` 测试四态交互。
  3. 验证 `console.warn` 在纹理缺失时的触发（`assertFallbackWarn`）。
  4. 验证 `setDisabled(true)` 后 `disableInteractive()` 被调用。
- **依据**: PRD-004 第 7.3 节测试要求；`designs/issue-18/design-spec.md` 第 4 节测试基础设施设计。

### 7. `UIInput` 未清理全局事件监听器，存在内存泄漏风险
- **位置**: `apps/phaser3/src/ui/UIInput.ts:96-117`
- **问题**: `setupInteraction()` 中向 `this.scene.input.on('pointerdown', ...)` 和 `this.scene.input.keyboard?.on('keydown', ...)` 注册了全局监听器，但组件未提供 `destroy()` 重写以移除这些监听器。PRD-004 第 2.2 节边缘场景明确要求：「组件在 Scene 切换后未销毁 → 自动清理交互和 tween，避免内存泄漏」。
- **影响**: Scene 切换后，已销毁的 `UIInput` 的键盘和点击回调仍留在事件系统中，造成内存泄漏和意外行为。
- **修复**: 在 `UIInput` 中重写 `destroy()` 或添加 `cleanup()` 方法，移除 `this.scene.input` 和 `keyboard` 上注册的事件监听器。
- **依据**: PRD-004 第 2.2 节边缘场景；`designs/issue-18/design-spec.md` 第 4.3 节约束规则（每个 helper 必须处理事件监听器清理）。

### 8. 文档零更新
- **位置**: `docs/`
- **问题**: `.doc-coverage.txt` 显示「PR 中变更的文档文件」为空。PRD-004 第 9.3 节文档验收要求更新：`docs/frontend/ui-components.md`、`docs/modules/client/ui.md`、`docs/modules/client/INDEX.md`、`docs/design-docs/ui-style-guide/03-component.md`。

- **修复**: 
  1. 重写 `docs/frontend/ui-components.md` 为新版组件 API 文档。
  2. 更新 `docs/modules/client/ui.md` 和 `INDEX.md`。
  3. 同步更新 `docs/design-docs/ui-style-guide/03-component.md`。
- **依据**: PRD-004 第 9.3 节文档验收清单；`AGENTS.md` 维护规则第 6 条。

## 轻微问题（Minor）

### 9. 部分组件仍使用魔法数字色值
- **位置**:
  - `UIDialog.ts:78-83` — 直接写 `'#ffffff'`、`'#cccccc'` 而非通过 `UITheme` 或 `getInkTextStyle`
  - `UIListItem.ts:83` — `'#dddddd'`、`'#888888'`
  - `UIScrollView.ts:71` — `0x333333`
  - `UITooltip.ts:46` — `UITheme.inkBlack` 正确，但 `0.95` 是魔法数字透明度
- **影响**: 风格漂移风险，后期难以统一修改。
- **修复**: 统一使用 `UITheme` 常量，文字颜色通过 `getInkTextStyle` 注入。
- **依据**: `docs/design-docs/ui-style-guide/01-foundation.md` 第 1.1 节；`docs/frontend/agent-design-guide.md` 3.2 节检查清单。

### 10. 组件使用 `export default`
- **位置**: 几乎所有组件文件末尾有 `export default Xxx;`
- **问题**: `designs/issue-18/design-spec.md` 第 2.3 节约束规则要求：「所有类型以 `export` 暴露，不使用 `export default` — 强制命名导入」。虽然该约束写于 `types.ts` 章节，但 `index.ts` 也使用命名导出，组件文件使用 `default export` 不一致。
- **影响**: 导入风格不一致，可能导致 tree-shaking 问题。
- **修复**: 移除所有组件文件的 `export default`，统一使用命名导出。
- **依据**: `designs/issue-18/design-spec.md` 第 2.3 节约束规则。

### 11. `UIListItem` 在未选中态仍创建可交互背景
- **位置**: `UIListItem.ts:59-61`
- **问题**: `createBackground()` 中无条件调用 `this.background.setInteractive({ useHandCursor: true })`。当 `UIListItem` 被嵌入 `UIList` 使用时，`UIList` 在容器级别处理交互（`UIList.ts:114-128`），导致 `UIListItem` 背景和容器同时响应指针事件，可能触发重复回调。
- **影响**: 潜在的重复事件触发和交互冲突。
- **修复**: `UIListItem` 的 `setInteractive` 改为可选（由传入参数 `isInteractive` 控制），或由 `UIList` 统一控制交互。
- **依据**: PRD-004 第 5.7 节 `UIListItem` 接口设计。

### 12. `UIResourceDisplay` 默认背景色异常
- **位置**: `UIResourceDisplay.ts:38` — `bgColor: config.bgColor ?? 0xffffff`
- **问题**: 默认背景色为纯白 `0xffffff`，与 `UITheme` 规范不符，且 `createBackground()` 中设置了 `alpha: 0.05` 和 `strokeColor` 透明度 `0.3`，导致资源显示组件几乎透明。
- **影响**: 视觉不一致，在复杂背景下可能不可见。
- **修复**: 默认 `bgColor` 改为 `UITheme.inkBlack` 或不绘制背景（仅保留边框）。
- **依据**: `docs/design-docs/ui-style-guide/01-foundation.md` 第 1.1 节色板规范。

## 风险与建议

### 高风险
1. **合并后业务模块开发受阻**: 由于 Layer 2 组件缺失 + Layer 1 接口与 PRD-004 不符，后续开发 Fish、Weapon 等模块的 Agent 会频繁遇到「组件没有这个 API」的问题，导致大量返工。
2. **视觉验收必挂**: emoji/字符图标 + 深度值错误 + 动画叠加，在视觉验收 SOP 中多项不通过。

### 建议修复优先级
1. **P0（阻塞）**: 补充 Layer 2 组件 或 缩减 PR 范围（如果本 PR 仅计划做 Layer 1，需在 PR 描述中明确并修改 PRD-004 范围）
2. **P0（阻塞）**: 移除所有 emoji/字符图标，改用纹理回退机制
3. **P0（阻塞）**: 修正 `UIDialog` 和 `UIToast` 的 depth 值
4. **P1（严重）**: 补齐 PRD-004 规定的缺失接口（`UIPanel.show/hide`、`UIDialog` 继承 `UIPanel`、`UIAvatar` 品质/圆形遮罩 等）
5. **P1（严重）**: 补充缺失的单元测试文件，提升测试深度
6. **P1（严重）**: 为 `UIInput` 添加事件监听器清理逻辑
7. **P2（轻微）**: 统一移除 `export default`，清理魔法数字色值

### 关于 PR 范围的建议
当前 PR diff 仅修改了 Layer 1 组件（+ `UIInput`、`UIScrollView`），未涉及 Layer 2。如果本 PR 的定位就是「Phase 1 基础设施 + Layer 1 重做」，建议：
1. 修改 PR 标题和描述，明确排除 Layer 2 组件
2. 在 PRD-004 或执行计划中创建后续 Issue 跟踪 Layer 2 实现
3. 但即便如此，Layer 1 组件的接口偏差（`UIPanel.show/hide`、`UIDialog` 继承 `UIPanel` 等）仍需在本 PR 修复，因为这些是 Layer 1 原子组件的基础契约。

## 审查结论

审查结论: **NEEDS_MAJOR_FIX**

PR #35（分支 `feat/issue-5`）实现了 IM V1 实时聊天系统（issue #5）及配套的 E2E 框架调整（issue #64），同时包含前端 UI 组件的增量修改。后端分层架构遵循了项目规范，但存在 **2 个阻塞性问题** 导致构建/运行失败，以及 **5 个严重问题** 影响数据一致性和需求完整性。

## 阻塞问题（Blocking）

### 1. Protobuf 生成文件缺失，导致编译失败
- **位置**: `api/new-world-api/v1/im.proto`
- **问题**: PR 新增了 `im.proto`，但未提交对应的 Go 生成产物（`.pb.go`、`_grpc.pb.go`、`_http.pb.go`）。
- **影响**: 任何引用 `apipb` 中 IM 相关类型的代码（如 `internal/biz/wapi/im.go`、`internal/service/wapi/im.go`）都无法编译，CI/本地构建直接失败。
- **修复**:
  ```bash
  make api
  # 或
  make proto
  ```
将生成的 `api/new-world-api/v1/im.pb.go`、`im_grpc.pb.go`、`im_http.pb.go` 加入 Git 并提交。
- **依据**: `docs/design-docs/proto.md` 第 86-93 行明确规定："Go 消息代码位于 `api/**/*.pb.go`（与 proto 同目录）"；"新增或修改 `.proto` 源后，必须重新执行 `make proto-frontend` 并提交变更"。

### 2. MySQL Driver 依赖被误删，运行时数据库连接将失败
- **位置**: `go.mod`（第 16 行变更）
- **问题**: `go.mod` 删除了直接依赖 `github.com/go-sql-driver/mysql v1.9.3`（main 分支存在）。项目中无任何代码直接 `import` 该包，但 `internal/data/world/ent/client.go:192` 的 `Open()` 函数调用 `sql.Open("mysql", dataSourceName)`，依赖该 driver 的 `init()` 副作用注册。`go mod why` 在 feat/issue-5 上明确返回 "main module does not need package github.com/go-sql-driver/mysql"。
- **影响**: 服务启动时调用 `world.Open("mysql", ...)` 不会立即报错（`sql.Open` 懒加载），但在首次执行实际 SQL 查询时会因找不到 "mysql" driver 而 panic/fail。
- **修复**:
在 `go.mod` 中恢复该依赖，或在 `internal/data/wapi/data.go` 中添加 side-effect import：
  ```go
  import _ "github.com/go-sql-driver/mysql"
  ```
然后执行 `go mod tidy`。
- **依据**: `docs/design-docs/ent.md` 明确技术栈为 "驱动：`github.com/go-sql-driver/mysql`"；`internal/data/world/ent/client.go` 使用 `sql.Open("mysql", ...)`。

## 严重问题（Major）

### 3. SendMessage 多表写操作缺少事务保护
- **位置**: `internal/biz/wapi/im.go:80-88`
- **问题**: `SendMessage` 依次调用 `CreateMessage` → `UpdateConversationLastMessage` → `IncrUnreadCount`，三者涉及 `im_messages` 和 `conversations` 两张表的写入，但没有任何事务边界。若中间步骤失败，会产生消息已入库但会话未更新、或未读数未增加的数据不一致。
- **影响**: 高并发或异常场景下，会话列表与消息状态不一致，用户体验受损。
- **修复**:
在 `IMRepo` 接口中增加 `Tx` 版本的方法（如 `CreateMessageTx`、`UpdateConversationLastMessageTx`、`IncrUnreadCountTx`），并在 `IMUseCase.SendMessage` 中使用 `repo.TxProvider.WithTx` 编排事务：
  ```go
  return uc.txHelper.WithTx(ctx, func(tx repo.Tx) error {
      if err := uc.imRepo.CreateMessageTx(ctx, tx, msg); err != nil { return err }
      if err := uc.imRepo.UpdateConversationLastMessageTx(ctx, tx, ...); err != nil { return err }
      if err := uc.imRepo.IncrUnreadCountTx(ctx, tx, ...); err != nil { return err }
      return nil
  })
  ```
- **依据**: `docs/architecture/layers.md` 第 12.1 节 "多表更新需要事务"、第 12.2 节 "事务应该在 Biz 层开启"。

### 4. MarkAsRead 对持久化失败静默处理，返回虚假成功
- **位置**: `internal/biz/wapi/im.go:252-259`
- **问题**: `BatchCreateReadReceipts` 和 `ClearUnreadCount` 出错时仅记录 `Warn` 日志，函数仍然返回 `read_count = len(req.MessageIds)` 的成功响应。
- **影响**: 客户端收到 "已读成功"，但服务端可能未写入已读回执或未清空未读数。重新拉取历史消息时，已读状态回退，未读数不减少。
- **修复**:
  ```go
  if err := uc.imRepo.BatchCreateReadReceipts(ctx, receipts); err != nil {
      return nil, err  // 返回错误，让客户端重试
  }
  if err := uc.imRepo.ClearUnreadCount(ctx, req.ConversationId, uidStr); err != nil {
      return nil, err
  }
  ```
- **依据**: `docs/design-docs/error-handling.md` 第 3.1 节 "不要创建临时错误"、第 10.4 节 "不丢失原始错误"；`layers.md` 第 10.1 节 Biz 层应返回业务错误。

### 5. 缺少 IM 核心功能的 E2E 测试
- **位置**: `test/e2e/scenario/`
- **问题**: PR commit message 包含 "fix: resolve #64 - E2E for Issue #64"，但变更文件列表中没有 IM 相关的 E2E 场景（如 `im_flow_test.go`）。现有变更仅包含：
  - E2E 客户端框架从 JSON 改为 Protobuf（`http_client.go`、`base.go`）
  - `auth_flow_test.go` 和 `leaderboard_flow_test.go` 的适配
- **影响**: IM 的 4 个核心接口（SendMessage、GetConversationList、GetMessageHistory、MarkAsRead）及 WebSocket 推送链路没有任何端到端自动化验证，无法保证功能在集成环境中的可用性。
- **修复**:
新增 `test/e2e/scenario/im_flow_test.go`，覆盖以下场景：
  1. 用户 A 注册 → 用户 B 注册 → A 给 B 发送消息
  2. B 获取会话列表，断言未读数为 1
  3. B 获取消息历史，断言消息内容正确
  4. B 标记已读，断言返回 read_count = 1
  5. B 再次获取会话列表，断言未读数为 0
- **依据**: `docs/design-docs/development-workflow.md`（测试流程）要求核心功能具备 E2E 覆盖；PR 自身标题声明 "E2E for Issue #64"。

### 6. 新增 IM 模块但未更新模块文档索引
- **位置**: `docs/modules/INDEX.md`
- **问题**: PR 新增了一套完整的 IM 后端模块（Proto → Service → Biz → Data → Ent Schema → SQL），但 `docs/modules/INDEX.md` 中没有任何 IM 模块的条目，也未创建 `docs/modules/im/` 目录。
- **影响**: 后续开发者无法从文档索引找到 IM 模块的实现位置，违反 "禁止孤立文档" 原则。
- **修复**:
  1. 在 `docs/modules/` 下新建 `im/README.md`，包含模块概述、代码位置（`internal/biz/wapi/im.go`、`internal/service/wapi/im.go` 等）、跨模块依赖（user 模块）。
  2. 在 `docs/modules/INDEX.md` 的模块列表中增加 IM 行。
- **依据**: `AGENTS.md` 第 98 行 "新增文档后更新对应索引"；`docs/modules/INDEX.md` 自身定义的 "新增模块流程"。

### 7. UISegmentedControl 接口变更与 PRD-004 规范不一致
- **位置**: `apps/phaser3/src/ui/UISegmentedControl.ts:13-14`
- **问题**: PR 将 `UISegmentedOption` 的字段从 `key` 改为 `id`，但关联 PRD `PRD-004-ui-component-library-redesign.md` 的 5.13 节明确规定接口为 `key: string`。虽然 PRD-004 状态为 Draft，但前端 UI 组件库重做是 issue #17 的规范，当前变更引入了与规范不一致的接口命名。
- **影响**: 如果其他开发者或设计资产基于 PRD-004 的 `key` 接口进行开发，会产生集成冲突。
- **修复**:
方案 A：恢复 `key` 命名，与 PRD-004 保持一致； 方案 B：如果团队已决定采用 `id`，则同步更新 PRD-004 的 5.13 节，并检查所有设计资产和调用方。
- **依据**: `prd/PRD-004-ui-component-library-redesign.md` 第 738-741 行定义 `UISegmentedOption.key`；`docs/frontend/agent-design-guide.md` 要求 "代码编写前读取规范"。

## 轻微问题（Minor）

### 8. GetMessageHistory 未填充 sender_name
- **位置**: `internal/biz/wapi/im.go:215-224`
- **问题**: `IMMessageContent` proto 定义包含 `sender_name`（第 54 行），但 `GetMessageHistory` 的响应构建中只填充了 `sender_id`，未填充 `sender_name`。前端在聊天历史中可能只显示 UID 而非用户名。
- **修复**: 在循环中通过 `userRepo.FindOne` 查询发送者名称，或调整 proto 让前端自行查询。

### 9. GetMessageHistory 分页边界条件处理不完善
- **位置**: `internal/data/wapi/im.go:178-184`
- **问题**: 当 `beforeMessageID` 不为空但对应消息不存在时，代码静默忽略（`if err == nil && refMsg != nil`），导致 `query` 没有添加 `SentAtLT` 条件，可能返回最新的消息而非预期分页。
- **修复**: 若 `beforeMessageID` 存在但查不到消息，应返回空列表或错误，避免返回不相关的最新消息。

### 10. go.mod 中额外删除了未确认是否安全的依赖
- **位置**: `go.mod` 变更
- **问题**: 除 `go-sql-driver/mysql` 外，`go.mod` 还删除了 `go.uber.org/automaxprocs`、`filippo.io/edwards25519`、`golang.org/x/tools` 等。`automaxprocs` 可能被某些初始化逻辑依赖（如容器环境 CPU 限制自适应）。
- **修复**: 单独审查 `go.mod` 的 tidy 变更，确认每个被删除的依赖是否真的没有直接或间接使用者。建议将 `go mod tidy` 的变更单独作为一个 commit，并说明理由。

## 风险与建议

1. **运行时数据一致性风险（高）**：`SendMessage` 无事务保护是当前最大的业务风险。建议在合并前通过 `repo.TxProvider` 补充事务，或至少记录技术债务并在 `docs/exec-plans/tech-debt-tracker.md` 中跟踪。
2. **前端 API 兼容性风险（中）**：`UIBadge.setSize` → `setBadgeSize` 的改名、`UISegmentedControl` 的 `key` → `id` 改名，都是公共 API 的破坏性变更。虽然 grep 未在本仓库发现调用方，但如果存在未提交的分支或外部插件依赖这些接口，会产生编译错误。建议通过全局搜索（含 `designs/` 目录）再次确认。
3. **依赖清理风险（中）**：`go.mod` 的 tidy 操作删除了多个依赖。建议在独立 PR 中进行依赖清理，并附带 `go mod why` 的输出作为证据，避免与业务功能 PR 混合。
4. **WebSocket 推送未经验证**：`IMWsPushHandler` 通过 `biz.WwsHandler.Publish` 推送消息和已读回执，但 PR 中无 WebSocket 相关的 E2E 或集成测试。建议补充至少一个覆盖 "发送消息后接收者通过 WS 收到推送" 的测试。
5. **建议分阶段合并**：如果阻塞问题（protobuf 生成文件、MySQL driver）修复迅速，可考虑先合并后端 IM 核心，再补充 E2E 和文档。但事务问题和 MarkAsRead 错误处理问题建议在合并前修复。

## 严重问题（Major）

### 1. `setup.ts` 硬编码导入路径多了 1 层 `../`

- **位置**: `designs/issue-18/src/ui/__tests__/setup.ts:17`
- **问题**: 导入语句使用了 `../../../../../../apps/phaser3/src/test-setup`（6 个 `../`），但从文件所在目录 `designs/issue-18/src/ui/__tests__/` 向上只需 5 层即可到达仓库根目录，6 层会指向仓库父目录，导致路径解析失败。
- **影响**: 任何尝试在 design 目录内直接编译、类型检查或运行测试的操作都会因找不到模块而失败；IDE 也会标红该导入。
- **修复**: 将第 17 行的 `../../../../../../apps/phaser3/src/test-setup` 改为 `../../../../../apps/phaser3/src/test-setup`（去掉 1 个 `../`）。
- **依据**: 通过 Node.js `path.resolve` 验证：
  ```
  6 ../ → /private/.../T/apps/phaser3/src/test-setup      (不存在)
  5 ../ → /private/.../T/review-pr-27-2328859522/apps/... (存在)
  ```

### 2. `createFallbackRect` 的 `textureKey` 为可选参数，破坏 PRD-004 回退告警契约

- **位置**: `designs/issue-18/src/ui/utils.ts:145-151`
- **问题**: `textureKey?: string` 被声明为可选。当调用方省略该参数时，`createFallbackRect` 不会打印 `console.warn('[UI] fallback: ...')`，直接静默回退。PRD-004 3.3 明确要求**所有**纹理缺失回退必须打印 warn，且 `docs/design-docs/ui-style-guide/01-foundation.md` F-08 禁止"纹理缺失时静默失败"。
- **影响**: 后续 Developer Agent 在使用该工具函数时可能遗漏 `textureKey`，导致回退行为静默，违反验收标准（"每个纹理缺失组件都输出 `[UI] fallback: ${key}` warn"）。
- **修复**: 将 `textureKey?: string` 改为 `textureKey: string`，强制调用方提供纹理键；若某些场景确实不需要 warn，应单独提供 `createInkBackground` 而非使用 `createFallbackRect`。
- **依据**: PRD-004 第 3.3 节"资源回退统一标准"第 3 条；`docs/design-docs/ui-style-guide/01-foundation.md` F-08。

### 3. 迁移说明中提到的 `TEST_SETUP_PATH` 常量在实际文件中不存在

- **位置**: 
  - `designs/issue-18/src/ui/index.ts:119`
  - `designs/issue-18/design-spec.md:267`
- **问题**: 迁移指南写道"将文件内的 `TEST_SETUP_PATH` 替换为 `'../../test-setup'`"，但 `setup.ts` 中并没有名为 `TEST_SETUP_PATH` 的常量或变量，而是直接使用字面量路径 `../../../../../../apps/phaser3/src/test-setup`（见问题 1）。Developer Agent 按此说明执行时会找不到替换目标，导致迁移步骤卡住或遗漏。
- **影响**: 开发 Agent 迁移文档时产生困惑，增加沟通成本；可能导致路径替换遗漏，留下错误的硬编码路径。
- **修复**: 
  - 在 `index.ts:119` 将说明改为："将文件内的 `../../../../../../apps/phaser3/src/test-setup` 替换为 `'../../test-setup'`"
  - 在 `design-spec.md:267` 同步修正相同描述，或删除该 checklist 项（因为字面量路径替换已在 `index.ts` 的注释中说明）。
- **依据**: `grep -r "TEST_SETUP_PATH" designs/issue-18/` 仅在 `index.ts` 和 `design-spec.md` 命中，未在 `setup.ts` 中出现。

## 轻微问题（Minor）

### 4. `expectTweenCreated` 返回 `boolean` 而非内部断言

- **位置**: `designs/issue-18/src/ui/__tests__/setup.ts:253-258`
- **问题**: 该 helper 返回 `boolean`，调用方必须手动写 `expect(expectTweenCreated(scene, target)).toBe(true)`。若调用方忘记外层 `expect`，测试将始终"通过"（返回值被忽略）。
- **修复**: 在函数内部使用 `expect(...).toBe(true)` 直接断言，并修改返回类型为 `void`。
- **依据**: 测试 helper 的设计惯例——断言辅助函数应自行断言，减少调用方出错概率。

### 5. `cleanup()` 未清理 `mockTexture` 补丁与 depth 计数器

- **位置**: `designs/issue-18/src/ui/__tests__/setup.ts:319-321`
- **问题**: `cleanup()` 仅委托 `baseCleanup()`，没有调用 `restoreMockTextures()` 或 `resetDepthCounters()`。虽然 `beforeEach`通常会新建 `MockScene`，但如果某个测试复用场景对象（或因弱引用未立即回收），`textures.exists` 的猴子补丁和 `depthCounters` 中的状态会泄漏到下一个测试。
- **修复**: 在 `cleanup()` 中补充对全局/模块级状态的清理；或在 `setup.ts` 顶部导出 `restoreMockTextures` 的包装，并在文档中说明"每个测试必须新建 UIMockScene"。
- **依据**: 设计规范 `design-spec.md` 4.3 第 2 条："每个 helper 必须处理事件监听器清理 — 防止测试间状态泄漏"。

### 6. `UIMockScene` 未代理 `scene.sys`、`scene.game`、`scene.events` 等常用属性

- **位置**: `designs/issue-18/src/ui/__tests__/setup.ts:35-121`
- **问题**: `UIMockScene` 只代理了 `cameras`、`tweens`、`time`、`input`、`textures`、`add` 以及事件方法。某些组件可能在 `shutdown` 清理逻辑中访问 `scene.sys` 或 `scene.events`，此时会抛出 `undefined` 异常。
- **修复**: 追加 `get sys(): any { return this._mockScene.sys; }` 等常用属性的 proxy getter；或在类注释中显式列出"不支持的属性清单"。
- **依据**: 设计规范 `design-spec.md` 4.1："扩展现有 MockScene"，应尽可能兼容现有 MockScene 的接口面。

## 风险与建议

1. **迁移路径风险**：`designs/issue-18/src/ui/` 中的代码最终需被 Developer Agent 复制到 `apps/phaser3/src/ui/`。建议在下一次开发 PR 中：
   - 修复上述 Major 问题后再进行复制；
   - 同步将 `docs/frontend/ui-components.md` 按 PRD-004 重写（PRD-004 9.3 要求）。

2. **类型兼容性风险**：`types.ts` 将旧版分散的 `ToastType`、`UIAlertType`、`UISegmentedOption` 等统一/重命名（如 `UINotificationType`、`UIOptionItem`）。迁移时需检查 `apps/phaser3/src/ui/` 下的现有组件（`UIToast.ts`、`UIAlert.ts`、`UISegmentedControl.ts` 等）是否已按新类型名更新，避免 `tsc` 报错。

3. **测试 helper 成熟度**：当前 `setup.ts` 缺少拖拽事件模拟（`pointermove` 序列），而 PRD-004 Layer 2 的 `UISlider` 需要此类能力。建议在开发 Layer 2 组件前补充 `simulateDrag(from, to)` 或 `simulatePointerMove(target, x, y)` helper。

4. **色板一致性**：`utils.ts` 新增 `UITheme.warning: 0xffaa00`，但 `docs/design-docs/ui-style-guide/01-foundation.md` 的色板表格尚未收录该颜色。建议在合并后追加一行说明，或在开发 PR 中同步更新 Foundation 文档。

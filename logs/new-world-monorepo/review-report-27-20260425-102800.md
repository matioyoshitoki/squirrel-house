## 轻微问题（Minor）

### 1. `UIFallbackConfig` 类型与实际工具函数签名不匹配
- **位置**: `designs/issue-18/src/ui/types.ts:90-99` 与 `designs/issue-18/src/ui/utils.ts:127-160`
- **问题**: `types.ts` 导出了 `UIFallbackConfig` 接口（含 `fillColor / strokeColor / alpha / textureKey`），但 `utils.ts` 中的 `createFallbackRect()` 并未消费该类型，而是将参数平铺为独立形参，且 `alpha` 字段在 `createFallbackRect` 中被硬编码为 `0.9`，无法通过配置传入。
- **影响**: 类型与实际 API 脱节，开发者可能误以为可以传入 `UIFallbackConfig` 对象。
- **修复**: 
  - 方案 A：新增 `createFallbackRect(scene, config: UIFallbackConfig & { width: number; height: number })` 重载或替换现有签名；
  - 方案 B：若该类型仅作未来预留，在 `UIFallbackConfig` 的 JSDoc 中标注 `@internal 预留类型，暂未与 createFallbackRect 绑定`，并删除 `index.ts` 中的导出，待 Phase 2/3 统一接入。
- **依据**: `designs/issue-18/design-spec.md` §3.3 约束规则要求“所有新增工具函数必须有 JSDoc”，且 `types.ts` 约束规则要求类型应说明典型使用场景；当前 `UIFallbackConfig` 无使用场景。

### 2. `qualityColorMap` 色值与“国风色板”风格存在偏差
- **位置**: `designs/issue-18/src/ui/utils.ts:320-330`
- **问题**: `qualityColorMap` 使用了标准 Material Design 色系（`0x4caf50`、`0x2196f3`、`0x7c4dff` 等），而项目 `UITheme` 定义的是低饱和度国风水墨色系（`0x1a1a1a`、`0xc45c48`、`0x3d5a80` 等）。PRD-004 §5.16 明确要求“品质类型 `rare/epic/legendary` 映射到**国风色值**（与 `quality-icon` 色系对齐）”。
- **影响**: 纹理缺失时的回退纯色底色风格与整体国风水墨体系不一致。
- **修复**: 将 `qualityColorMap` 色值调整为与 `UITheme` 同谱系的低饱和度国风色，例如：
  ```ts
  export const qualityColorMap: UIQualityColorMap = {
    1: 0xcccccc, // 白
    2: 0x8fbc8f, // 国风浅绿
    3: 0x5f9ea0, // 国风青绿
    4: 0x4a9eff, // 石青（与 UITheme.stoneCyan 对齐）
    5: 0x3d5a80, // 黛蓝（与 UITheme.indigoBlue 对齐）
    6: 0x6a5acd, // 国风紫
    7: 0xc9a227, // 金棕（与 UITheme.goldBrown 对齐）
    8: 0xc45c48, // 朱砂红（与 UITheme.cinnabarRed 对齐）
    9: 0xa52a2a, // 国风深红
  };
  ```
具体值需与美术 `quality-icon-*.png` 实际色系对齐，但应避免高饱和荧光色。
- **依据**: PRD-004 §5.16 / `docs/design-docs/ui-style-guide/01-foundation.md` 色板规范 / `docs/design-docs/ui-style-guide/03-component.md` 资源回退契约。

### 3. `UIDepthLayer` 6 层模型与现有 Foundation 文档的 4 层模型冲突，需跟踪技术债务
- **位置**: `designs/issue-18/src/ui/types.ts:55-62` vs `docs/design-docs/ui-style-guide/01-foundation.md` §1.4
- **问题**: 新增 `UIDepthLayer` 将深度拆分为 6 层（Scene/UI/Panel/Dialog/Overlay/Toast），而现有 `01-foundation.md` 为 4 层（Scene/ UI / Dialog&Panel / Modal&Toast）。设计规格文档 `design-spec.md` 已明确承认该差异，并声明“将在 Phase 4 文档同步时统一更新”。
- **影响**: 若开发 Agent 在 Phase 2/3 直接使用 6 层模型，而 Foundation 文档未同步更新，会导致文档与实现漂移。
- **修复**: 在 `docs/exec-plans/tech-debt-tracker.md` 追加条目：
  ```markdow

  - [ ] UI 深度层级模型同步：`designs/issue-18` 引入 6 层 UIDepthLayer，需回写 `docs/design-docs/ui-style-guide/01-foundation.md` §1.4
    - 责任人: Phase 4 文档同步 Agent
    - 截止日期: PRD-004 Phase 4 完成前
  ```
- **依据**: `AGENTS.md` 维护规则第 5 条：“债务同步：发现代码与文档不一致时，更新对应文档或在 `exec-plans/tech-debt-tracker.md` 中追加条目。”

### 4. `UIMockScene` 过度使用 `any` 类型
- **位置**: `designs/issue-18/src/ui/__tests__/setup.ts:35-127`
- **问题**: `UIMockScene` 内部 `_mockScene` 及所有 proxy getter 均声明为 `any`，削弱了 TypeScript 类型保护。虽然测试基础设施需要灵活性，但核心属性（`cameras`、`tweens`、`textures`、`add` 等）可基于现有 `MockScene` 的结构赋予更具体的类型。
- **影响**: 测试代码中使用 `scene.native` 或 `scene.add` 时丢失自动补全和编译期检查。
- **修复**: 为 `_mockScene` 添加内部接口类型（可基于 `apps/phaser3/src/test-setup.ts` 中的 `MockScene` 类类型），并将 proxy getter 的返回类型从 `any` 收窄到 `MockScene['xxx']`。
- **依据**: `docs/design-docs/testing.md` 未明确禁止测试 mock 中使用 `any`，但项目整体 TypeScript 规范倾向强类型；且 `pnpm --filter phaser3 type-check` 若开启严格模式会对此类 `any` 产生警告。

### 5. `index.ts` 注释中遗留旧类型名 `UIAlertType`
- **位置**: `designs/issue-18/src/ui/index.ts:93`
- **问题**: Layer 1 注释导出中使用了 `type UIAlertType`，而 PRD-004 §5.2 已将 Alert/Toast/Badge 的统一通知类型定义为 `UINotificationType`。`types.ts` 也正确导出了 `UINotificationType`。
- **影响**: 开发 Agent 在 Phase 2 取消注释时，可能沿用旧类型名，导致与 PRD-004 规范不一致。
- **修复**: 将注释中的 `type UIAlertType` 改为 `type UINotificationType`，或说明 `UIAlert` 内部若仍保留 `UIAlertType` 别名，则该别名应基于 `UINotificationType` 定义。
- **依据**: PRD-004 §3.2 命名规范 / PRD-004 §5.2 公共类型定义。

## 风险与建议

1. **迁移路径风险**：`designs/issue-18/src/ui/__tests__/setup.ts` 第 17 行硬编码了 `../../../../../apps/phaser3/src/test-setup` 的相对路径。当开发 Agent 将其复制到 `apps/phaser3/src/ui/__tests__/setup.ts` 后，必须手动替换为 `'../../test-setup'`。建议在下一次迁移 PR 中，将路径修正作为首条 commit，避免编译失败。

2. **旧分支产物污染风险**：`design-spec.md` 明确报告了 `origin/feature/issue-18-ui-infrastructure` 分支中存在 React + Tailwind CSS 方向的错误产物。建议仓库维护者**删除或归档**该错误分支，防止后续 Agent 误用。

3. **`createInkBackground` 透明度行为变更风险**：现有 `apps/phaser3/src/ui/utils.ts` 中 `createInkBackground` 的默认 alpha 为 `0.95`，新设计改为 `0.9`。迁移覆盖后，依赖旧透明度的组件（如 `UIPanel`、`UIDialog`）视觉可能微变。建议在 Phase 4 视觉验收时重点关注面板透明度差异。

4. **Layer 1 / Layer 2 组件尚未实现**：`index.ts` 中全部 22 个组件导出均被注释。这是 Phase 1 的预期状态，但意味着本 PR 合并后，`apps/phaser3/src/ui/` 的运行时代码**没有任何变化**。后续 Phase 2/3 的 PR 将体积庞大，建议拆分为每批 3-5 个组件的增量 PR，便于审查。

5. **测试基础设施自测建议**：`getNextDepth` 的 WeakMap 隔离逻辑、`killTweensOf` 的回调触发顺序、`mockTexture` / `restoreMockTextures` / `cleanup` 的全局状态清理，均建议在 `designs/issue-18/src/ui/__tests__/setup.test.ts` 中补充单元测试，确保测试 helper 自身正确。

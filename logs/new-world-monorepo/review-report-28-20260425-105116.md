## 阻塞问题（Blocking）

### 1. UIAlert 新增 `height` 配置项但渲染时仍硬编码 44px
- **位置**: `apps/phaser3/src/ui/UIAlert.ts:32` 与 `:72`
- **问题**: PR 在 `Required<UIAlertConfig>` 中新增了 `height: config.height ?? 44`，但在 `createBackground()` 方法中仍硬编码 `44`：
  ```typescript
  // 第 32 行：已加入 height 到 Required 配置
  height: config.height ?? 44,
  
  // 第 72 行：仍硬编码 44，未使用 this.config.height
  this.background = this.scene.add.rectangle(0, 0, this.config.width, 44, UITheme.inkBlack, 0.95);
  ```
- **影响**: 调用方传入 `height` 参数后期望改变 Alert 高度，但实际高度始终为 44px，造成接口契约与行为不一致。
- **修复**: 将第 72 行的 `44` 替换为 `this.config.height`：
  ```typescript
  this.background = this.scene.add.rectangle(0, 0, this.config.width, this.config.height, UITheme.inkBlack, 0.95);
  ```
同时检查 `createBackground()` 中的 `leftBar` 高度（第 77 行 `40`）是否需要基于 `this.config.height` 动态计算，或保持固定内边距设计。
- **依据**: `git diff` 显示 `height` 是本次 PR 新增到 `UIAlertConfig` 的字段，但对应的渲染逻辑未同步更新。

## 严重问题（Major）

### 1. `createInkBackground()` 与 `createFallbackRect()` 原点设置不一致
- **位置**: `apps/phaser3/src/ui/utils.ts:138-148` 与 `:164-177`
- **问题**: `createFallbackRect()` 内部调用 `rect.setOrigin(0.5)`，而 `createInkBackground()` 没有。在 Phaser3 中 `add.rectangle()` 默认原点是左上角 `(0, 0)`，因此两个函数的返回对象在容器中的定位行为不同：
  - `createFallbackRect` → 矩形中心在 `(0, 0)`
  - `createInkBackground` → 矩形左上角在 `(0, 0)`
- **影响**: 现有组件（如 `UIButton`、`UIPanel`）在使用 `createInkBackground` 后都**额外补了一行** `(this.background).setOrigin(0.5)` 来修正。如果未来有开发者误以为 `createInkBackground` 和 `createFallbackRect` 行为一致，直接替换使用，会导致组件位置偏移半个宽高。
- **修复**: 统一两个函数的原点行为。推荐在 `createInkBackground` 中追加 `bg.setOrigin(0.5)`，并同步移除 `UIButton.ts`、`UIPanel.ts` 等组件中冗余的 `setOrigin(0.5)` 调用（共 2 处，搜索 `(this.background).setOrigin(0.5)` 即可找到）。
- **依据**: 
  - PRD-004 第 3.3 节要求"回退元素尺寸与预期纹理一致"，隐含要求定位行为一致。
  - `docs/design-docs/ui-style-guide/01-foundation.md` F-02 禁止直接 `new Rectangle` 硬编码，而工具层本身应提供行为一致的封装。

## 轻微问题（Minor）

### 1. `expectFallbackWarn()` 命名 misleading
- **位置**: `apps/phaser3/src/ui/__tests__/setup.ts:107-110`
- **问题**: 函数名以 `expect` 开头，容易让人误以为是 Vitest 风格的断言（如 `expect(x).toBe(y)`），但它实际上只是创建并返回一个 `vi.spyOn(console, 'warn')`。真正的断言逻辑在 `assertFallbackWarn()` 中。
- **影响**: 开发者可能写出 `expectFallbackWarn('missing-tex')` 以为会自动断言，导致测试假通过。
- **修复建议**: 将函数重命名为 `createFallbackWarnSpy()` 或 `spyOnFallbackWarn()`，以明确其"创建 spy"而非"断言"的职责。
- **依据**: Issue #18 设计规范 4.2 节描述 `expectFallbackWarn(key)` 应为"验证 `console.warn` 调用"，而当前实现将其拆成了两个函数，命名与规范意图不符。

### 2. `getDepthCounters` 中多余的 `key` 变量与类型断言
- **位置**: `apps/phaser3/src/ui/utils.ts:224-237`
- **问题**: `const key = DEPTH_COUNTER_KEY as keyof typeof scene;` 这一行对 TypeScript 说了谎——`'__ui_depth_counters'` 并不是 `Phaser.Scene` 的合法 key。虽然后续通过 `as unknown` 绕过，但这行本身没有实际作用（下一行直接用了 `(scene as unknown as Record<string, unknown>)[DEPTH_COUNTER_KEY]`）。
- **修复建议**: 删除 `key` 变量，直接使用 `sceneRecord` 模式：
  ```typescript
  const sceneRecord = scene as unknown as Record<string, unknown>;
  if (!sceneRecord[DEPTH_COUNTER_KEY]) {
    sceneRecord[DEPTH_COUNTER_KEY] = new Map<UIDepthLayer, number>();
  }
  return sceneRecord[DEPTH_COUNTER_KEY] as Map<UIDepthLayer, number>;
  ```

### 3. 部分现有组件仍含魔法数字色值（Pre-existing，本次 PR 未引入）
- **位置**: `UIBadge.ts`、`UIToast.ts`、`UIDialog.ts`、`UIList.ts` 等
- **说明**: 这些组件中仍散落大量硬编码色值（如 `0x4caf50`、`0xff9800`、`0x000000`、`0x333333` 等）。**注意**：这些是 Phase 1 之前已存在的代码，本次 PR 仅替换了 Config 接口的继承方式，未触碰组件内部渲染逻辑。因此不视为本次 PR 的阻塞项，但应在 Phase 2 组件重做时按 `docs/design-docs/ui-style-guide/01-foundation.md` F-07 逐项清理。

## 风险与建议

1. **Phase 2 组件重做风险**：本次 PR 将 17 个组件的 Config 接口统一改为 `extends UIPositionConfig, Partial<UISizeConfig>`，这为 Phase 2 的接口重命名（如 `disabled` → `isDisabled`、`autoHide` → `isAutoHide`）奠定了基类，但也意味着 Phase 2 的改动面会很大。建议在 Phase 2 开始前，先基于当前 `index.ts` 导出列表梳理一份「组件接口变更对照表」，避免遗漏。

2. **`UIListItem` 类型混淆风险**：当前 `index.ts` 导出 `type UIListItem` 来自 `UIList.ts`（它是一个数据接口）。PRD-004 规划中将新增独立的 `UIListItem` 组件类。届时 `index.ts` 中需要区分 `type UIListItemData` 与 `UIListItem` 类，建议提前在 Phase 2 设计文档中明确命名，避免消费者代码因同名而报错。

3. **深度计数器全局回退**：`getNextDepth(layer)` 在不传 `scene` 时使用全局计数器。这在多 Scene 并发测试或 future 的多 Scene 同时渲染场景下可能导致 depth 冲突。建议在生产代码中**强制**传入 `scene` 参数，并将 `scene` 改为必填，或至少在 JSDoc 中标注"生产环境必须传入 scene"。

4. **Node modules 缺失导致 lint/type-check 未验证**：本次审查无法运行 `pnpm --filter phaser3 lint` 和 `type-check`。建议在合并前由开发 Agent 在本地补全依赖后执行一次全量检查，确认无 TypeScript 编译错误（尤其是 `import type` 语法与项目 `tsconfig.json` 的兼容性）。

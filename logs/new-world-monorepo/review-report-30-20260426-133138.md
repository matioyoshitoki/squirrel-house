## 阻塞问题（Blocking）

### 1. UIDialog 构造函数在 `super()` 之前通过箭头函数引用 `this`

- **位置**: `apps/phaser3/src/ui/UIDialog.ts:63-67`
- **问题**: `panelConfig` 对象字面量在 `super(scene, panelConfig)` 之前创建，其中包含箭头函数 `onOverlayClick: () => { this.cancel(); }`。箭头函数在定义时会词法捕获外围的 `this`，而在 `super()` 调用之前，`this` 绑定处于未初始化状态（TDZ）。这会导致：
  1. **TypeScript 编译错误**: `ts(17009) 'super' must be called before accessing 'this'`
  2. **运行时 ReferenceError**: 在严格模式下访问未初始化的 `this` 会抛出异常
- **影响**: UIDialog 组件完全无法实例化，任何使用 `new UIDialog()` 或 `UIDialog.show()` 的代码都会崩溃。
- **修复**:
  ```typescript
  // apps/phaser3/src/ui/UIDialog.ts
  constructor(scene: Scene, config: UIDialogConfig) {
    const camera = scene.cameras.main;
    const panelConfig = {
      x: config.x ?? camera.width / 2,
      y: config.y ?? camera.height / 2,
      width: config.width ?? 360,
      height: config.height ?? 220,
      title: config.title,
      showClose: false,
      showOverlay: true,
      overlayAlpha: 0.6,
      isModal: true,
      depth: config.depth ?? getNextDepth(UIDepthLayer.Dialog, scene),
      onClose: config.onClose,
      // 不在 super 前传递依赖 this 的回调
      onOverlayClick: undefined,
    };

    super(scene, panelConfig);

    // super() 之后手动绑定 overlay 点击事件
    if (config.closeOnOverlay && this.overlay) {
      this.overlay.removeListener('pointerdown');
      this.overlay.on('pointerdown', () => this.cancel());
    }

    // ...后续代码不变
  }
  ```
- **依据**: ECMAScript Class Fields & TDZ 规范；TypeScript `noImplicitAny`/`strict` 模式下编译器明确报错。

## 严重问题（Major）

### 2. UIButton `onClick` 在 yoyo tween `onComplete` 中触发，造成 ~100ms 交互延迟

- **位置**: `apps/phaser3/src/ui/UIButton.ts:116-132`
- **问题**: `pointerdown` 事件启动了一个 `scale: 0.95` + `yoyo: true` 的 tween（总时长 100ms），并在 `onComplete` 中调用 `this.config.onClick?.()`。这意味着用户点击按钮后，必须等待 100ms 的动画完成后才能触发业务逻辑。快速点击或需要即时响应的场景（如战斗、关闭弹窗）会明显感到迟滞。此外，`pointerup` 事件中仅恢复背景颜色，从不触发 `onClick`。
- **影响**: 所有使用 `UIButton` 的交互场景都存在非预期的点击延迟；快速连续点击时第二次点击可能被第一次的 tween 覆盖（虽然已有 `killTweensOf` 保护，但 onComplete 的时序仍然混乱）。
- **修复**:
  ```typescript
  // apps/phaser3/src/ui/UIButton.ts
  private setupInteraction(): void {
    const target = this.background as Phaser.GameObjects.Rectangle & Phaser.GameObjects.Image;
    target.setInteractive({ useHandCursor: true });

    // pointerdown: 仅处理视觉反馈（Pressed 态）
    target.on('pointerdown', () => {
      if (this.config.isDisabled) return;
      killTweensOf(this.scene, this);
      if (this.background instanceof Phaser.GameObjects.Rectangle && this.config.pressedColor !== -1) {
        this.background.setFillStyle(this.config.pressedColor, 1);
      }
      this.scene.tweens.add({
        targets: this,
        scale: 0.95,
        duration: UITweens.press.duration,
        ease: UITweens.press.ease,
        yoyo: true,
        // 移除 onComplete 中的 onClick 调用
      });
    });

    // pointerup: 恢复视觉并立即触发点击
    target.on('pointerup', () => {
      if (this.config.isDisabled) return;
      if (this.background instanceof Phaser.GameObjects.Rectangle) {
        this.background.setFillStyle(this.config.bgColor, this.originalFillAlpha);
      }
      this.config.onClick?.(); // ← 立即触发
    });

    // pointerupoutside: 取消 Pressed 态，不触发点击
    target.on('pointerupoutside', () => {
      if (this.config.isDisabled) return;
      if (this.background instanceof Phaser.GameObjects.Rectangle) {
        this.background.setFillStyle(this.config.bgColor, this.originalFillAlpha);
      }
      killTweensOf(this.scene, this);
      this.setScale(1);
    });

    // ...isDisabled 处理不变
  }
  ```
同步修改测试文件 `UIButton.test.ts`：
  ```typescript
  // 需要同时触发 pointerdown + pointerup 才能触发 onClick
  emitEvent(bg, 'pointerdown');
  emitEvent(bg, 'pointerup');
  expect(onClick).toHaveBeenCalled();
  ```
- **依据**: PRD-004 3.6 节组件状态机图：`[PRESSED] ──pointerup──▶ [HOVER]`，点击触发应在 `pointerup` 时完成；03-component.md 3.1 节 "Pressed 缩小...yoyo 恢复" 仅描述视觉动画，不意味着回调应在动画完成后触发。

### 3. SceneSwitchBar 误用 `emit()` 返回值作为权限判断依据

- **位置**: `apps/phaser3/src/city-builder/ui/SceneSwitchBar.ts:102-107`
- **问题**: `const canSwitch = this.emit('before-switch', sceneType);` 中，`Phaser.Events.EventEmitter.emit()` 的返回值是 `boolean`（表示该事件是否至少有一个监听器），**不是监听器的返回值**。因此：
  - 若没有 `before-switch` 监听器 → `emit` 返回 `false` → 错误地阻止切换
  - 若有监听器（即使监听器返回 `false` 表示不允许）→ `emit` 返回 `true` → 错误地允许切换
- **影响**: `before-switch` 事件机制完全失效，业务层无法通过事件监听阻止场景切换。
- **修复**:
  ```typescript
  // apps/phaser3/src/city-builder/ui/SceneSwitchBar.ts
  private switchScene(sceneType: SceneType): void {
    if (this.currentScene === sceneType) return;

    if (sceneType === SceneType.Battle) {
      const listeners = this.listeners('before-switch') as Array<(type: SceneType) => boolean>;
      const canSwitch = listeners.length === 0 || listeners.every(fn => fn(sceneType));
      if (!canSwitch) {
        this.segmentedControl.setSelectedIndex(this.currentScene === SceneType.City ? 0 : 1);
        return;
      }
    }

    this.currentScene = sceneType;
    this.emit('scene-switch', sceneType);
  }
  ```
- **依据**: Phaser 3 API 文档 `Phaser.Events.EventEmitter#emit` 返回类型为 `boolean`；代码逻辑意图（根据监听器返回值判断是否允许切换）与 API 实际行为不符。

### 4. UIPanel 和 UIDialog 单元测试未从 `designs/` 迁移到 `apps/phaser3/src/ui/__tests__/`

- **位置**: `designs/issue-19/src/ui/__tests__/UIPanel.test.ts`、`UIDialog.test.ts`
- **问题**: Issue #19 的设计资产目录中包含了 UIPanel 和 UIDialog 的完整单元测试（各 ~120 行），但 PR 的变更文件列表（`.doc-coverage.txt`）和实际 `apps/phaser3/src/ui/__tests__/` 目录中均缺少这两个文件。四个核心交付组件中，仅 UIAlert 和 UIToast 的测试被正确迁移。
- **影响**: 主代码库缺少对 UIPanel（show/hide 动画、遮罩、关闭按钮、内容区计算）和 UIDialog（继承行为、确认/取消回调时序、静态工厂方法）的自动化验证。后续重构极易引入回归缺陷。
- **修复**:
  1. 将 `designs/issue-19/src/ui/__tests__/UIPanel.test.ts` 和 `UIDialog.test.ts` 复制到 `apps/phaser3/src/ui/__tests__/`。
  2. **注意**：designs 中的测试使用了 `scene.native` API（来自 designs 自己的 setup.ts），而 `apps/phaser3/src/ui/__tests__/setup.ts` 中的 `UIMockScene` 是直接继承 `MockScene` 的，没有 `.native` 属性。迁移时需要将 `scene.native` 替换为 `scene`（因为 apps 中的 `UIMockScene` 本身就已经是 MockScene 的扩展）。
- **依据**: PRD-004 7.3 节测试要求："每个组件独立测试文件，覆盖初始化、交互状态、资源回退、动画、内存清理"；design-spec.md 9.1/9.2 验收标准明确要求 UIPanel/UIDialog 的 show/hide 动画和遮罩逻辑通过测试验证。

### 5. UIToast / UIAlert `show()` 在重复调用时导致 y 坐标累积漂移

- **位置**: 
  - `apps/phaser3/src/ui/UIToast.ts:104-106` 和 `show():222-235`
  - `apps/phaser3/src/ui/UIAlert.ts:59-61` 和 `show():119-132`
- **问题**: 构造函数中执行 `this.y += 20`，`show()` 的 tween 目标为 `this.y - 20`（即回到原始位置）。但如果 `show()` 被调用两次（例如手动调用，或 tween 被 kill 后重新 show），第二次的 tween 目标会变成 `原始位置 - 20`，导致 Toast/Alert 位置上移。UIToast 的 `show()` 在旧 API 兼容分支（`config.title || config.startTime`）下会自动调用，与新 API 的手动调用叠加时也可能导致问题。
- **影响**: Toast/Alert 在特定交互序列下位置错位；多个 Toast 的垂直堆叠逻辑（ToastManager）可能因此产生不可预期的间距。
- **修复**:
  ```typescript
  // apps/phaser3/src/ui/UIToast.ts（UIAlert 同理）
  export class UIToast extends Phaser.GameObjects.Container {
    private config: Required<UIToastConfig>;
    private originY: number; // 保存原始目标 Y 坐标

    constructor(scene: Scene, config: UIToastConfig) {
      // ...
      this.originY = this.y; // 在 y += 20 之前保存原始位置
      this.y += 20;
      // ...
    }

    show(): void {
      // ...
      killTweensOf(this.scene, this);
      this.scene.tweens.add({
        targets: this,
        alpha: 1,
        y: this.originY, // 使用保存的原始位置，而非 this.y - 20
        duration: UITweens.toastIn.duration,
        ease: UITweens.toastIn.ease,
      });
      // ...
    }
  }
  ```
- **依据**: design-spec.md 3.3/3.4 中动画标准描述为 "y: +20 → 0"（相对于目标位置），意味着无论 show() 被调用多少次，最终位置应稳定在同一目标点；当前实现违反幂等性原则。

## 轻微问题（Minor）

### 6. UIAlert `hide()` duration 硬编码，与 `UITweens` 常量不一致

- **位置**: `apps/phaser3/src/ui/UIAlert.ts:153`
- **问题**: `duration: 200` 直接硬编码，而项目已定义 `UITweens.toastOut.duration = 200`。
- **修复**: 将 `duration: 200` 改为 `duration: UITweens.toastOut.duration`。
- **依据**: PRD-004 3.4 节要求统一使用 `UITweens` 常量；agent-design-guide.md 3.1 节禁止魔法数字。

### 7. `apps/phaser3/src/ui/__tests__/setup.ts` 与 `designs/issue-19/src/ui/__tests__/setup.ts` API 不兼容

- **位置**: `apps/phaser3/src/ui/__tests__/setup.ts` vs `designs/issue-19/src/ui/__tests__/setup.ts`
- **问题**: 两个目录下的 `UIMockScene` 设计完全不同：
  - `apps/` 中：`class UIMockScene extends MockScene`
  - `designs/` 中：`class UIMockScene { get native(): MockScene }`（包装器模式）
这导致 designs 中的测试代码（如 `UIPanel.test.ts` 使用 `scene.native`）无法直接在 apps 的测试基础设施上运行。
- **修复**: 统一测试基础设施，或迁移 designs 测试时同步替换 `scene.native` 为 `scene`。
- **依据**: AGENTS.md 维护规则第 5 条："债务同步：发现代码与文档不一致时，更新对应文档或在 tech-debt-tracker 中追加条目"。两套测试基础设施并存是隐性技术债务。

## 风险与建议

1. **UIDialog 运行时崩溃风险（高）**：Blocking 问题导致 UIDialog 完全不可用。建议将此修复作为最高优先级，并在修复后补充一个 E2E 或集成测试，验证 `UIDialog.show()` 和 `UIDialog.alert()` 能正常弹出不抛异常。

2. **UIButton 交互延迟风险（中）**：100ms 的点击延迟在战斗、建造等高频操作场景下会显著降低用户体验。建议在修复后，通过 Playwright 截图验收或手动测试验证快速连续点击的响应性。

3. **测试基础设施碎片化风险（中）**：`apps/` 和 `designs/` 下的测试 helper API 不统一，长期会导致 designs 中的测试无法作为回归测试资产复用。建议后续由专门的测试 Agent 统一两套 setup.ts，或明确约定 designs 中的测试在迁移时必须适配 apps 的 API。

4. **SceneSwitchBar 业务逻辑失效风险（中）**：`before-switch` 事件无法正确阻止场景切换，可能导致玩家在编辑模式下误切到战斗场景。建议修复后，在 `SceneSwitchBar` 的测试或 city-builder 的 E2E 场景中覆盖此边界条件。

5. **PR 范围外组件的规范漂移（低）**：本次 PR 仅重做了四个容器组件，但 `UIButton.ts`、`UIBadge.ts`、`UISegmentedControl.ts` 等也在 diff 中被修改。虽然改动较小（如移除 `export default`、增加 `colorToHex`），但分散了 review 注意力。建议未来将纯重构（如统一导出方式、统一颜色调用）单独提一个 "代码清理" PR，降低单个 PR 的认知负荷。

## 阻塞问题（Blocking）

### 1. CreateOrGetConversation 存在并发竞态，会导致唯一索引冲突报错
- **位置**: `internal/data/wapi/im.go:31-61`
- **问题**: 该函数先 Query 再 Create，但两步不在同一个事务中。当两个并发请求同时为新参与者对创建会话时，两者都可能查询到记录不存在，然后同时执行 `INSERT`，导致其中一个请求触发数据库唯一索引冲突（`UNIQUE KEY uk_conversation (participant_a, participant_b)`），并将原始 DB 错误直接返回给调用方，破坏了"获取或创建"的语义契约。
- **影响**: 高并发场景下（IM 消息发送是典型的并发场景），用户有概率收到内部数据库错误，消息发送失败。
- **修复**: 
  ```go
  // 方案 A：使用事务包裹查询+创建，并在创建失败时重试查询
  func (r *imRepo) CreateOrGetConversation(ctx context.Context, a, b string) (*repo.Conversation, error) {
      p1, p2 := a, b
      if a > b { p1, p2 = b, a }

      // 先尝试查询
      conv, err := r.worldDB.Conversation.Query()...
      if err == nil {
          return r.toRepoConversation(conv), nil
      }
      if !ent.IsNotFound(err) {
          return nil, err
      }

      // 创建失败时，如果是唯一冲突，则重新查询
      now := time.Now().UnixMilli()
      convID := fmt.Sprintf("conv_%d_%s_%s", now, p1, p2)
      newConv, err := r.worldDB.Conversation.Create()...Save(ctx)
      if err != nil {
          if ent.IsConstraintError(err) {
              conv, qerr := r.worldDB.Conversation.Query()...
              if qerr == nil {
                  return r.toRepoConversation(conv), nil
              }
          }
          return nil, err
      }
      return r.toRepoConversation(newConv), nil
  }
  ```
- **依据**: `docs/design-docs/ent.md` 要求 Schema 变更需同步迁移脚本；`internal/data/world/ent/schema/conversation.go` 定义了唯一索引 `index.Fields("participant_a", "participant_b").Unique()`，代码必须正确处理该约束。

### 2. MarkAsRead 存在部分提交风险，失败后重试会永久失败
- **位置**: `internal/biz/wapi/im.go:273-324`
- **问题**: `MarkAsRead` 的执行顺序是：(1) 批量写入已读回执 → (2) 清空未读数 → (3) 查询消息获取发送者 → (4) 推送 WS 已读回执。如果第 (3) 步失败（如数据库连接超时），API 返回错误，但第 (1)(2) 步已经提交。客户端重试时，第 (1) 步的批量插入会因为唯一索引 `(message_id, reader_id)` 冲突而永久失败，导致用户再也无法成功调用该接口标记这些消息为已读。
- **影响**: 在偶发的数据库抖动后，特定用户针对特定消息的已读标记功能永久不可用。
- **修复**: 将读取操作前置，或把写操作纳入事务。推荐方案：
  ```go
  func (uc *IMUseCase) MarkAsRead(ctx context.Context, uid int64, req *apipb.MarkAsReadRequest) (*apipb.MarkAsReadResponse, error) {
      if len(req.MessageIds) == 0 {
          return &apipb.MarkAsReadResponse{ReadCount: 0}, nil
      }

      // 1. 先查询消息（无副作用，失败可安全重试）
      messages, err := uc.imRepo.GetMessagesByIDs(ctx, req.MessageIds)
      if err != nil {
          return nil, err
      }

      uidStr := fmt.Sprintf("d", uid)
      now := uc.nowMs()

      // 2. 写操作纳入事务
      tx, err := uc.txHelper.BeginWorldTx(ctx)
      if err != nil {
          return nil, err
      }
      defer func() { _ = tx.Rollback() }()

      receipts := make([]*repo.ReadReceipt, 0, len(req.MessageIds))
      for _, msgID := range req.MessageIds {
          receipts = append(receipts, &repo.ReadReceipt{...})
      }
      // 需要为 BatchCreateReadReceipts 和 ClearUnreadCount 增加 Tx 版本
      if err := uc.imRepo.BatchCreateReadReceiptsTx(ctx, tx, receipts); err != nil {
          return nil, err
      }
      if err := uc.imRepo.ClearUnreadCountTx(ctx, tx, req.ConversationId, uidStr); err != nil {
          return nil, err
      }
      if err := tx.Commit(); err != nil {
          return nil, err
      }

      // 3. 推送（允许失败，仅记日志）
      ...
  }
  ```
- **依据**: `docs/architecture/layers.md` 第 12.1 节"何时开启事务"明确说明"多表更新需要事务"；`docs/design-docs/transactions.md` 要求事务在 Biz 层开启。

### 4. GetMessageHistory 中接收消息的 is_read 字段逻辑错误

- **影响**: API 契约语义失真，客户端无法准确展示未读状态。
- **修复**:
  ```go
  } else {
      // 对方发的消息，检查当前用户是否有已读回执

      for _, r := range readers {

              break
          }
      }
  ```

- **依据**: `api/new-world-api/v1/im.proto` 中 `IMMessageContent.is_read` 字段的存在意义就是标识阅读状态；当前实现使其对接收方消息失去意义。

## 风险与建议

1. **并发风险（高）**: `CreateOrGetConversation` 的竞态条件在 IM 高频调用场景下会被放大。建议在 IM 模块上线前必须修复，或至少对唯一约束错误做优雅降级（重试查询）。

2. **事务边界风险（高）**: `MarkAsRead` 的部分提交问题在分布式环境下（网络抖动、DB 超时）可能导致用户功能永久性损坏。建议将写操作纳入事务，并将无副作用的读取操作前置。

3. **前端规范债务（中）**: `SceneSwitchBar` 的魔法数字和 `export default` 问题属于已有的规范债务。如果本次 PR 的目的只是修复 pre-push hook 编译错误，则不应引入新的规范违规；如果属于 PRD-004 实施的一部分，则必须按规范执行。

4. **API 契约完整性（中）**: `is_read` 字段对接收方消息的逻辑需要修正，否则客户端无法依赖该字段做未读红点展示。

5. **OpenAPI 缺失（低）**: 虽然项目内部通过 proto 生成 HTTP 网关，但 `openapi.yaml` 作为外部文档入口，应及时补充。

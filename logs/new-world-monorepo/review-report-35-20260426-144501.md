## 严重问题（Major）

### 1. GetMessageHistory 缺少会话参与者权限校验

- **位置**: `internal/biz/wapi/im.go:191`
- **问题**: `GetMessageHistory` 业务逻辑未验证请求用户是否为该 `conversation_id` 的参与者。任何已登录用户只要猜测到 `conversation_id`，即可读取任意会话的消息历史。
- **影响**: 用户隐私泄露 —— 恶意用户可读取他人私聊内容。
- **修复**: 在查询消息历史前，先查询会话并校验 `uid` 是否等于 `participant_a` 或 `participant_b`：
  ```go
  // 在 GetMessageHistory 开头增加校验
  conv, err := uc.imRepo.GetConversationByID(ctx, req.ConversationId)
  if err != nil {
      return nil, err
  }
  uidStr := fmt.Sprintf("%d", uid)
  if conv.ParticipantA != uidStr && conv.ParticipantB != uidStr {
      return nil, status.Error(codes.PermissionDenied, "not a participant")
  }
  ```
- **依据**: 根据 `docs/architecture/layers.md` 第 10.3 节错误处理规范，权限不足应返回 `codes.PermissionDenied`；IM 系统基本安全边界要求参与者隔离。

### 2. MarkAsRead 缺少会话参与者权限校验 + 未验证消息归属

- **位置**: `internal/biz/wapi/im.go:279-337`
- **问题**: 
  1. `MarkAsRead` 未验证请求用户是否为该 `conversation_id` 的参与者，导致任意用户可清除任意会话的未读数。
  2. `BatchCreateReadReceiptsTx` 为所有传入的 `message_ids` 创建回执，但未验证这些消息是否属于 `req.ConversationId`。用户可传入其他会话的消息 ID，导致数据不一致（会话 A 的未读被清零，但回执记录在会话 B 的消息上）。
- **影响**: 隐私边界破坏 + 数据不一致。
- **修复**: 
  ```go
  // 1. 校验参与者身份（同 Major-1）
  // 2. 过滤消息：仅保留属于当前 conversation_id 的消息
  validMessageIDs := make([]string, 0, len(messages))
  for _, m := range messages {
      if m.ConversationID == req.ConversationId {
          validMessageIDs = append(validMessageIDs, m.MessageID)
      }
  }
  // 3. 仅对 validMessageIDs 创建回执和计算 readCount
  ```
- **依据**: IM 业务基本一致性约束；`docs/architecture/layers.md` 第 9.1 节数据流转规范要求 Biz 层负责业务规则校验。

## 轻微问题（Minor）

### 3. UIBadge.setBadgeSize 命名与 PRD-004 不一致

- **位置**: `apps/phaser3/src/ui/UIBadge.ts:128`
- **问题**: PRD-004 明确规定接口为 `setSize(size: 'small' | 'medium' | 'large'): void`，但实现使用了 `setBadgeSize(...)`。
- **影响**: 接口一致性偏差，开发者按 PRD 调用时会遇到类型错误。
- **修复**: 将方法名从 `setBadgeSize` 改为 `setSize`，并同步更新测试文件 `UIBadge.test.ts:65,70`。
- **依据**: `prd/PRD-004-ui-component-library-redesign.md` 第 5.16 节 UIBadge 接口设计。

### 4. UISegmentedControl 扩展字段未在 PRD-004 中定义

- **位置**: `apps/phaser3/src/ui/UISegmentedControl.ts:19-35`
- **问题**: `UISegmentedControlConfig` 中增加了 `selectedKey`、`width`、`height`、`onSelect` 等字段，但 PRD-004 的 5.13 节接口设计中未包含这些字段。PRD 明确将本 PRD 的交付范围定义为 Layer 1 重做，且强调 "违反即视为验收不通过" 的命名规范。
- **影响**: 接口契约与 PRD 不一致，后续维护者难以判断标准来源。
- **修复**: 
  - 若业务确实需要 `selectedKey` / `width` / `height` / `onSelect`，应在 PRD-004 中补充变更记录；
  - 或移除 `selectedKey`（PRD 仅要求 `selectedIndex`），将 `width`/`height` 重命名为符合 PRD 的 `itemWidth`/`itemHeight`，移除 `onSelect`（PRD 仅要求 `onChange`）。
- **依据**: `prd/PRD-004-ui-component-library-redesign.md` 第 3.2 节命名规范（强制性）及 5.13 节接口设计。

### 5. SendMessage 忽略 fmt.Sscanf 错误

- **位置**: `internal/biz/wapi/im.go:121-122`
- **问题**: 
  ```go
  var ruid int64
  fmt.Sscanf(req.ReceiverId, "%d", &ruid)
  ```
未处理 `Sscanf` 的返回值。若 `ReceiverId` 格式异常（如包含非数字字符），`ruid` 为 0，WebSocket 推送被静默跳过，调用方无从知晓推送失败。
- **影响**: 调试困难，异常输入导致静默行为。
- **修复**: 
  ```go
  var ruid int64
  if _, err := fmt.Sscanf(req.ReceiverId, "%d", &ruid); err != nil {
      uc.log.Warnw("invalid receiver_id format", "receiver_id", req.ReceiverId, "error", err)
  } else if ruid > 0 {
      if err := uc.pusher.PushMessage(ctx, ruid, resp); err != nil {
          uc.log.Warnw("push message failed", "error", err, "receiver", req.ReceiverId)
      }
  }
  ```
- **依据**: `docs/QUALITY_SCORE.md` 及 Go 最佳实践 —— 不应忽略 error 返回值。

### 6. GetConversationList / GetMessageHistory 存在 N+1 查询

- **位置**: `internal/biz/wapi/im.go:163-169`（会话列表逐条查用户）、`internal/biz/wapi/im.go:223-227`（消息历史逐条查用户）
- **问题**: 在循环中逐个调用 `uc.userRepo.FindOne`，导致 N+1 查询问题。每页 20 条数据会产生 20 次用户查询。
- **影响**: 接口延迟随数据量线性增长，P99 可能超标。
- **修复**: 在 `repo.UserRepo` 中增加 `BatchFindByIDs(ctx, []int64) (map[int64]*User, error)` 方法，Biz 层先收集所有 UID，批量查询后建立映射表。
- **依据**: `prd/PRD-003-leaderboard-system.md` 第 5.1 节性能要求（榜单查询 < 50ms P99）体现了项目对查询性能的关注；IM 消息历史查询同样应符合性能基线。

### 7. SendMessage 缺少基础业务校验

- **位置**: `internal/biz/wapi/im.go:58-130`
- **问题**: 未校验以下内容：
  - `req.ReceiverId` 是否为空
  - `senderID` 是否等于 `req.ReceiverId`（自己给自己发消息是否允许）
  - `req.Content` 是否为空或超长
  - `req.MessageType` 是否为有效枚举值
- **影响**: 脏数据入库、潜在滥用风险。
- **修复**: 在函数开头增加参数校验：
  ```go
  if req.ReceiverId == "" || req.ReceiverId == senderStr {
      return nil, status.Error(codes.InvalidArgument, "invalid receiver")
  }
  if strings.TrimSpace(req.Content) == "" {
      return nil, status.Error(codes.InvalidArgument, "content is empty")
  }
  ```
- **依据**: `docs/architecture/layers.md` 第 10.3 节标准错误码映射，`biz.ErrInvalidArgument` 应对应 `codes.InvalidArgument`。

## 风险与建议

### 1. 安全边界风险（高）
IM V1 的会话隔离机制缺失是最大风险。建议在合并前务必修复 **Major-1** 和 **Major-2**，否则上线后存在用户隐私泄露和数据不一致风险。

### 2. 性能风险（中）
`GetConversationList` 和 `GetMessageHistory` 的 N+1 查询（Minor-6）在上线初期用户量小无感知，但随着会话和消息量增长，P99 会明显劣化。建议在后续迭代中增加批量查询接口。

### 3. UI 规范漂移风险（低）
PR #35 同时修改了后端 IM 和前端 UI 组件。UI 组件（UIBadge、UISegmentedControl）的接口与 PRD-004 存在偏差，若不及时对齐，后续业务模块 Scene 开发时会积累接口不一致的技术债务。建议：
- 要么同步更新 PRD-004 记录实际接口；
- 要么调整代码严格对齐 PRD。

### 4. 前端 E2E / 视觉验收
PRD-004 第 7.5 节要求前端 PR 必须通过视觉验收（截图对比）。本 PR 未提供 `designs/issue-5/screenshots/` 或等效验收产物。由于 UI 变更量较小（2 个组件 + 1 个业务组件适配），建议在合并前补充至少以下状态的截图基线：
- `UIBadge-normal.png` / `UIBadge-hover.png` / `UIBadge-disabled.png`
- `UISegmentedControl-normal.png` / `UISegmentedControl-hover.png`

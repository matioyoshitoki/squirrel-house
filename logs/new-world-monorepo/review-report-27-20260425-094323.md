## 严重问题（Major）

### 1. `createFallbackRect` 必需参数位于可选参数之后，API 可用性受损
- **位置**: `designs/issue-27/src/ui/utils.ts:145-151`
- **问题**: 函数签名将必需参数 `textureKey: string` 置于带默认值的可选参数 `fillColor`、`strokeColor` 之后：
  ```typescript
  export function createFallbackRect(
    scene: Scene,
    width: number,
    height: number,
    fillColor: number = UITheme.inkBlack,
    strokeColor: number = UITheme.goldBrown,
    textureKey: string   // ← 必需，但位于可选参数之后
  ): Phaser.GameObjects.Rectangle
  ```
这导致调用者无法直接写 `createFallbackRect(scene, 100, 100, 'missing')`，而必须显式传入前两个可选参数：
  ```typescript
  createFallbackRect(scene, 100, 100, undefined, undefined, 'missing');
  ```
与 PRD-004 强调的“统一命名/行为/回退”和“不需要记每个组件的特殊参数名”目标相冲突。
- **影响**: 所有依赖 `createFallbackRect` 的组件在纹理缺失时的回退代码都将变得冗长且易错，增加 Phase 2/3 组件开发的认知负担。
- **修复**: 将 `textureKey` 提前到 `width`/`height` 之后，或改为对象配置参数：
  ```typescript
  export function createFallbackRect(
    scene: Scene,
    width: number,
    height: number,
    textureKey: string,
    fillColor: number = UITheme.inkBlack,
    strokeColor: number = UITheme.goldBrown,
  ): Phaser.GameObjects.Rectangle
  ```
同时同步更新 `design-spec.md` 3.2 节的签名表格。
- **依据**: PRD-004 3.2 命名规范要求接口简洁统一；`docs/frontend/agent-design-guide.md` 3.2 要求组件使用配置参数而非复杂位置参数。

## 轻微问题（Minor）

### 2. `UITheme` JSDoc 声明为“9 色”但实际包含 10 个颜色条目
- **位置**: `designs/issue-18/src/ui/utils.ts:15-37`
- **问题**: JSDoc 注释为 `/** 国风水墨主题色板（9 色） */`，但实际对象包含 `inkBlack`、`paperYellow`、`cinnabarRed`、`stoneCyan`、`indigoBlue`、`goldBrown`、`lightInk`、`white`、`inkText`、`warning` 共 10 个条目。`warning: 0xffaa00` 不在 `docs/design-docs/ui-style-guide/01-foundation.md` 1.1 的 9 色基础色板表中（该颜色仅在 Component 层的 UIAlert 规范中出现）。
- **影响**: 误导后续开发 Agent 对色板范围的理解，可能导致硬编码 `0xffaa00` 或错误地认为 `warning` 属于 Foundation 9 色。
- **修复**: 将 JSDoc 改为 `（10 色）`，或在 `design-spec.md` 中增加说明解释 `warning` 被纳入 `UITheme` 的原因；若坚持 9 色，则将 `warning` 移出 `UITheme` 作为独立常量。
- **依据**: `docs/design-docs/ui-style-guide/01-foundation.md` 1.1 明确列出 9 个 `UITheme` 颜色。

### 3. `design-spec.md` 中 UIPanel 深度示例值错误
- **位置**: `designs/issue-18/design-spec.md:224`
- **问题**: 6.2 节“深度分配状态”表格中，示例行为写道：
  > 首次创建 | `new UIPanel(...)` | `depth = UIDepthLayer.Dialog`（3000）

但普通 `UIPanel`（非弹窗模式）应使用 `UIDepthLayer.Panel`（2000）。仅当 UIPanel 以弹窗模式使用时才应使用 3000（PRD-004 3.5）。
- **影响**: 开发 Agent 在 Phase 2 实现 `UIPanel` 时可能错误地将默认 depth 设为 3000，导致普通面板与弹窗层混淆。
- **修复**: 将示例改为 `UIDepthLayer.Panel`（2000），并补充一行说明弹窗模式下可覆盖为 `Dialog`（3000）。
- **依据**: PRD-004 3.5 深度规范表；`types.ts` 中 `UIDepthLayer.Panel = 2000`。

### 4. `restoreMockTextures` 未从 `patchedScenes` 中移除 Scene，导致 `cleanup()` 重复处理
- **位置**: `designs/issue-18/src/ui/__tests__/setup.ts:217-229`
- **问题**: `restoreMockTextures(scene)` 恢复了 `textures.exists` 并清除了 `__mockTextures`，但没有将该 `scene` 从模块级 `patchedScenes` Set 中移除。后续 `cleanup()` 会再次遍历该 Scene 并尝试恢复，虽然由于 `__originalTexturesExists` 已被删除而不会产生实际错误，但属于状态泄漏和重复操作。
- **影响**: 测试间状态隔离不彻底，长期可能埋下难以调试的测试顺序依赖问题。
- **修复**: 在 `restoreMockTextures` 末尾添加 `patchedScenes.delete(scene);`。
- **依据**: `design-spec.md` 4.3 约束规则 2 要求“每个 helper 必须处理事件监听器清理，防止测试间状态泄漏”。

### 5. 深度层级枚举与现有 `01-foundation.md` 存在粒度差异
- **位置**: `designs/issue-18/src/ui/types.ts:55-62`
- **问题**: `UIDepthLayer` 定义了 6 层（Scene=0, UI=1000, Panel=2000, Dialog=3000, Overlay=4000, Toast=5000），但 `docs/design-docs/ui-style-guide/01-foundation.md` 1.4 仅定义 4 层（0~999, 1000, 3000, 5000+），且将“弹窗/面板”合并为 3000，“模态遮罩/Toast”合并为 5000+。新枚举更细粒度（符合 PRD-004），但与当前生效的 Foundation 标准不一致。
- **影响**: 当前正在维护 `apps/phaser3/src/ui/` 的开发者可能因参考旧标准而产生混淆。
- **修复**: 在 `design-spec.md` 的“重要冲突报告”章节或 6.2 节中显式增加一条说明，指出本设计资产遵循 PRD-004 的 6 层深度模型，与旧版 `01-foundation.md` 的差异将在 Phase 4 文档同步时统一更新。无需修改代码。
- **依据**: `docs/design-docs/ui-style-guide/01-foundation.md` 1.4；PRD-004 3.5。

### 6. `simulateClick` 未传递 Pointer 事件数据
- **位置**: `designs/issue-18/src/ui/__tests__/setup.ts:138-141`
- **问题**: `simulateClick` 仅发射事件名：
  ```typescript
  target.emit('pointerdown');
  target.emit('pointerup');
  ```
真实 Phaser3 的 pointer 事件会传递 `Pointer` 对象作为参数。部分组件的事件处理逻辑（如判断点击坐标、区分左右键）可能依赖该对象。
- **影响**: 对依赖 pointer 数据的组件进行单元测试时，`simulateClick` 可能无法覆盖完整交互路径，测试可能通过但生产环境行为不同。
- **修复**: 为 `simulateClick` 增加可选的 `pointerData` 参数并透传：
  ```typescript
  export function simulateClick(
    target: { emit: (event: string, ...args: unknown[]) => unknown },
    pointerData?: Record<string, unknown>
  ): void {
    target.emit('pointerdown', pointerData);
    target.emit('pointerup', pointerData);
  }
  ```
- **依据**: Phaser3 官方输入事件契约；PRD-004 3.6 状态机涉及 `pointerdown`/`pointerup` 的完整语义。

## 风险与建议

1. **迁移路径风险**: `designs/issue-18/src/ui/__tests__/setup.ts` 中硬编码了 `../../../../../apps/phaser3/src/test-setup` 导入路径。开发 Agent 在迁移时容易遗漏替换为 `../../test-setup`。建议在 `design-spec.md` 的迁移指南中用 **粗体或代码块高亮** 该替换操作，或在 `setup.ts` 顶部增加更醒目的 `// TODO(MIGRATION)` 标记。

2. **Phase 2 衔接建议**: `index.ts` 中所有 22 个组件导出均为注释状态。建议在 `design-spec.md` 中明确 Phase 2 的组件实现优先级顺序（PRD-004 建议 UIPanel → UIButton → UIText → UIList → UIDialog → 其余），并规定每实现一个组件即取消对应注释，避免最终一次性大规模解注释带来的合并冲突。

3. **MockScene 类型兼容性**: `getNextDepth` 接受 `Phaser.Scene`，而现有测试使用 `MockScene`（非 Phaser.Scene 子类）。在 Phase 2 组件单元测试中，若组件内部调用 `getNextDepth`，TypeScript 可能报类型错误。建议在 `setup.ts` 中为 `UIMockScene` 增加 `.native` 的类型断言说明，或在 `getNextDepth` 的 JSDoc 中注明测试场景下可传入任意对象。

4. **样式拆分建议**: PRD-004 5.1 目录结构规划了 `styles/ui-colors.ts`，但本设计将 `UITheme` 保留在 `utils.ts` 中。若后续 tree-shaking 成为瓶颈，可在 Phase 4 再拆分，但建议在 `design-spec.md` 中增加一条设计决策记录（ADR）说明保留原因。

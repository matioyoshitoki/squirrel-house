## 严重问题（Major）

### 1. 纹理缺失回退未统一使用 `createFallbackRect()`，且未打印 war

- **位置**: 
  - `apps/phaser3/src/ui/UIButton.ts:57-62`
  - `apps/phaser3/src/ui/UIPanel.ts:52-57`
  - `apps/phaser3/src/ui/UIIcon.ts:47-54`
  - `apps/phaser3/src/ui/UIAvatar.ts:62-65`
- **问题**: 本次 PR 在 `utils.ts` 中新引入了 `createFallbackRect()`，其职责就是统一处理纹理缺失回退并打印 `console.warn(\`[UI] fallback: ${key}\`)`。但上述组件在传入的 `texture` 参数存在但缺失时，仍然使用旧的回退方式（`createInkBackground` 或直接 `add.rectangle`），**不打印 war
**。
- **影响**: 
  - 开发阶段无法通过控制台快速定位缺失的纹理资源。
  - 与 PRD-004 3.3 "资源回退统一标准" 及设计资产 3.3 "纹理缺失时必须打印 `console.warn`" 的强制契约不一致。
- **修复**: 
  - 在 `UIButton`、`UIPanel` 的 `createBackground()` 中，当 `texture && !hasTexture(...)` 时，使用 `createFallbackRect(this.scene, width, height, texture)` 替代 `createInkBackground`。
  - 在 `UIIcon`、`UIAvatar` 的纹理缺失分支中，使用 `createFallbackRect` 替代直接 `add.rectangle`。
- **依据**: 
  - PRD-004 第 3.3 节："所有依赖纹理的组件必须执行以下回退契约：... 告警：`console.warn(\`[UI] fallback: ${textureKey}\`)`"
  - 设计资产 `designs/issue-18/design-spec.md` 第 3.3 节："纹理缺失时必须打印 `console.warn('[UI] fallback: ${textureKey}')` — PRD-004 强制要求"

## 轻微问题（Minor）

### 3. `createInkBackground` alpha 与组件恢复逻辑不一致
- **位置**: 
  - `apps/phaser3/src/ui/UIButton.ts:99`
  - `apps/phaser3/src/ui/UIIconButton.ts:124`
- **问题**: 本次 PR 将 `createInkBackground` 的默认 `fillAlpha` 从 `0.95` 改为 `0.9`，但 `UIButton` 和 `UIIconButton` 在 `pointerout` 恢复背景时仍硬编码 `setFillStyle(this.config.bgColor, 0.95)`，导致 Hover 后 alpha 不一致。
- **影响**: 视觉闪烁，Hover 离开后背景透明度从 0.9 变成 0.95。
- **修复**: 将 `0.95` 改为 `0.9`，或读取 `this.background.fillAlpha` 进行恢复。
- **依据**: PR diff 中 `utils.ts` 的 `createInkBackground` 已将 alpha 调整为 `0.9`

### 4. 布尔参数命名未按规范统一为 `isXxx`
- **位置**:
  - `apps/phaser3/src/ui/UIButton.ts:19`：`disabled`
  - `apps/phaser3/src/ui/UIIconButton.ts:20`：`disabled`
  - `apps/phaser3/src/ui/UIAlert.ts:18`：`autoHide`
  - `apps/phaser3/src/ui/UIIcon.ts:16`：`interactive`
  - `apps/phaser3/src/ui/UIAvatar.ts:18`：`interactive`
- **问题**: PRD-004 3.2 强制要求所有布尔参数使用 `isXxx` 或 `showXxx` 前缀。本次 PR 修改了这些组件的 Config 接口（让它们继承新基类），但未同步调整布尔字段命名。
- **影响**: 接口命名不一致，开发者需要记忆不同组件的特殊参数名。
- **修复**: 
  - `disabled` → `isDisabled`
  - `autoHide` → `isAutoHide`
  - `interactive` → `isInteractive`
  - 同步更新组件内部引用和默认值处理。
- **依据**: PRD-004 3.2 命名规范（强制性）："布尔开关：`isXxx` 或 `showXxx`"

### 5. `UIAlert` 缺少 depth 设置
- **位置**: `apps/phaser3/src/ui/UIAlert.ts`
- **问题**: `UIAlert` 未调用 `setDepth()`，默认 depth 为 `0`，会被普通 UI（depth 1000）、面板（depth 2000+）等遮挡。
- **影响**: UIAlert 在实际场景中可能被其他 UI 元素覆盖，玩家看不到提示。
- **修复**: 在构造函数中设置 `this.setDepth(UIDepthLayer.Toast)`（需从 `types.ts` 导入 `UIDepthLayer`）。
- **依据**: 
  - PRD-004 3.5 深度规范：Toast / Alert / Tooltip = 5000+
  - `docs/design-docs/ui-style-guide/01-foundation.md` 1.4：模态遮罩 / Toast = `5000+`

### 6. `UIToast` 退场动画时长与规范不符
- **位置**: `apps/phaser3/src/ui/UIToast.ts:251-261`
- **问题**: 退场动画 `duration: 300`，但 PRD-004 3.4 规定 Toast 退场应为 `200ms`。
- **影响**: 动画拖沓，影响操作响应。
- **修复**: 将 `duration: 300` 改为 `duration: 200`，并使用 `UITweens.fade` 配置。
- **依据**: PRD-004 3.4 动画统一标准："Toast 退场 | 200ms | Power2 | y: 0 → -10, alpha: 1 → 0"

### 7. `UIAlert` 左侧类型色条宽度不符
- **位置**: `apps/phaser3/src/ui/UIAlert.ts:77`
- **问题**: 左侧色条宽度为 `6`，但 PRD-004 5.15 要求"增加左侧 4px 色条作为类型标识"。
- **影响**: 视觉规范偏差。
- **修复**: 将宽度从 `6` 改为 `4`，并调整位置计算。
- **依据**: PRD-004 5.15："增加左侧 4px 色条作为类型标识"

### 8. 组件级单元测试缺失
- **位置**: `apps/phaser3/src/ui/__tests__/`
- **问题**: 目录下仅有 `setup.ts` 和 `utils.test.ts`，没有任何组件的单元测试。PRD-004 1.2 成功指标要求单元测试覆盖率 ≥ 80%，5.1 目录结构要求每个组件都有对应测试文件。
- **影响**: 无法自动化验证组件行为，后续重构缺乏回归保护。
- **修复**: 为核心交互组件（如 `UIButton`、`UIPanel`、`UIDialog`）补充基础测试，覆盖 show/hide、交互状态、资源回退路径。
- **依据**: PRD-004 1.2 成功指标："单元测试覆盖率 ~60% → ≥ 80%"；5.1 目录结构列出 22 个测试文件。

### 9. `ToastManager` 未实现队列机制
- **位置**: `apps/phaser3/src/ui/UIToast.ts:314-377`
- **问题**: `ToastManager` 直接将新 Toast 推入数组，没有队列上限和排队逻辑。PRD-004 5.11 要求"队列满时（最多 3 个），最早的 Toast 提前 dismiss"。
- **影响**: 大量 Toast 同时触发时会堆叠溢出屏幕。
- **修复**: 在 `show()` 方法中增加队列长度检查，超过 3 个时调用 `this.toasts[0].dismiss()` 或 `markAsComplete()`。
- **依据**: PRD-004 5.11："`ToastManager` 支持队列，避免多个 Toast 重叠；队列满时（最多 3 个），最早的 Toast 提前 dismiss"

### 10. `UIList` 仍使用 `UIListItem` interface 而非独立组件
- **位置**: `apps/phaser3/src/ui/UIList.ts:11-16`
- **问题**: PRD-004 5.7 要求从 `UIList` 中拆出独立的 `UIListItem` class 组件，但当前 `UIListItem` 仍是一个数据 interface。
- **影响**: `UIList` 的 item 渲染与数据耦合，无法独立复用列表项。
- **修复**: 将 `UIListItem` 拆分为独立的 `UIListItem.ts` 文件，作为 `Phaser.GameObjects.Container` 子类。
- **依据**: PRD-004 5.7 变更要点："从 UIList 中拆出独立的 `UIListItem` 组件"

## 风险与建议

1. **渐进式重做的边界风险**：PRD-004 是一个大型重做 PRD，当前 PR 只覆盖了 Phase 1 基础设施。建议在后续 PR 中明确划分 Phase 2（Layer 1 组件完整重做）和 Phase 3（Layer 2 新增组件）的范围，避免一次 PR 过大或关键要求被遗漏。

2. **现有组件的 magic number 清理**：`UIToast.ts` 和 `UIBadge.ts` 中的魔法数字色值最多，建议在下一次组件重做 PR 中优先清理，并考虑在 `UITheme` 中新增业务语义色（如 `toastBuild`、`toastUpgrade`、`badgeSuccess`、`badgeRare` 等）。

3. **测试基础设施的利用**：本次 PR 提供的 `UIMockScene`、`simulateClick`、`assertFallbackWarn` 等 helper 质量很好，建议后续组件重做 PR 立即利用这些设施补齐组件测试。

4. **Depth 统一治理**：`UIAlert`、`UIToast`、`UIDialog` 的 depth 设置散落在各组件中，建议统一使用 `getNextDepth(UIDepthLayer.Xxx)` 替代硬编码，确保同层级组件后创建时自动递增。

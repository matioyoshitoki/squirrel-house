## 严重问题（Major）

### 1. `__tests__/setup.ts` 在 ESM 环境下使用 `require()`，存在运行时兼容性风险
- **位置**: `designs/issue-18/src/ui/__tests__/setup.ts:15`, `designs/issue-18/src/ui/__tests__/setup.ts:41`, `designs/issue-18/src/ui/__tests__/setup.ts:276`
- **问题**: 文件 3 处使用 `require('../../../../../../apps/phaser3/src/test-setup')` 导入 `MockScene` 与 `cleanup`。`apps/phaser3/package.json` 已声明 `"type": "module"`，所有 `.ts` 文件在 Vite/Vitest 环境下按 ESM 处理。`require()` 在严格 ESM 中会被抛出 `ReferenceError: require is not defined`。虽然 Vitest 内部有 CJS 兼容层，但这属于未定义行为，可能在升级构建工具后失效。
- **影响**: 当该文件按迁移指南复制到 `apps/phaser3/src/ui/__tests__/setup.ts` 后，测试可能无法启动。
- **修复**: 将 3 处 `require(...)` 改为顶层 ES Module `import`：
  ```ts
  // 迁移前（当前代码）
  export { MockScene, emitEvent } from '../../../../../../apps/phaser3/src/test-setup';
  // 已在第 15 行正确使用 ESM import，但第 41 行和第 276 行仍使用 require
  ```
具体修改：
  - 第 41 行：删除 `const { MockScene } = require(...)`，改为在构造函数外通过已导入的 `MockScene` 使用（或直接复用顶层的 `MockScene` import）。
  - 第 276 行：删除 `const { cleanup: baseCleanup } = require(...)`，改为在文件顶部 `import { cleanup as baseCleanup } from '../../../../../../apps/phaser3/src/test-setup'`。
- **依据**: `apps/phaser3/package.json` 第 6 行 `"type": "module"`；`docs/frontend/agent-design-guide.md` 3.1 节要求代码组织符合项目模块规范。

### 2. `simulateSceneShutdown` 与 `UIMockScene` 不兼容，无法测试 Scene 生命周期
- **位置**: `designs/issue-18/src/ui/__tests__/setup.ts:289-293`
- **问题**: `simulateSceneShutdown(scene)` 要求 `scene.emit('shutdown')`，但：
  1. `UIMockScene` 未暴露 `emit` 方法（仅代理了 `cameras`、`tweens`、`time`、`input`、`textures`、`add`）。
  2. 底层 `MockScene`（`apps/phaser3/src/test-setup.ts`）同样没有 `emit` 方法——它未继承 `EventEmitter`。
- **影响**: PRD-004 边缘场景明确要求“组件内部监听 Scene 的 `shutdown` 事件，自动清理交互和 tween”。此 helper 无法用于验证该场景，导致测试基础设施不完整。
- **修复**: 
  1. 在 `UIMockScene` 中增加对 `emit`/`on`/`off`/`once` 的代理（指向底层 `_mockScene`）。
  2. 在 `apps/phaser3/src/test-setup.ts` 的 `MockScene` 中补充 `EventEmitter` 能力（或在新设计中提供一个完整的 `MockScene` 替代）。
- **依据**: PRD-004 第 2.2 节边缘场景：“组件在 Scene 切换后未销毁 → 组件内部监听 Scene 的 `shutdown` 事件”；`__tests__/setup.ts` 第 285 行 JSDoc 声明其用途为“验证组件是否正确清理”。

### 3. `mockTexture` 对同一 Scene 重复调用会产生嵌套补丁，导致状态泄漏
- **位置**: `designs/issue-18/src/ui/__tests__/setup.ts:142-159`
- **问题**: 
  ```ts
  const originalExists = scene.textures.exists; // 若已被 patch，拿到的是 patch 后的函数
  if ((scene as any).__mockTextures) {
    (scene as any).__mockTextures.add(key);
    return; // 直接返回，但 originalExists 已被覆盖且未使用
  }
  scene.textures.exists = (k: string) => registered.has(k) || originalExists(k);
  ```
第二次调用时，`originalExists` 已经是第一次 patch 后的函数，形成闭包嵌套。虽然逻辑结果暂时正确，但存在风险：如果后续需要 `unpatch`（恢复原始函数），将无法做到。
- **影响**: 测试间状态隔离不可靠；`clearMockTextures` 只能清空注册表，不能恢复原始 `exists` 实现。
- **修复**: 
  1. 将 `originalExists` 保存在 Scene 实例的隐藏属性上（如 `__originalTexturesExists`），仅保存一次。
  2. 修改 `clearMockTextures` 为 `restoreMockTextures`，在清理注册表的同时恢复 `scene.textures.exists = scene.__originalTexturesExists`。
- **依据**: `__tests__/setup.ts` 第 4.3 节约束规则：“每个 helper 必须处理事件监听器清理——防止测试间状态泄漏”；`docs/QUALITY_SCORE.md` 要求测试基础设施可靠。

### 4. `qualityColorMap` 类型与 `UIQualityColorMap` 不匹配
- **位置**: `designs/issue-18/src/ui/utils.ts:320`
- **问题**: 
  ```ts
  export const qualityColorMap: Record<number, number> = { ... };
  ```
而 `types.ts` 定义：
  ```ts
  export type UIQualityColorMap = Record<UIQualityLevel, number>;
  ```
`UIQualityLevel` 是 `1 | 2 | ... | 9`，比 `number` 更严格。不一致的类型会削弱类型安全，消费者若使用 `UIQualityColorMap` 类型却无法与 `qualityColorMap` 值直接兼容。
- **影响**: 类型系统无法保证 `qualityColorMap` 的键仅限于 1~9。
- **修复**: 将 `utils.ts` 第 320 行改为：
  ```ts
  export const qualityColorMap: UIQualityColorMap = {
    1: 0xcccccc,
    // ...
    9: 0xf44336,
  };
  ```
- **依据**: `types.ts` 第 133 行 `export type UIQualityColorMap = Record<UIQualityLevel, number>`；设计规范 `design-spec.md` 第 2.2 节要求“类型定义…与现有组件向后兼容”。

### 5. `__tests__/setup.ts` 硬编码超长相对路径，目录结构变更即断裂
- **位置**: `designs/issue-18/src/ui/__tests__/setup.ts:15`, `41`, `276`
- **问题**: 使用 `../../../../../../apps/phaser3/src/test-setup` 穿越 6 层目录。若 `designs/` 目录层级调整（如新增子目录或迁移到 `packages/designs`），路径将失效。
- **影响**: 设计资产的“即拷即用”承诺受损；迁移时开发者必须手动修正 3 处路径。
- **修复**: 在 `design-spec.md` 的迁移指南中补充显式检查清单（Checklist），要求“全文搜索并替换 `apps/phaser3/src/test-setup` 的引用路径”；或在 `setup.ts` 顶部使用一个常量 `const TEST_SETUP_PATH = '../../../../../../apps/phaser3/src/test-setup'` 集中管理，减少遗漏。
- **依据**: `design-spec.md` 第 10 节验收标准要求“无导入错误”；`AGENTS.md` 维护规则第 5 条要求“发现代码与文档不一致时…追加条目”。

## 轻微问题（Minor）

### 1. `UITheme` 缺少 PRD-004 中使用的 `warning` 色值
- **位置**: `designs/issue-18/src/ui/utils.ts:16-35`
- **问题**: PRD-004 第 5.15 节定义 `warning: 0xffaa00`，但 `UITheme` 未包含该色值。开发者可能继续使用魔法数字 `0xffaa00`。
- **修复**: 在 `UITheme` 中增加 `warning: 0xffaa00`（或更国风化的命名如 `amber: 0xffaa00`），并在 `UIAlert.getTypeColor` 等实现中引用它。
- **依据**: PRD-004 第 5.15 节；`docs/frontend/agent-design-guide.md` 3.2 节禁止魔法数字色值。

### 2. `UIButtonState` 与 `UIInteractiveState` 完全等价，增加不必要的概念
- **位置**: `designs/issue-18/src/ui/types.ts:72-74`
- **问题**: `export type UIButtonState = UIInteractiveState;`。PRD-004 3.6 节统一使用 `UIInteractiveState`（normal/hover/pressed/disabled），没有证据表明按钮需要单独的状态类型。
- **修复**: 删除 `UIButtonState`，所有组件统一使用 `UIInteractiveState`；若确有按钮专属扩展需求，应在需要时创建（如 `UIButtonState extends UIInteractiveState | 'loading'`）。
- **依据**: PRD-004 3.6 节仅定义了一套交互状态机；设计规范应遵循最小化原则。

### 3. `index.ts` 注释中的向后兼容导出使用了已弃用的类型名 `ToastType`
- **位置**: `designs/issue-18/src/ui/index.ts:90`
- **问题**: 注释行 `// export { UIToast, ToastManager, type UIToastConfig, type ToastType } from './UIToast';` 保留了旧类型名 `ToastType`。PRD-004 已将该概念统一为 `UINotificationType`（定义于 `types.ts`）。
- **修复**: 将注释更新为 `type UINotificationType`，提示后续组件实现者使用新类型名。
- **依据**: PRD-004 第 5.2 节定义 `UINotificationType`；现有 `apps/phaser3/src/ui/index.ts` 第 24 行仍使用 `ToastType`，迁移时应一并更名。

### 4. `createFallbackRect` 与 `createInkBackground` 默认透明度不一致
- **位置**: `designs/issue-18/src/ui/utils.ts:120`, `151`
- **问题**: `createInkBackground` 使用 `alpha = 0.95`，而 `createFallbackRect` 使用 `alpha = 0.9`。PRD-004 3.3 要求“透明度 alpha ≥ 0.8”，两者均合规，但不一致的默认值可能导致同一面板内背景与回退元素视觉不统一。
- **修复**: 统一两者默认 alpha 为相同值（建议 `0.9`），或在 `UIFallbackConfig` 中显式定义默认 `alpha` 常量。
- **依据**: PRD-004 第 3.3 节资源回退统一标准。

## 风险与建议

1. **迁移路径风险**：设计资产位于 `designs/issue-18/src/ui/`，后续 Developer Agent 需手动复制到 `apps/phaser3/src/ui/`。建议在当前 PR 或后续 PR 中提供 Makefile / npm script（如 `make sync-design issue=18`），将复制步骤自动化，减少人为遗漏。

2. **旧组件兼容风险**：新的 `utils.ts` 扩展了 `UITweens`（新增 `press`、`pulse`、`tweenValue`、`toastIn`、`toastOut`），旧组件无影响；但 `index.ts` 注释掉的 Layer 1/2 导出意味着正式启用前，旧 `apps/phaser3/src/ui/index.ts` 仍继续生效。建议在下一次“替换”PR 中制定明确的组件级迁移顺序（按 PRD-004 Phase 2 的 `UIPanel → UIButton → UIText → UIList → UIDialog → 其余`）。

3. **测试基础设施先行**：`__tests__/setup.ts` 的缺陷会阻塞所有 22 个组件的单元测试开发。建议在合并本 PR 前优先修复 Major 问题 1~3，或单独提交一个修复 PR 作为本 PR 的前置依赖。

4. **视觉验收缺口**：本 PR 未包含任何 Playwright 截图或 Demo Scene。虽然设计资产阶段允许仅交付代码规格，但建议在 Phase 1 结束前补充一张 `utils` + `types` 的 Demo Scene（展示色板、缓动示例），以便提前验证视觉风格与 Foundation 规范的一致性。
